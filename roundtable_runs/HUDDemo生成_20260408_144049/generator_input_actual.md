# Generator Input

## Final Proposal
## 方案描述

本方案旨在生成一个零依赖、可双击运行的HTML文件，用于演示智能骑行头盔HUD的核心体验。方案严格遵循“一瞥可读”原则与“消灭停车掏手机”的使命，通过一个清晰的状态机模型和精心编排的自动剧本，在3分钟内展现产品的核心价值。

**1. 视觉与交互架构**
- [偏好·--] **布局骨架**：采用“下缘主信息带 + 顶部轻提示条 + 中央覆盖层 + 四角状态角标”的固定布局。下缘主信息带（占视野下方15%）用于显示当前基础态的核心信息（如导航箭头、速度、音乐信息）。顶部轻提示条（占视野上方5%）用于显示次要状态（如连接状态、时间）。中央区域绝对留空，仅在最高优先级事件（如ADAS预警、来电）时，于中央偏下位置显示瞬时弹层。四角用于固定、低干扰的状态指示（如蓝牙、电量、录制、组队）。此设计回应约束[CDO-1]与[CDO-3]，确保信息架构极简且稳定。
- [偏好·--] **功能色语义系统**：建立一套全彩配色语义，以颜色强化信息类型识别：
    - **速度/基础信息**：白色 (#FFFFFF) — 中性、高可读。
    - **导航**：蓝色 (#4A90E2) — 通用指引色。
    - **ADAS安全预警**：红色 (#FF3B30) — 高冲击力、危险警示。
    - **Mesh组队通信**：青色 (#5AC8FA) — 区别于导航，代表社交连接。
    - **音乐/媒体**：紫色 (#AF52DE) — 代表娱乐。
    - **系统/录音**：琥珀色 (#FF9500) — 代表系统级操作。
- [判断·高] **ADAS预警视觉方案**：摒弃全屏遮罩，采用“角落强闪烁+中央简明图标”的组合。当预警触发时，对应方向的角落（如左后方盲区预警，则左下角）的“BSD”角标将高频闪烁红色，同时中央偏下出现一个简洁的红色箭头图标（指向危险方向）和“注意！”文案，持续2秒后消失。这既提供了强烈的视觉冲击，又避免了遮挡主要路面视野，直接回应约束[CDO-4]和[CMO-2]。
- [判断·中] **技术实现与视觉稳定性**：遵循[CTO-9, CTO-17]的硬约束，仅使用系统字体栈（`-apple-system, BlinkMacSystemFont, ‘Segoe UI’, sans-serif`）和纯CSS/SVG绘制图形，确保`file://`协议下的完美兼容与渲染一致性。暂不实现A/B光学方案切换，以降低首版复杂度，回应约束[CDO-10]。

**2. 状态机与优先级模型**
- [判断·高] **状态定义**：采用“基础态 + 覆盖态”的双层模型，以管理复杂度并实现优先级抢占，回应约束[CDO-6]与[CTO-11, CTO-15]。
    - **基础态 (5个，互斥)**：
        1.  `IDLE`：主界面，显示速度、时间、基础状态。
        2.  `NAVIGATING`：导航中，下缘显示箭头、距离、路名。
        3.  `MUSIC_PLAYING`：音乐播放中，下缘显示歌名、歌手、播放进度。
        4.  `GROUP_TALKING`：组队通话中，下缘显示“与[组名]通话中”。
        5.  `RECORDING`：行车记录仪录制中，右上角有红色“REC”角标。
    - **覆盖态 (2个，可叠加在任何基础态之上，高优先级)**：
        1.  `ADAS_ALERT`：ADAS预警，持续2-3秒。
        2.  `INCOMING_CALL`：来电提醒，持续直至接听或挂断。
    - **系统菜单被移除**，复杂设置留给手机App，符合[CTO-8]的双屏协同理念。
- [判断·高] **优先级链条**：覆盖态 > 任何基础态。在覆盖态激活时，基础态信息暂时弱化（如降低透明度），但布局骨架不变。同一时间只显示一个覆盖态。这确保了最高优先级的安全信息能被立即感知。

**3. 演示控制与自动剧本**
- [判断·高] **演示控制台**：在HUD可视化区域下方，集成一个控制面板，回应约束[CDO-7]。
    - **底部时间轴**：可视化显示自动剧本的进度条，支持点击跳转至特定步骤。
    - **控制按钮**：提供“播放/暂停”、“上一场景”、“下一场景”、“停止”按钮。
    - **沙盒控制面板**：右侧提供一组按钮，允许手动触发各个状态（如“触发盲区预警”、“开始导航”、“模拟来电”），用于交互式演示。
- [判断·高] **键盘事件全覆盖**：定义四类按键，回应约束[CTO-3]与[CTO-12]。
    - **数字键 1-5**：直接切换至对应的基础态（1:IDLE, 2:NAV, 3:MUSIC, 4:GROUP, 5:REC）。
    - **字母键 A/B/C**：手动触发覆盖态（A:ADAS预警，B:来电，C:挂断/清除覆盖态）。
    - **空格键**：播放/暂停自动剧本。
    - **方向键左/右**：剧本上一场景/下一场景。
- [判断·高] **自动剧本叙事节奏**：设计一个约180秒的自动剧本，以“建立信任→展示核心价值→展望未来”的节奏展开，回应约束[CMO-2]与[CTO-13]。
    1.  **(0-10s) 开机与自检**：模拟HUD启动，显示品牌Logo和“系统就绪”字样，传递可靠感[CMO-6]。
    2.  **(10-40s) 核心场景：导航与语音**：自动进入`NAVIGATING`态，模拟语音指令“导航去西山森林公园”，展示关键转向信息。体现“消灭掏手机”[CDO-2, CDO-8]。
    3.  **(40-80s) 核心差异化：ADAS安全守护**：在导航基础上，依次触发“右侧盲区预警”、“左侧开门杀预警”、“前车碰撞预警”。每次预警都伴随简短的提示音（通过`<audio>`标签和用户手动触发播放按钮来规避自动播放限制[CTO-17]）和前述的红色闪烁视觉方案，强力传达“被保护着”[CMO-1, CMO-3]。
    4.  **(80-120s) 玩乐场景：组队与音乐**：切换到`GROUP_TALKING`和`MUSIC_PLAYING`态，展示社交与娱乐功能。
    5.  **(120-150s) 全程守护：录制与状态集成**：展示`RECORDING`态，并快速循环所有基础态，体现系统在不同任务间的无缝切换。
    6.  **(150-180s) 收尾与品牌**：返回`IDLE`态，显示速度、电量，最后淡出品牌Slogan。

**4. 代码结构**
- [判断·高] **单文件内分层**：在单个HTML文件内，通过`<script>`标签实现逻辑分层，回应约束[CTO-5, CTO-10, CTO-16]。
    1.  **常量定义层**：集中定义`STATE_MACHINE`、`SCENARIO_SCRIPT`、`KEYMAP`等配置数据。
    2.  **状态渲染层**：一个`render(state)`函数，根据当前状态对象

## Reviewer Amendments
[CTO] 1. [事实·高] 7 个页面态与优先级切换机制未被完整满足 — 来源：你的约束第2条要求“7个页面态与优先级切换”；方案实际定义为 5 个基础态 + 2 个覆盖态，但没有清晰证明这就是“7个页面态”的一一映射，也没有给出完整状态表、转移表、抢占/恢复规则
[判断·高] 当前方案更像“7类显示状态元素”，不是“7个页面态的完整有限状态机” — 基于：缺少状态枚举、进入条件、退出条件、优先级冲突处理、恢复策略
[判断·高] 建议修改：补一张明确状态表，至少包含 `stateId / 类型(基础或覆盖) / 优先级 / 可被谁抢占 / 退出后恢复到谁 / 默认持续时间 / 可响应按键`，并明确“7个页面态”究竟是哪 7 个
2. [事实·高] 键盘事件全覆盖未满足 — 来源：你的约束第3条要求“键盘事件全链路，而不是少量 happy path 按键”；方案只定义了数字键 1-5、A/B/C、空格、左右方向键
[判断·高] 这只能算“基础演示快捷键”，不能算“全链路” — 基于：缺少确认/取消、剧本终止/恢复、覆盖态接受/拒绝、焦点保护、按键冲突处理、长按/连按策略、禁用态处理

- 场景触发键：A=ADAS，C=Call，V=Voice，R=Record，N=Nav 等
- 确认取消键：Enter / Esc
- 剧本控制键：Space 播放暂停，[`[`]/`]`] 跳步，S 停止，P 恢复，Home 重置
同时定义“覆盖态激活时哪些键仍可用”
