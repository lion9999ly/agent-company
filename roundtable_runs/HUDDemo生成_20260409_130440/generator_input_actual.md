## Final Proposal
# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个零依赖、单HTML文件的HUD交互演示原型，用于向供应商和投资人直观展示智能骑行座舱的核心体验。方案严格遵循“安全第一、一瞥可读”的原则，并围绕“消灭停车掏手机”的核心使命进行设计。

**1. 核心视觉架构与布局**
- [偏好·--] 采用 **四角信息锚点 + 中央绝对留空** 的布局。左上角(LT)为系统状态（连接、电量），右上角(RT)为环境信息（时间、温度），左下角(LB)为次要动态信息（音乐/组队），右下角(RB)为主信息区（导航/速度/ADAS预警）。此布局确保骑手视线无需大幅移动，用余光即可捕捉关键信息。
- [偏好·--] 定义 **全彩功能色语义系统**：速度/基础信息为白色（#FFFFFF），导航指示为蓝色（#4A90E2），ADAS安全预警为高饱和红色（#FF3B30）并伴随闪烁，Mesh组队状态为青色（#5AC8FA），音乐控制为紫色（#AF52DE）。单绿模式时，所有颜色映射为不同明度的绿色。
- [判断·高] 所有文本采用无衬线字体（系统默认 `sans-serif`），主信息区字体大小确保在模拟的2-3米虚像距离下清晰可读。信息密度随模拟速度（S0-S3）动态调整：低速时显示更多细节（如下一个转弯后的路名），高速时仅保留箭头、距离和关键预警。

**2. 状态机与页面态设计**
- [判断·高] 采用 **“基础巡航态 + 瞬时覆盖态”** 的两层状态机模型，而非7个平级页面。
  - **基础态 (1个)**：`CRUISING`。默认状态，整合速度、基础导航、系统状态。
  - **覆盖态 (6个)**：`NAV_FULL`（详细导航）、`ADAS_WARNING`（预警）、`VOICE_LISTENING`（语音聆听）、`VOICE_FEEDBACK`（语音反馈）、`MESH_TALKING`（组队通话）、`MEDIA_CONTROL`（媒体控制）。覆盖态以非模态形式在对应信息区强化显示，并遵循严格的优先级抢占规则（例如，`ADAS_WARNING` 可抢占除 `VOICE_FEEDBACK` 外的所有状态）。
- [判断·中] 提供完整的 `key x currentState x effect` 矩阵定义键盘快捷键。例如，数字键1-7直接切换至对应页面态，空格键暂停/继续自动剧本，`R`键重置Demo，`A`键从头播放剧本，`ESC`键清除当前覆盖层回到纯巡航态。

**3. 自动剧本与叙事节奏**
- [判断·中] 设计三条独立剧本，通过底部时间轴组件可视化并可控：
  1.  **日常通勤**：展示从启动、语音设导航、听音乐到抵达的流畅体验，重点呈现“30秒语音操作”和导航信息的一瞥可读。
  2.  **紧急场景**：作为核心，重点演绎ADAS预警链路。模拟BSD盲区报警、开门杀预警、前车碰撞预警（FCW）。预警通过对应角落（如右后盲区预警在RB区）的红色高亮闪烁实现，**禁止使用中央弹窗**，确保惊吓但不遮挡视野。
  3.  **组队骑行**：展示Mesh组队连接、队友位置状态（LB区）、组队通话请求与接听。
- [偏好·--] 剧本节奏张弛有度，紧急场景集中爆发，日常场景平稳流畅，整体演示时长控制在3分钟内。

**4. 技术实现与演示控制**
- [判断·高] 采用 **单舞台（Single Stage）架构**。一个 `div#hud-stage` 容纳所有视觉元素，通过JavaScript状态机驱动CSS类名和内容更新，实现零跳转的平滑过渡。
- [偏好·--] **右侧沙盒控制面板**默认隐藏，仅通过快捷键（如 `~` 键）呼出一个极简浮层。面板内提供A/B光学方案（全彩/单绿）的CSS变量切换滑块、S0-S3速度模拟滑块、以及手动触发各页面态的按钮，用于现场灵活演示。
- [判断·中] 时间轴拖拽与状态机联动通过记录每个剧本关键帧的时间戳和状态实现。拖拽时，计算当前时间点对应的状态并强制状态机跳转，同时更新所有视觉组件。

**5. 视觉细节与动效**
- [偏好·--] 整体视觉为深空黑背景（#000000），模拟头盔内部视野。HUD元素使用发光效果（`text-shadow`）和轻微的呼吸动画（关键信息低频透明度变化）来增强可读性和科技感，但避免过度炫技。
- [判断·中] 开机动画设计为“系统自检”：从中心Logo扫描式亮起，快速扩展到四角信息区依次点亮，最后稳定到巡航态。时长控制在3秒内，传达可靠与安全的品牌感知。

## 决策点回应
- **决策1：采用四角布局，中央绝对留空** → 回应约束：[CDO-1]、[CDO-7]、[CMO-2]。这是安全底线和实现一瞥可读的基石。
- **决策2：采用“基础态+覆盖态”两层状态机模型** → 回应约束：[CTO-9]、[CTO-11]、[CDO-5]。这简化了优先级逻辑，使7个页面态的管理更清晰，符合HMI设计范式。
- **决策3：预警通过角落闪烁实现，禁用中央弹窗** → 回应约束：[CDO-2]、[CTO-6]、[CMO-1]。在确保安全警示冲击力的同时，绝对保障中央视野清净，这是“被保护着”而非“被干扰着”的关键体现。
- **决策4：集成右侧沙盒面板与底部时间轴** → 回应约束：[CDO-3]、[CDO-10]、[CTO-13]。这是Demo可控、可演示的核心，必须提供直观的调试与剧本控制能力。
- **决策5：实现A/B光学方案与速度响应式UI** → 回应约束：[CDO-4]。通过CSS变量切换和基于速度的信息密度逻辑，展示产品的环境自适应能力。
- **决策6：键盘快捷键全覆盖，并与自动剧本共用事件入口** → 回应约束：[CDO-9]、[CDO-12]、[CTO-10]、[CTO-13]。确保演示的稳定性和操作的容错性，是专业演示的必备条件。
- **决策7：设计三条包含ADAS紧急场景的自动剧本** → 回应约束：[CDO-6]、[CTO-5]、[CTO-6]、[CMO-1]、[CMO-3]。直接回应产品核心差异化与演示目标，将“被保护着”的体验故事化。

## 自评：最大风险
**最大风险在于状态机优先级抢占逻辑的完备性与演示稳定性。** 在单线程JavaScript中模拟多路并发事件（如导航提示中突然出现盲区预警，同时语音反馈响起），如果状态切换规则定义有遗漏或冲突，可能导致UI显示错乱、剧本卡死，在投资人面前造成致命演示事故。必须通过详尽的 `key x currentState x effect` 矩阵和大量的边界条件测试来规避。其次，在零依赖下用纯CSS实现稳定、流畅的发光和呼吸动画，在不同浏览器和性能的电脑上可能存在渲染差异。
