# Generator Input

## Final Proposal
## 方案描述

本方案旨在生成一个用于演示的、离线运行的HUD体验HTML文件。方案严格遵循“一瞥可读”和“被保护着”的核心原则，通过一个精心设计的有限状态机（FSM）来管理所有视觉状态，确保信息清晰、交互可控、演示流畅。

**1. 视觉与信息架构设计**
- [偏好·--] **四角信息容器布局**：采用LT（左上）、RT（右上）、RB（右下）、LB（左下）四个固定角区作为信息容器。中央视野绝对留空，确保骑手视线焦点始终在路面上。这直接回应了安全硬约束。
- [判断·高] **功能色语义系统**：建立一套高对比度、功能明确的配色体系，确保余光扫视下的快速识别。
    - **速度/基础信息（白色）**：用于LT区的速度、时间等常驻信息。
    - **导航指引（蓝色）**：用于RB区的导航箭头、距离、路名。
    - **安全预警（红色）**：用于任何ADAS预警信息（如RT区的盲区图标），采用高饱和度红色并辅以呼吸/闪烁动效，最大化视觉冲击力。
    - **组队通信（青色）**：用于LB区显示组队成员状态、语音通话状态。
    - **音乐/系统（紫色）**：用于RT区显示当前播放的歌曲信息。
- [判断·中] **信息密度与字体控制**：每个角区同一时间仅显示1-2个核心信息元素，主信息字体巨大（模拟7米外观看效果），次要信息极小化或通过状态切换查看。避免任何形式的滚动或长文本。

**2. 状态机与交互逻辑**
- [判断·高] **七态有限状态机（FSM）**：状态机是Demo的核心逻辑引擎，明确定义7个状态及其转换关系，确保演示逻辑清晰可控。
    1.  **巡航态**：基础状态，显示速度、时间（LT），无其他任务。
    2.  **导航态**：激活导航，在RB区显示箭头、距离、下一道路名。
    3.  **音乐态**：激活音乐播放，在RT区显示歌名、歌手。
    4.  **组队态**：激活Mesh组队，在LB区显示队友在线状态。
    5.  **记录态**：示意行车记录仪工作中，在RT区显示极小录制图标。
    6.  **通话态**：来电提醒，作为高优先级中断，占据RT区并脉冲提示。
    7.  **预警态**：ADAS安全预警（如BSD），作为最高优先级中断，以红色视觉方案占据RT区（或对应威胁方向角区），并强制中断其他非核心信息。
- [判断·高] **优先级仲裁与中断机制**：状态机内置优先级（预警 > 通话 > 导航/音乐/组队 > 记录 > 巡航）。高优先级状态可中断低优先级状态的显示，并在结束后恢复原状态。这完美体现了“被保护着”的产品内核。
- [事实·高] **全链路键盘控制**：实现完整的键盘事件映射，确保演示者完全掌控。
    - `数字键1-7`：直接切换到对应状态（1:巡航，2:导航...）。
    - `A`：启动/停止**自动剧本**播放。
    - `P`/`S`：在自动剧本播放时，暂停/继续。
    - `Esc`：强制退出任何状态，回到巡航态。
    - `Tab`：在非中断态（如导航、音乐）间循环切换。

**3. 演示叙事与自动剧本**
- [判断·中] **三条自动剧本**：预设三条时序驱动的剧本，模拟真实骑行场景，用于无人讲解时的自动演示。
    1.  **日常通勤**：巡航 → 导航（模拟去公司）→ 音乐（播放歌曲）→ 巡航。
    2.  **紧急安全**：巡航 → 导航 → **预警（模拟右侧盲区车辆）** → 巡航。重点突出ADAS的及时介入。
    3.  **组队摩旅**：巡航 → 组队（显示队友）→ 导航 → 音乐 → 巡航。展现社交与娱乐结合。
- [偏好·--] **剧本可视化控制**：在界面右侧设计一个“沙盒控制面板”，实时显示当前状态、自动剧本进度条，并提供播放/暂停/跳转的控制按钮。底部设计一个时间轴，可视化展示剧本的关键事件点（如“第8秒：触发盲区预警”），支持点击跳转。

**4. 技术实现策略**
- [判断·高] **原生单文件架构**：采用纯HTML、CSS、Vanilla JS编写，所有代码、样式、资源（如字体）内联或Base64嵌入，确保生成一个双击即可运行的`.html`文件，零外部依赖。
- [判断·高] **模块化代码组织**：在单文件内，通过清晰的注释将代码划分为：`<style>`样式区块、`<script>`中的状态机定义、事件监听器、自动剧本时序配置、DOM操作函数。虽然在一个文件内，但结构清晰，便于后续微调。
- [判断·中] **视觉差异表达**：通过CSS变量定义两套光学方案（如A方案：高对比度深色UI；B方案：半透明浅色UI）。在沙盒面板中提供切换按钮，直观展示不同光学方案下的视觉效果差异。

## 决策点回应
- **决策1：采用“四角布局”并弃用“顶部状态条+中央弹层”** → 回应约束：[CDO-2]（置信度：[事实·高]）。这是来自技术/安全评审的硬性规定，必须遵守以确保中央视野无遮挡。
- **决策2：实现基于优先级的七态有限状态机（FSM）** → 回应约束：[CDO-7]、[CTO-2]、[CTO-8]、[CTO-12]（置信度：[判断·高]）。这是满足演示完整性、体现安全中断优先级、并保证代码可控不混乱的核心架构决策。
- **决策3：设计全链路键盘控制与可视化沙盒控制面板** → 回应约束：[CDO-4]、[CDO-8]、[CTO-3]、[CTO-13]（置信度：[判断·高]）。这是满足演示交付硬约束（时间轴、控制面板）和确保现场演示绝对可控的关键。
- **决策4：使用纯原生技术栈实现单HTML文件** → 回应约束：[CTO-1]、[CTO-11]（置信度：[判断·高]）。这是满足零依赖、离线运行、低故障率演示目标的最稳妥技术路径。
- **决策5：定义“功能色语义系统”并赋予预警红色最高视觉权重** → 回应约束：[CMO-1]、[CMO-4]、[议题专属指导]（置信度：[判断·高]）。这是直观传达“被保护着”差异化、并确保安全信息一瞥可读的核心视觉策略。

## 自评：最大风险
**最大风险在于“视觉保真度与真实硬件效果的差距”**。本Demo在普通电脑屏幕上运行，无法完全模拟真实HUD的光学特性，如**环境光自适应亮度、透明显示器的视觉融合效果、特定视场角（FOV）下的畸变与清晰度**。投资人或供应商若对光学有深度了解，可能会质疑最终量产效果。为缓解此风险，方案中设计了A/B光学方案切换，并会在演示时明确说明当前为“功能与交互逻辑模拟”，真实效果需参考光学样机。
