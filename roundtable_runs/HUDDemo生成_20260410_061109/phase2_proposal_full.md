# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个**零依赖、单HTML文件**的HUD演示系统，核心目标是在**3分钟内**通过自动剧本，向投资人和供应商清晰传达产品的核心价值——“被保护着”的ADAS安全体验，并展示“消灭停车掏手机”的交互逻辑。方案严格遵循所有约束，并采用“四段生成法”确保代码可维护性。

**1. 视觉架构与布局 (回应CDO约束1, 3, 4, 5)**
- [偏好·--] **四角布局**：严格遵循AC3/AC4约束，将视野中央区域（AC4）完全留空，所有信息元素仅置于四个角落。
    - **左上角 (LT)**: **速度信息**。采用大字号、高对比度白色数字，符合AC3验收标准。
    - **右上角 (RT)**: **状态指示区**。用于显示低优状态图标，如组队连接（青色）、音乐播放（紫色）、蓝牙/Wi-Fi连接状态。在S3高速状态下，此区域信息将降级或隐藏（AC11）。
    - **左下角 (LB)**: **导航辅助区**。显示下一个导航动作（如“300m后左转”），使用导航蓝色。
    - **右下角 (RB)**: **主信息/操作区**。作为导航、来电、语音反馈等主要信息的承载区域，采用逐向提示，避免完整地图铺屏。
- [判断·高] **瞬时预警机制**：ADAS预警（BSD、开门杀、FCW）将**严格禁用中央弹层**。当预警触发时，对应威胁方向的角落（如左侧来车威胁触发LB角）将进行**高饱和度红色闪烁**（AC7），并伴随简洁图标（如车辆轮廓、警告三角），确保余光即可感知，冲击力强。

**2. 状态机与优先级管理 (回应CDO约束7, 8, CTO约束2, 8, 10, 11)**
- [判断·中] **7个显式页面态枚举**：`BOOT`（开机自检）、`NORMAL`（常态）、`NAVIGATION`（导航）、`CALL`（来电）、`ADAS_ALERT`（ADAS预警）、`GROUP_COMM`（组队通话）、`VOICE_FEEDBACK`（语音反馈）。
- [判断·中] **两层优先级模型**：
    1.  **基础态**：`NORMAL`、`NAVIGATION`、`GROUP_COMM`。它们互斥，构成骑手主动选择的主要界面。
    2.  **瞬时覆盖态**：`ADAS_ALERT`（最高）、`CALL`（高）、`VOICE_FEEDBACK`（中）。它们可以抢占基础态的显示，并在结束后自动恢复原状。例如，导航中被来电打断，接听/拒接后自动返回导航界面。
- [事实·高] **提供优先级状态转移矩阵**：在Demo的沙盒面板中，清晰展示7个状态的优先级、抢占规则及恢复逻辑（AC5, AC6）。

**3. 交互与演示逻辑 (回应CDO约束9, 13, CTO约束3, 4, 12, 13)**
- [判断·中] **全覆盖键盘快捷键系统**：支持单键与组合键，确保现场演示流畅可控。
    - `H`：显示帮助面板（列出所有快捷键）。
    - `Space`：播放/暂停自动剧本。
    - `→` / `←`：剧本单步前进/后退。
    - `R`：重置Demo。
    - `1-7`：直接切换到对应页面态（用于手动演示）。
    - `C`：模拟来电；`A`：触发ADAS预警；`V`：模拟语音反馈。
    - `Enter`/`Y`：确认（接听、确认导航）；`Esc`/`N`：拒绝（拒接、取消）。
- [判断·中] **3条自动剧本时序驱动**：剧本通过派发标准事件（如 `dispatchEvent(new CustomEvent('triggerAlert', {detail: {type: 'BSD_LEFT'}}))`）驱动状态机，而非直接操作DOM，保证逻辑统一。
    1.  **安全守护剧本**：开机自检 → 常态骑行 → 触发左侧盲区预警 → 预警解除 → 触发前向碰撞预警 → 预警解除。重点展示ADAS的响应速度与视觉冲击。
    2.  **无缝导航剧本**：语音指令“导航去虹桥机场” → 语音反馈“正在规划路线” → 导航界面启动 → 模拟骑行中逐向提示 → 来电打断 → 拒接后恢复导航。展示语音消灭掏手机、及状态无缝切换。
    3.  **组队摩旅剧本**：组队连接 → 组队通话中 → 触发ADAS预警（展示高优先级抢占） → 预警后恢复通话。
- [事实·高] **沙盒与调试面板**：界面右侧固定显示沙盒面板（AC9），包含：当前状态、速度等级（S0-S3）、激活的剧本、以及一个只读的时间轴，用于观察剧本进度和状态跳转。

**4. 视觉表现与适配 (回应CDO约束10, 11, 12)**
- [事实·高] **开机自检（BOOT）动画**：模拟系统启动，从品牌Logo过渡到关键系统状态检查（如GPS、ADAS、通信），最后淡入到`NORMAL`态，建立可靠性的第一印象（AC14）。
- [判断·中] **速度情境化降级**：定义S0（静止）、S1（低速）、S2（中速）、S3（高速）。在S3时，RT角的音乐/社交类图标自动隐藏，仅保留最必要的安全相关状态。
- [事实·高] **A/B光学方案模拟**：提供一键切换（快捷键 `B`），将全彩UI强制映射为**不同明度的单绿色**，准确模拟光波导方案的视觉效果（AC10）。全彩模式下，严格执行功能色语义：速度白、导航蓝、预警红、组队青、音乐紫。

**5. 技术实现路径 (回应CTO约束1, 5, 6, 9, 14, 15)**
- [判断·高] **技术栈**：纯原生HTML + CSS + Vanilla JS，零依赖，确保单文件双击可运行。
- [判断·中] **四段生成法**：
    1.  **定义表阶段**：用JS对象明确定义状态表、事件表、优先级矩阵、剧本步骤表。
    2.  **核心逻辑阶段**：生成状态机引擎、事件总线、剧本播放器。
    3.  **UI骨架阶段**：生成HTML结构和基础CSS，使用CSS绘制图标（如用border绘制三角形、圆形表示状态），避免Base64图片。
    4.  **整合与样式阶段**：应用详细样式（模拟HUD发光效果、透明度）、绑定事件、并最终合并为一个格式良好的HTML文件。

## 决策点回应
- **决策1：采用严格四角布局，中央留空** → 回应约束：[CDO4, CDO5]（置信度：[事实·高]）。这是安全底线，必须遵守。
- **决策2：采用“基础态+瞬时覆盖态”两层状态机模型** → 回应约束：[CDO7, CDO8, CTO10, CTO11]（置信度：[判断·中]）。此模型能清晰表达ADAS等高优信息的抢占与恢复，逻辑更可控。
- **决策3：预警禁用中央弹层，采用角落色彩闪烁** → 回应约束：[CDO6, CMO1]（置信度：[事实·高]）。这既是硬约束，也是突出ADAS差异化、制造视觉冲击的关键。
- **决策4：使用纯CSS绘制UI，避免图片** → 回应约束：[CTO1, CTO15]（置信度：[偏好·--]）。在单文件零依赖和可维护性要求下的最优折衷。
- **决策5：剧本驱动通过事件派发，而非直接DOM操作** → 回应约束：[CTO13]（置信度：[判断·中]）。确保自动演示与手动操作逻辑同源，避免维护噩梦。

## 自评：最大风险
**最大风险在于“单文件零依赖”与“丰富、流畅的视觉表现及复杂状态逻辑”之间的平衡。** 使用纯CSS绘制所有图形和动画，在模拟HUD的发光质感、平滑过渡以及处理多个并发状态切换的动画时，可能面临CSS复杂度陡增和性能挑战。在低性能设备上运行剧本时，可能会出现动画卡顿，影响演示的流畅性与专业感。**缓解措施**：极致简化非关键动画，优先保证状态切换的逻辑正确性与响应速度；在生成代码时，对CSS进行严格优化和合并。
