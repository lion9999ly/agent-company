## Final Proposal
# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个面向供应商与投资人的、可离线运行的HUD体验演示HTML文件。方案的核心是构建一个**以安全为导向、状态机驱动、支持自动剧本与手动控制**的交互内核，并严格遵循“一瞥可读”与“消灭停车掏手机”的核心使命。

**1. 视觉框架与布局 (回应约束: CDO-4, CDO-1, CMO-6)**
- [偏好·--] **四角分区布局**：严格遵守“中央视野完全留空”的安全铁律。信息分布于四个角落，形成稳定的视觉锚点。
    - **左上角 (LT)**：状态区。显示连接状态（蓝牙/组队）、行车记录仪图标（录制中/待机）。
    - **右上角 (RT)**：次要信息区。显示时间、天气（简图标）、电量。
    - **左下角 (LB)**：动态功能区。作为音乐/来电/语音交互态的主显示区。
    - **右下角 (RB)**：**核心信息区**。作为导航、速度/档位等核心行车信息的主显示位置，符合人眼余光扫视习惯。
- [判断·高] **信息密度自适应 (S0-S3)** — 基于：P0-4反馈。建立四档速度模型，驱动UI复杂度动态变化：
    - **S0 (静止/极低速)**：显示完整信息（如导航全预览、音乐封面）。
    - **S1 (城市巡航)**：简化信息，导航仅显示下一个转向箭头和距离。
    - **S2 (快速路/高速)**：极简模式，仅保留核心导航箭头和速度。
    - **S3 (紧急/高负荷)**：仅显示最高优先级的ADAS预警（红色闪烁），其他所有信息隐藏。

**2. 色彩与语义系统 (回应约束: CDO-6, CDO-7)**
- [偏好·--] **全彩功能色语义**：建立快速、无歧义的色彩认知体系。
    - **速度/基础信息**：白色。确保基础可读性。
    - **导航**：蓝色。用于路径线、转向箭头、目的地标识。
    - **预警 (ADAS)**：高饱和红色 (#FF3B30)。**必须配合对应角落的脉冲式闪烁动画**，以最高优先级抢占注意力。
    - **组队通信**：青色 (#5AC8FA)。用于组队成员状态、对讲激活提示。
    - **音乐/媒体**：紫色 (#AF52DE)。用于播放控制、歌曲信息。
- [判断·高] **A/B光学方案视觉表达** — 基于：类似领域类推。通过CSS主题变量一键切换，模拟两种显示效果：
    - **方案A (OLED全彩)**：启用完整色彩语义，使用高对比度、纯色填充。
    - **方案B (光波导单绿)**：将所有色彩映射为不同明度的绿色（#00FF00及其变体），并降低非核心元素的饱和度，模拟单色显示在强光下的可读性挑战与应对。

**3. 交互内核：状态机与优先级 (回应约束: CTO-7, CTO-10, CDO-12)**
- [判断·高] **双层状态机模型** — 基于：P0-5反馈与状态机工程经验。
    - **Base State (底层状态)**：定义骑手主动发起的、持续性的任务态。如：`巡航`、`导航中`、`音乐播放`、`语音交互中`、`设置`。这些状态共享同一布局区域（如导航在RB，音乐在LB），互斥切换。
    - **Overlay State (覆盖层状态)**：定义系统主动触发的、高优先级的瞬时态。如：`ADAS预警`、`来电`、`消息通知`。此类状态可**打断并冻结**当前的Base State，以强视觉形式（闪烁、放大）在对应角落（ADAS预警在目标方位角）显示，结束后自动恢复被冻结的Base State。
- [判断·高] **事件系统与演示控制** — 基于：CTO-8, CTO-9要求。构建中央事件总线，统一处理：
    - **键盘事件**：定义“场景动作映射表”（如按`1`触发BSD左预警，按`N`进入导航），确保手动演示全覆盖。
    - **自动剧本**：采用声明式JSON脚本数组，定义每个事件触发的时间点、类型和参数。剧本与手动控制共享同一套状态变更接口，杜绝双轨逻辑。
    - **演示控制面板**：在HUD视图外，固定显示**底部时间轴**（控制剧本播放/暂停/跳转）和**右侧沙盒面板**（列出所有可手动触发的事件按钮），满足评审需求。

**4. 叙事节奏与Demo结构 (回应约束: CMO-2, CDO-8, CTO-4)**
- [判断·中] **多场景剧本串联** — 基于：P0-3反馈与演示设计经验。设计三个核心剧本，在3分钟内展现产品价值：
    1.  **日常通勤**：启动→导航至公司→途中听音乐→来电接听（语音控制）→结束。展示流畅的语音交互和基础信息呈现。
    2.  **安全守护**：巡航中依次触发**左侧BSD预警**、**前向FCW预警**、**右侧“开门杀”预警**。重点展示不同方位预警的视觉表达（对应角落红色闪烁）和优先级（覆盖音乐界面）。
    3.  **组队摩旅**：进入组队模式→查看队友位置（LB区显示）→发起语音对讲→接收队友消息。展示社交与安全结合的场景。
- [偏好·--] **开场与收尾**：Demo启动后，首先以**品牌动画（系统自检风格）** 建立信任感，然后自动进入“暂停”状态的日常通勤剧本，等待演示者通过控制面板或键盘开始“表演”。

## 决策点回应
- **决策1：采用“四角布局+中央留空”并严格分区** → 回应约束：[CDO-4]（置信度：[事实·高]）、[CDO-1]（置信度：[事实·高]）。这是安全设计的底线，不可妥协。
- **决策2：建立S0-S3速度自适应信息密度逻辑** → 回应约束：[CDO-11]（置信度：[事实·高]）、[CDO-2]（置信度：[判断·高]）。这是实现“一瞥可读”在不同骑行工况下的关键技术路径。
- **决策3：采用双层（Base + Overlay）状态机模型处理预警打断** → 回应约束：[CDO-12]（置信度：[事实·高]）、[CTO-7]（置信度：[判断·高]）。此模型能清晰界定预警与常规任务的优先级关系，确保系统行为可控、可预测。
- **决策4：将A/B光学方案差异实现为CSS主题切换** → 回应约束：[CDO-5]（置信度：[事实·中]）、[CTO-11]（置信度：[判断·中]）。以低成本方式直观传达技术选择对用户体验的影响，便于讨论。
- **决策5：集成底部时间轴与右侧事件面板作为Demo标配** → 回应约束：[CDO-9]（置信度：[事实·高]）、[CTO-4]（置信度：[事实·高]）。这直接服务于“给供应商和投资人展示”的核心目标，确保演示过程稳定、可控、信息全面。

## 自评：最大风险
**最大风险在于“视觉冲击力”与“一瞥可读”的平衡，尤其是在ADAS预警设计上。** [判断·中]
- 为了体现“被保护着”的差异化，预警需要强烈的视觉抢占（闪烁、红色、大区域）。但过度强烈的动画或过大元素可能**短暂干扰骑手对真实路况的观察**，违背HUD辅助安全的初衷。
- **缓解策略**：在Demo中
