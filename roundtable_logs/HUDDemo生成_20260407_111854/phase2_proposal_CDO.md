# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个零依赖、单HTML文件的HUD演示系统，严格遵循“消灭停车掏手机”的核心使命，并突出“被保护着”的ADAS差异化价值。整个Demo将围绕一个**固定视觉骨架**和一套**严谨的状态机**构建。

**1. 视觉骨架与信息架构**
- [偏好·--] **四象限固定布局**：严格遵循“中央视野留空”的安全原则。左上角(LT)为ADAS预警区，左下角(LB)为辅助信息区（如音乐、录制状态），右上角(RT)为系统状态区（如连接、电量），右下角(RB)为主信息区（导航、速度）。这是所有信息呈现的绝对基础。
- [判断·高] **全彩功能色语义系统**：基于“一瞥可读”原则，建立强语义关联。速度显示为白色（中性、高亮）；导航信息为蓝色（引导、冷静）；ADAS预警为红色（危险、紧急），并伴有对应象限的脉冲式高亮；组队状态为青色（连接、协作）；音乐控制为紫色（娱乐、次要）。这回应了视觉冲击力与功能辨识度的需求。
- [判断·中] **A/B光学方案视觉差异表达**：在Demo的调试面板中设置切换按钮。A方案（全彩OLED）展示完整色彩与图标；B方案（单绿光波导）将所有UI元素转换为不同明度的绿色，并简化图标为轮廓，直观呈现技术约束下的视觉降级。

**2. 状态机与交互逻辑**
- [事实·高] **7态优先级栈**：严格实现 `预警 > 来电 > 导航 > 组队 > 音乐 > 录制 > 主界面` 的优先级逻辑。高优先级状态（如ADAS预警）会完全抢占并覆盖低优先级界面，并在结束后自动恢复原状态。所有状态切换通过 `setMode` 和 `emitEvent` 两类输入协议驱动，确保逻辑可验证。
- [判断·高] **S0-S3速度自适应降噪**：模拟真实骑行场景。当“车速”由低（S0-S1）升高至高速（S2-S3）时，LB区的音乐信息、组队成员列表等非关键信息将动态隐藏，仅保留RT区的精简状态图标和RB区的核心导航信息，确保高速时界面极致简洁。
- [事实·高] **ADAS预警的空间化直觉表达**：当检测到左后盲区车辆时，LT预警区将出现红色闪烁的车辆图标，同时左侧视野边缘辅以红色的光晕脉冲，**绝不使用中央弹窗或纯文字**。这直接体现了“被保护着”的核心差异化，并符合安全约束。

**3. 演示叙事与自动剧本**
- [判断·中] **三段式自动剧本**：剧本设计遵循“建立信任 -> 展示核心 -> 展望未来”的叙事节奏。
    1.  **剧本A（基础体验）**：开机自检 -> 语音唤醒设置导航 -> 展示导航界面 -> 触发低速音乐播放 -> 展示降噪逻辑。
    2.  **剧本B（安全核心）**：在导航过程中，依次触发右侧BSD预警、左侧开门杀预警、前向FCW预警，强力展示ADAS的空间化提示与状态抢占。
    3.  **剧本C（协同场景）**：模拟组队骑行，展示组队通话接入与导航信息共享。
- [事实·高] **集成演示控制层**：界面底部固定显示时间轴与剧本控制按钮（播放/暂停/跳转）；右侧为沙盒面板，可手动触发任何状态或事件；顶部为A/B光学方案切换与速度滑块。这确保了演示的完全可控与可解释。

**4. 技术实现框架**
- [事实·高] **单文件分层架构**：代码在单个HTML文件内，按 `CONFIG`（常量与配色）、`STORE/FSM`（状态存储与状态机）、`RENDER`（渲染函数）、`INPUT`（键盘/剧本事件映射）、`SCRIPT`（自动剧本数组）、`DEBUG`（调试面板UI）分段组织。确保生成过程可控，最终产物可维护。
- [判断·高] **输入协议与键盘映射**：键盘数字键1-7直接`setMode`切换至对应状态（如‘1’切预警）；字母键用于`emitEvent`触发特定场景（如‘B’触发左侧BSD，‘C’接听来电）。空格键控制剧本播放/暂停。这满足了键盘事件全覆盖与交互测试的需求。

## 决策点回应
- **决策1：采用严格四象限固定布局，中央完全留空** → 回应约束：[CDO-1]（置信度：[事实·高]）。这是安全底线，不可妥协。
- **决策2：ADAS预警采用对应象限视觉强化，禁用中央弹层** → 回应约束：[CDO-2]（置信度：[事实·高]）。这是产品核心差异化价值的直观体现。
- **决策3：实现基于栈的7态优先级状态机** → 回应约束：[CDO-3], [CTO-2], [CTO-11]（置信度：[事实·高]/[判断·高]）。这是确保多信息源有序管理、逻辑可验证的基础。
- **决策4：设计包含控制层、沙盒面板、A/B切换的完整演示界面** → 回应约束：[CDO-4], [CTO-15]（置信度：[事实·高]/[偏好·--]）。这是面向投资人/供应商演示稳定性和可解释性的关键。
- **决策5：实现速度自适应降噪逻辑** → 回应约束：[CDO-5]（置信度：[事实·高]）。这是让Demo脱离静态展示，体现场景智能的关键设计。
- **决策6：采用原生JS分层代码结构，明确定义`setMode`与`emitEvent`** → 回应约束：[CDO-6], [CDO-7], [CTO-10], [CTO-14]（置信度：[事实·高]/[判断·高]）。这是在单文件零依赖约束下，保证代码可维护性与状态机严谨性的最优解。
- **决策7：设计三段式自动剧本，聚焦导航、ADAS、组队** → 回应约束：[CTO-4], [CTO-8], [CTO-9], [CMO-1], [CMO-3]（置信度：[判断·中]/[事实·高]）。这是在3分钟内讲清故事、突出核心价值的必要叙事设计。

## 自评：最大风险
- [判断·中] **最大风险在于“单文件复杂度与生成质量”**。方案要求在一个HTML文件内实现一个包含完整状态机、渲染引擎、剧本调度器和调试UI的复杂系统，代码量极易超过500行。在分段生成并最终拼装的过程中，任何一段的逻辑错误或接口不一致都可能导致整个Demo无法运行。**缓解策略**：严格遵循分段架构，先独立验证核心状态机（FSM）和渲染层（RENDER）的逻辑正确性，再依次集成输入、剧本和调试面板，每一步都进行功能测试。
