# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个零依赖、单HTML文件的HUD演示系统，核心目标是向供应商和投资人直观传达“智能骑行座舱”的核心价值：**消灭停车掏手机**与**被保护着**的安全体验。方案将严格遵循所有硬约束，构建一个稳定、可控、叙事清晰的演示。

**1. 视觉骨架与信息架构**
- **[偏好·--] 采用“四角徽章 + 右下主信息区 + 顶部状态条”的固定布局。** 这是对“右下/下缘主信息区”方案的演进，完全规避了中央视野。四角仅承载极小、高辨识度的状态徽标（如速度数字、电量图标），主导航信息（转向箭头、距离、路名）集中在右下角，确保骑手视线焦点稳定。顶部状态条用于轻量提示（如语音唤醒、组队状态）。
- **[判断·高] 严格遵循四角信息映射与ADAS方位闪烁原则。** — 基于：P0问题清单。LT角：白色大号速度数字，下方附骑行状态（S0-S3）小字。RT角：设备状态图标（电量、蓝牙/Wi-Fi）。RB角：导航信息区，使用导航蓝。LB角：在S0/S1态显示音乐控制条（紫色），S2/S3态隐藏或仅显示通知图标。ADAS预警时，对应角落（如LB角为左侧盲区）将用**预警红色**进行高频、高亮度闪烁，覆盖该角原有信息，形成强烈的空间方位感。

**2. 状态机与优先级模型**
- **[判断·高] 定义7个互斥主状态：`BOOT`(启动)、`IDLE`(待机)、`NAV`(导航)、`TEAM`(组队)、`CRUISE`(巡航信息)、`PARKED`(停车)、`SUMMARY`(行程总结)。** — 基于：CTO建议，避免状态爆炸。音乐、设置作为从`IDLE`或`PARKED`状态调出的面板，不占用主状态。
- **[判断·高] 将`ADAS`、`CALL`、`VOICE`、`REC_EVENT`定义为覆盖层(Overlay)。** — 基于：CTO建议。覆盖层拥有明确的优先级：`ADAS` > `CALL` > `VOICE`。高优先级覆盖层进入时，会冻结或压制低优先级层及底层主状态的非关键信息动画。
- **[判断·高] S0-S3速度等级作为“渲染密度参数”输入。** — 基于：CTO建议。状态机根据当前速度等级（可由剧本或控制台设置）决定各区域信息的显示粒度。例如，S3高速态下，LB角音乐条、RB角导航详情文字均隐藏，仅保留核心箭头和数字。

**3. 视觉风格与功能色系统**
- **[偏好·--] 采用高对比度、无衬线字体，确保一瞥可读。** 主数字字体粗壮，色彩语义明确：
    - **速度/状态：白色** — 中性，高优先级。
    - **导航：蓝色 (#4A90E2)** — 认知关联性强。
    - **预警：红色 (#FF3B30)** — 最高警示，配合闪烁。
    - **组队：青色 (#5AC8FA)** — 区别于导航。
    - **音乐：紫色 (#AF52DE)** — 娱乐相关。
- **[判断·高] 在控制台模拟A/B光学方案视觉差异。** — 基于：P0问题清单。提供切换开关：
    - **方案A（单色高对比）**：模拟单色MicroLED，整体去色，提高对比度和锐利度。
    - **方案B（全彩波导）**：启用全彩，但添加轻微高斯模糊和色彩溢光效果，模拟光波导的视觉特性。

**4. 演示叙事与自动剧本**
- **[判断·高] 实现基于时间轴的剧本引擎。** — 基于：CTO建议的类似系统经验。剧本是一个步骤数组，每个步骤包含`delay`(延迟)、`action`(触发事件，如`setState:NAV`、`triggerOverlay:ADAS_LEFT`)。引擎驱动游标，支持暂停、跳步。
- **[偏好·--] 设计一个3分钟核心叙事剧本：**
    1.  **启动与待机 (0-30s)**：展示开机自检动画与品牌标识，进入`IDLE`态，显示完整四角信息（S0态）。
    2.  **语音发起导航 (30-60s)**：触发`VOICE`覆盖层，显示“正在搜索去往雁栖湖的路线...”，然后切换到`NAV`状态，RB角出现蓝色导航信息。
    3.  **城市骑行与ADAS保护 (60-120s)**：剧本将速度等级提升至S1/S2。先后触发**左侧盲区预警(BSD)**（LB角红色闪烁）和**前向碰撞预警(FCW)**（LT/RT角同步红色闪烁），强烈展示“被保护着”。
    4.  **高速巡航与信息收敛 (120-180s)**：速度升至S3，直观展示信息极度收敛（仅速度、预警），强调安全优先。最后触发`REC_EVENT`覆盖层，提示“紧急事件已保存”，展示行车记录仪价值。

**5. 交互与控制台**
- **[判断·高] 实现功能完整的可视化控制台（默认隐藏，按 ~ 键呼出）。** — 基于：P0问题清单。控制台包含：
    - **底部时间轴**：显示剧本进度，可暂停、拖动。
    - **右侧沙盒面板**：按钮手动触发所有状态和覆盖层。
    - **A/B光学方案切换**。
    - **速度等级S0-S3滑块**。
- **[判断·高] 定义完整的键盘快捷键矩阵。** — 基于：CTO建议。例如：数字键1-7切换主状态，F1-F4触发ADAS预警，空格键播放/暂停剧本，`~`键切换控制台。输入路由层统一处理，确保与剧本播放的优先级协调。

**6. 技术实现策略**
- **[判断·中] 采用“分段生成，最终合并”的单文件策略。** — 基于：>500行代码质量风险。将代码按模块（配置枚举、状态机、渲染引擎、剧本引擎、样式、HTML骨架）分别生成，再合并为一个零外部依赖的HTML文件，确保`file://`协议下完美运行。
- **[偏好·--] 使用纯DOM+CSS实现，非Canvas。** 便于分段生成、调试和状态控制。所有图形使用内联SVG或CSS绘制。

## 自评：最大风险
**最大风险在于“分段生成代码的集成复杂度与最终文件稳定性”。** 虽然分段生成规避了单次生成的质量陷阱，但将多个模块（状态机、渲染、剧本、UI）手动或通过提示合并为一个完全自洽、无冲突的单文件HTML，过程极易出错。特别是状态机与渲染层、剧本引擎与键盘事件路由之间的接口契约，任何一处定义模糊或实现偏差都可能导致演示时出现状态不同步、优先级错乱或剧本卡死，严重损害演示可信度。必须在生成每个模块时明确定义其输入输出，并在最终合并后进行严格的功能测试（如所有键盘快捷键、所有剧本步骤）。
