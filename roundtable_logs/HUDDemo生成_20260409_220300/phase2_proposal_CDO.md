# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在创建一个用于供应商与投资人演示的、可双击运行的HTML文件，以展示智能骑行头盔HUD的核心体验。方案严格遵循“余光扫视，一瞥可读”原则，并围绕“消灭停车掏手机”的核心使命构建。

**1. 视觉架构与布局**
- [偏好·--] **四角布局与功能分区**：采用“右下主信息区 + 顶部状态条 + 中央瞬时弹层”的骨架。右下角为**巡航态**核心信息区，固定显示速度（白色）、时间、电量。顶部状态条用于低优先级提示，如组队连接状态（青色）、音乐播放（紫色）。中央区域为高优先级信息覆盖层，专用于**导航指引（蓝色）**与**ADAS安全预警（红色）**。
- [判断·高] **信息密度与层级控制**：严格遵循竞品已验证的极简范式。任何时刻，主信息区（右下）仅显示1-3个关键数据项；中央弹层一次只呈现一个核心任务（如“前方300米右转”或“左侧盲区有车”），确保一瞥可读 — 基于：HUD信息架构必须极简的约束与摩托车骑行注意力模型。
- [偏好·--] **全彩功能色语义系统**：建立强关联的色彩编码：速度/基础信息（#FFFFFF）、导航（#4A90E2）、ADAS预警（#FF3B30）、组队通信（#5AC8FA）、音乐控制（#AF52DE）。色彩不仅是装饰，更是信息类型的首要视觉过滤器。

**2. 交互内核：状态机与优先级**
- [判断·高] **七态有限状态机**：定义七个核心页面态：1) 待机/开机、2) 基础巡航、3) 主动导航、4) 语音交互、5) 组队对讲、6) 行车记录、7) ADAS预警。其中，**ADAS预警态为最高优先级**，可打断除自身外的任何其他状态 — 回应约束：[CTO 7] 与 [CDO 7]。
- [判断·高] **事件驱动与自动剧本**：交互内核由统一的状态机管理。支持两种演示模式：**手动模式**（通过键盘快捷键触发状态切换）与**自动剧本模式**（按预定义时序自动演示）。两者共享同一套状态变更函数，确保体验一致 — 基于：[CTO 8] 状态机工程经验。
- [事实·高] **键盘事件全覆盖**：预定义“场景动作映射表”，例如：`N`键启动导航，`B`键触发盲区预警，`S`键模拟语音唤醒，`Space`键暂停/继续自动剧本。所有监听逻辑集中管理 — 来源：议题专属指导中CTO对键盘事件的要求。

**3. 核心体验场景与叙事节奏**
- [判断·中] **自动剧本设计**：剧本采用声明式JSON数组定义，每个步骤包含时间戳、触发事件和参数。演示流程约2分30秒，节奏如下：
    1.  (0-10s) **开机与品牌展示**：简约动画，突出“安全守护”的科技感，进入基础巡航态。
    2.  (10-40s) **核心巡航信息**：展示速度、时间、电量，顶部提示组队已连接（青色）。
    3.  (40-80s) **语音发起导航**：模拟语音唤醒（“导航去西山森林公园”），中央弹出蓝色导航卡片，显示路线概览，随后右下角进入简洁的逐向导航模式（距离、箭头）。
    4.  (80-110s) **ADAS预警打断**：在导航过程中，突然触发左侧盲区监测（BSD）预警。中央瞬间被红色动态闪烁的警示图标和“左侧有车”文字覆盖，导航信息暂时淡出。3秒后预警解除，导航信息平滑恢复。
    5.  (110-150s) **多模态交互**：模拟来电接听（顶部提示）、音乐播放与控制（紫色图标）。展示信息优先级：预警 > 导航 > 通信 > 媒体。
    6.  (150-170s) **回归巡航**：所有任务结束，界面恢复至最简洁的基础巡航态，强调“信息来去无痕，守护始终在线”。
- [判断·高] **预警的视觉冲击力**：ADAS预警采用**中央位置 + 高饱和红色 + 高频微闪烁 + 模拟震动效果的边框动画**组合。这是视觉设计的最高优先级，必须瞬间抢占骑手余光 — 基于：[CDO 7] 对“被保护着”核心差异化的判断与安全信息优先级原则。

**4. 技术实现与视觉表达**
- [判断·中] **纯前端技术栈**：使用纯HTML/CSS/JS（SVG用于图标）实现，所有资源内联，确保单HTML文件双击运行无外部依赖 — 基于：[CTO 13] 本地文件运行约束与 [CTO 11] 维护成本考量。
- [判断·低] **A/B光学方案表达**：在Demo中通过一个隐藏的调试面板（`D`键触发）模拟两种显示效果：“方案A（高对比）”使用更高饱和度和纯黑背景；“方案B（广色域）”则使用稍浅的背景和更丰富的色彩渐变。这旨在向供应商展示我们对光学性能差异的考虑 — 基于：[CDO 5] 对供应商差异的个人推测。
- [事实·高] **字体与可读性**：采用系统默认无衬线字体族（如 `system-ui, -apple-system`），确保通用性。关键数据（如速度、预警文字）使用字重（`font-weight: 700`）和放大尺寸（速度字体约为其他信息的2倍）突出 — 来源：竞品分析中强调的关键字体尺寸要求。

- **决策4：内联所有资源并采用纯CSS/JS实现** → 回应约束：[CTO 1]（置信度：[事实·高]）、[CTO 11]（置信度：[判断·中]）、[CTO 13]（置信度：[判断·中]）。这是满足“双击打开、零依赖”硬性技术约束的唯一可靠路径。
- **决策5：严格限制单屏信息元素与色彩语义编码** → 回应约束：[CDO 1]（置信度：[事实·高]）、[CDO 2]（置信度：[判断·高]）、[CDO 6]（置信度：[偏好·--]）。这是保障“一瞥可读”体验下限的基石设计原则。

## 自评：最大风险
**最大风险在于“视觉方案的实现效果与真实光学硬件显示效果存在显著差距”**。
- [判断·中] 本Demo在标准显示器上渲染，而真实HUD的光学方案（如衍射光波导）在亮度、对比度、色域、视场角、重影、眼动范围（Eyebox）等方面有独特限制和表现。Demo中精心设计的色彩语义（如特定红色的警示效果）和微妙的透明度叠加，在真实硬件上可能无法完美还原，甚至出现可读性问题。
- **缓解措施**：1) 在Demo中提供简单的A/B显示模式切换，初步示意对光学差异的考虑；2) 在视觉规范中明确标注“此设计需与光学工程师就具体硬件参数进行联合校准”；3) 尽早与至格科技等供应商沟通，获取其光学模拟器或SDK，将设计验证前置。
