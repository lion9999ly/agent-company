# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在构建一个**单HTML文件、零依赖**的HUD演示系统，以视觉化呈现“智能骑行座舱”的核心体验。方案严格遵循“四角布局”原则，并围绕“消灭停车掏手机”的核心使命，通过一个精心编排的**自动剧本**，串联起导航、ADAS预警、组队、音乐等核心场景，最终交付一个可供投资人/供应商现场演示的、稳定且具有冲击力的交互原型。

**1. 视觉架构：四角布局与功能色语义**
- **布局骨架**：严格采用“右下主信息区 + 顶部状态条 + 左下/右上辅助信息区”的固定四角布局。中央视野始终保持纯净，仅允许瞬时、高优先级的全屏预警层（如碰撞预警）以半透明遮罩+中央大图标形式短暂覆盖。
    - **右下角 (RB)**：**导航核心区**。显示下一路口箭头、距离、路名。采用**导航蓝**作为主色调。
    - **左下角 (LB)**：**速度与基础状态区**。显示当前车速（大号白色数字）、时间、连接状态。
    - **右上角 (RT)**：**情景信息区**。动态显示当前激活模式，如“组队中”（青色）、“音乐播放中”（紫色）。
    - **顶部 (Top)**：**轻提示与系统状态条**。显示信号强度、电量、麦克风状态等。ADAS预警（如BSD）将在此区域以**预警红**进行方向性闪烁提示（如左侧BSD预警，则左侧边框高亮红色并闪烁）。
- **功能色语义体系**：
    - **速度白** (#FFFFFF)：用于车速、时间等核心读数，确保最高可读性。
    - **导航蓝** (#4A90E2)：用于所有导航相关元素（箭头、路名）。
    - **预警红** (#FF3B30)：用于所有安全预警（BSD、FCW、开门杀），通过闪烁和边框高亮增强冲击力。
    - **组队青** (#5AC8FA)：用于组队通信相关状态与提示。
    - **音乐紫** (#AF52DE)：用于音乐播放控制与状态显示。
- **A/B光学方案表达**：在Demo的“右侧沙盒面板”中提供“光学方案”切换开关。选择“A方案（全彩OLED）”时，UI正常显示全彩。选择“B方案（单绿光波导）”时，通过CSS滤镜将所有UI色彩转换为不同明度的绿色阶，并模拟可能的亮度与对比度下降，直观展示视觉差异。

**2. 交互与状态机：7个页面态与优先级仲裁**
- **状态定义**：将7个页面态建模为**一个基础态 + 多个瞬时/覆盖态**，通过统一的状态机调度器管理优先级。
    1.  **基础态 (Base)**：常态骑行界面，包含四角布局的所有基础信息。
    2.  **导航态 (Navigation)**：在基础态上，强化右下角导航信息。
    3.  **音乐控制态 (Music)**：语音触发音乐播放/切换时，在左下角短暂浮现大号播放/切歌图标。
    4.  **组队通话态 (Intercom)**：右上角显示“正在通话”图标与发言人标识。
    5.  **ADAS预警态 (Alert)**：高优先级。根据预警类型（左/右BSD、前向FCW）在对应角落（顶部或全屏）触发红色闪烁提示。
    6.  **语音聆听态 (Voice Listening)**：当用户唤醒语音时，中央下方出现动态麦克风波纹动画。
    7.  **系统通知态 (Notification)**：低优先级提示，如“蓝牙已连接”，在顶部状态条短暂显示。
- **优先级逻辑**：ADAS预警态 > 语音聆听态 > 组队/音乐控制态 > 导航态 > 基础态 > 系统通知态。高优先级状态可中断低优先级状态的显示。

**3. 演示引擎：自动剧本与沙盒控制**
- **自动剧本**：设计一个约90秒的自动演示循环，模拟一次完整的骑行旅程。剧本节奏如下：
    1.  **(0-15s) 开场与导航**：HUD启动，显示基础态，随后语音触发“导航去西山”，右下角出现导航信息。
    2.  **(16-40s) 城市骑行与预警**：模拟车速提升（S1->S3），触发右侧BSD预警（顶部右侧红闪），随后触发“开门杀”预警（全屏红色闪烁+警示图标）。
    3.  **(41-60s) 组队出游**：语音指令“加入车队”，右上角变为组队青，显示队友位置。
    4.  **(61-75s) 娱乐与语音**：语音指令“播放音乐”，左下角浮现音乐图标并显示歌名。
    5.  **(76-90s) 夜间模式与结束**：模拟进入隧道，UI亮度自动调节（变暗），剧本结束，恢复基础态。
- **右侧沙盒面板**：一个始终浮于右侧的调试面板，提供：
    - **剧本控制**：播放/暂停/跳转至关键场景。
    - **状态注入**：手动触发任意ADAS预警、模拟车速变化、切换光学方案。
    - **调试可视化**：实时显示当前活跃状态、优先级队列。
- **键盘快捷键矩阵**：全覆盖键盘控制，例如：空格键（播放/暂停剧本）、数字键1-7（直接跳转至对应状态）、A/B键（切换光学方案）、方向键（模拟BSD预警方向）。

**4. 技术实现与文件结构**
- 采用**单HTML文件**，内联所有CSS样式和JavaScript代码。
- 代码结构分为清晰区块：`<style>`（包含所有样式、A/B方案滤镜）、`<script>`（包含状态机State Machine、渲染器Renderer、剧本引擎ScriptEngine、输入处理器InputHandler）、`<body>`（包含HUD的DOM骨架和右侧沙盒面板的DOM）。
- 渲染逻辑基于**数据驱动**。当前状态和所有动态数据（如车速、导航信息）存储在一个中央`state`对象中，由`render()`函数根据此对象统一绘制四角UI，避免DOM操作混乱。

## 决策点回应
- **决策1：采用严格的“四角布局”与中央留空原则** → 回应约束：[CDO 3]（[事实·高]）、[CDO 6]（[判断·高]）。这是确保信息一瞥可读、避免认知过载与视线遮挡的基石。
- **决策2：建立“预警红”等全彩功能色语义，并在四角通过边框闪烁实现方向性预警** → 回应约束：[CDO 7]（[偏好·--]）、[CDO 8]（[判断·高]）、[CMO 1]（[事实·高]）。这直接服务于“被保护着”的核心差异化，并满足CMO对ADAS震撼效果的展示要求。
- **决策3：将7个页面态建模为“基础态+覆盖态”，并由统一状态机调度** → 回应约束：[CTO 11]（[判断·高]）、[CTO 15]（[判断·中]）、[CTO 2]（[事实·高]）。此架构确保了优先级逻辑清晰、演示稳定，且符合单页面应用要求。
- **决策4：设计包含叙事节奏的“自动剧本”与“右侧沙盒面板”** → 回应约束：[CDO 4]（[事实·高]）、[CDO 10]（[判断·中]）、[CTO 4]（[事实·高]）、[CMO 2]（[判断·高]）。剧本将功能转化为故事，沙盒面板保障了演示的灵活性与可控性。
- **决策5：在单文件内实现数据驱动的渲染与A/B光学方案模拟** → 回应约束：[CTO 1]（[事实·高]）、[CTO 13]（[判断·中]）、[CDO 5]（[事实·高]）、[CDO 9]（[事实·中]）。这是满足零依赖、可维护性要求，并直观表达技术约束对视觉影响的最优解。

## 自评：最大风险
**最大风险在于单文件内JavaScript状态机与渲染逻辑的复杂度控制**。在零依赖、且需支持状态优先级仲裁、自动剧本驱动、键盘控制、沙盒调试注入等多重要求下，代码量容易膨胀并超出LLM单次生成的质量稳定范围。即使分段生成，合并后的代码在浏览器中的执行效率、事件冲突排查也可能存在问题，可能导致现场演示时出现卡顿或状态错乱。**缓解措施**：极度简化数据模型，为状态机设计一个极其清晰、无嵌套的扁平状态转换表；在剧本引擎中避免使用复杂的`setTimeout`链，转而采用基于时间戳的单一循环驱动；并在最终合并后，进行严格的人工代码审查与多浏览器测试。
