# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在创建一个零依赖、单HTML文件的HUD交互演示，用于向供应商和投资人直观展示智能骑行座舱的核心体验。方案严格遵循“信息一瞥可读”的最高准则，围绕“消灭停车掏手机”的使命，通过一个结构清晰、可交互的Demo，展现从开机到骑行中关键场景的完整叙事。

**1. 视觉架构与布局**
- [偏好·--] **严格四角布局**：采用固定四角信息分区，确保余光扫视效率。
    - **左上角 (速度/状态)**：显示当前速度（大字体）、时间、电量/信号。速度值采用纯白色 (`#FFFFFF`)。
    - **右上角 (导航)**：显示下一个转向指令（图标+距离/路名）、预计到达时间。采用导航蓝色 (`#4A90E2`)。
    - **左下角 (通知/媒体)**：显示来电人、音乐播放控制（曲名/艺术家）。来电用高亮边框，音乐用紫色 (`#9B59B6`)。
    - **右下角 (ADAS/系统)**：显示组队状态（青色 `#1ABC9C`）、行车记录仪录制状态。ADAS预警时，此区域边框将高亮闪烁。
- [偏好·--] **中央视野100%留空**：严格遵守安全约束，中央区域仅用于模拟骑行背景（CSS动态渐变模拟道路移动感），禁止任何固定或弹层内容。
- [判断·高] **预警视觉方案**：基于约束24，预警与四角UI强绑定。
    - **FCW (前向碰撞预警)**：触发时，**左上角（速度）** 和 **右上角（导航）** 同时进行红框 (`#FF3B30`) 高频闪烁（频率：2Hz），模拟对前方威胁的全局注意力吸引。
    - **BSD (盲区监测) / 开门杀预警**：触发时，在威胁来源对应的**左下角（左侧威胁）** 或**右下角（右侧威胁）** 进行红框高频闪烁，提供方向性提示。
    - 所有闪烁动画采用 `box-shadow` 和 `border-color` 交替，确保在单绿色光学方案下仍可通过明暗变化识别。

**2. 状态机与交互逻辑**
- [判断·高] **三层状态架构**：采用 `basePageState` + `overlayState` + `modeState` 架构，回应约束6。
    - `basePageState`: 7个固定页面态 (`BOOT`, `HOME`, `NAVI`, `CALL`, `TEAM`, `MUSIC`, `RECORD`)。`HOME` 态为四角信息全显的基础态。
    - `overlayState`: 存放最高优先级的瞬时状态，如 `FCW_ALERT`, `BSD_LEFT_ALERT`, `BSD_RIGHT_ALERT`, `DOOR_ALERT`。其触发会打断并暂存当前 `basePageState`。
    - `modeState`: 管理演示模式，如 `MANUAL` (键盘控制) 或 `AUTO` (剧本播放)。
- [判断·高] **7级优先级仲裁与状态栈**：基于约束23，明确定义状态抢占顺序：`预警 > 来电 > 导航 > 组队 > 音乐 > 录制 > 主界面`。`overlayState`（预警）具有最高优先级，触发时将当前 `basePageState` 压入“状态栈”，预警结束后按 `resumePolicy` 恢复，回应约束20。
- [判断·中] **速度自适应信息降噪**：基于约束3与27，定义速度等级（S0:静止， S1:0-30km/h， S2:30-80km/h， S3:80+km/h）。在S2/S3高速下，自动隐藏左下角音乐详情、右下角非关键状态等次要信息，仅保留核心数据（速度、关键导航指令、预警），确保界面极度简洁。

**3. 演示与剧本系统**
- [判断·高] **双模式演示**：Demo包含“沙盒模式”（手动探索）和“剧本模式”（自动演示）。
    - **右侧控制面板 (沙盒模式)**：提供按钮和滑块，用于手动切换7个 `basePageState`、触发各类预警、调节模拟亮度/对比度、切换A/B光学方案。
    - **底部时间轴 (剧本模式)**：显示预编剧本的时间轴，支持播放、暂停、跳转至特定场景。
- [判断·中] **沙盒接管协议**：基于约束17，定义明确的模式互斥规则。当用户使用键盘或控制面板进行手动操作时，自动剧本会自动暂停，并弹出提示，防止状态冲突。
- [判断·高] **内嵌剧本数据**：基于约束21，将自动演示剧本定义为JS常量数组。每个剧本事件遵循统一结构 `{at, type, payload, duration, interruptible, resumePolicy}`，回应约束28。剧本将依次演示：开机动画 -> 主界面 -> 语音设置导航 -> 骑行中音乐播放 -> 触发BSD预警 -> 触发FCW预警 -> 模拟组队通话 -> 触发开门杀预警 -> 结束。

**4. 视觉差异化与细节**
- [偏好·--] **A/B光学方案一键切换**：在控制面板提供“全彩模式”与“单绿模式”切换按钮。切换时，界面所有功能色将映射为单绿色 (`#00FF00`) 的不同明度/饱和度，以演示在不同硬件方案下的视觉表现。
- [判断·中] **亮度/对比度模拟**：提供滑块，可调整整个HUD画布的 `filter: brightness(X%) contrast(Y%)`，模拟不同环境光下的可视性，回应CMO建议。
- [偏好·--] **极简开机动画**：`BOOT` 状态展示简洁的品牌Logo浮现与系统自检进度条，历时3秒，强调可靠性与科技感，避免炫技。

**5. 技术实现与交付**
- [判断·高] **单文件零依赖**：所有HTML、CSS、JS代码整合于一个文件内。CSS使用 `<style>` 标签，JS逻辑封装在 `<script>` 标签内，通过函数和对象组织，确保双击 `demo.html` 即可在任意现代浏览器中运行。
- [判断·中] **分段代码生成策略**：为避免单次生成超500行HTML的质量陷阱，将指示LLM按“基础DOM结构 -> 核心样式 -> 状态机逻辑 -> 剧本数据 -> 控制面板逻辑”的顺序分段生成代码，最后拼接。
- [判断·高] **键盘事件全覆盖**：定义完整的键盘映射表（如 `数字键1-7` 切换7个基础状态， `F键` 触发预警， `空格键` 播放/暂停剧本），键盘事件监听器具有最高优先级，可直接驱动状态机。

## 决策点回应
- **决策1：采用严格四角布局并绑定预警方向** → 回应约束：[CDO 4, 24], [CMO 2]（置信度：[判断·高]） — 这是实现“一瞥可读”和安全警示最直接的视觉范式，且与竞品分析结论一致。
- **决策2：实现三层状态机与优先级仲裁栈** → 回应约束：[CDO 6, 11, 20, 23], [CTO 2, 8]（置信度：[判断·高]） — 这是解决7个页面态与高优先级预警之间复杂切换关系的唯一可维护架构，是Demo交互逻辑的核心。
- **决策3：设计沙盒/剧本双模式与接管协议** → 回应约束：[CDO 16, 17], [CTO 4, 9]（置信度：[判断·中]） — 平衡了自动演示的流畅性与投资人手动探索的需求，避免了演示中的控制冲突。
- **决策4：将视觉样式（色彩、闪烁）与A/B光学方案解耦** → 回应约束：[CDO 22], [CMO 4]（置信度：[判断·中]） — 通过CSS变量和滤镜，用一套代码灵活演示两种硬件方案的视觉效果，高效回应了供应商关切。
- **决策5：内嵌剧本数据并采用统一事件结构** → 回应约束：[CDO 21, 28], [CTO 11]（置信度：[判断·高]） — 在满足零依赖交付的同时，保证了剧本系统的可维护性和可扩展性。

## 自评：最大风险
**最大风险在于状态机与剧本调度器的逻辑复杂性**。在单文件、零依赖的约束下，需要在一个 `<script>` 标签内清晰实现“状态仲裁”、“事件队列”、“剧本解释器”、“UI渲染映射”和“键盘/控制面板事件处理”等多层逻辑。任何一层的状态同步错误（例如，预警结束后未能正确恢复被中断的音乐界面，或剧本播放时未正确处理手动键盘输入）都会导致演示卡顿或行为异常，严重影响展示效果。这高度依赖代码的结构严谨性和LLM对复杂逻辑的生成能力。**缓解措施**：要求LLM为每个核心模块（如状态管理器、剧本播放器）输出清晰的流程图或伪代码作为注释，并优先保证手动键盘操作的绝对响应优先级，即使剧本逻辑暂时不完美，也能通过手动操作完成演示。
