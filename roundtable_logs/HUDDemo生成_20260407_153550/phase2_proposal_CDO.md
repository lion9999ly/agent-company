# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在创建一个高度沉浸、技术可靠的单文件HTML HUD演示，用于向供应商和投资人展示智能骑行座舱的核心体验。方案严格遵循“一瞥可读”原则，聚焦于消灭停车掏手机的核心使命，并通过精心编排的叙事节奏，突出“被保护着”的差异化价值。

**1. 视觉架构与布局**
- [偏好·--] 采用 **“四象限信息锚点”布局**，将视野划分为四个功能区域，信息密度由中心向外围递减。
    - **左上角**：状态区。显示时间、连接状态（蓝牙/Mesh）、电量。极简图标+小字号。
    - **右上角**：环境区。显示当前限速、ADAS系统状态（如“保护中”）。使用中性色。
    - **右下角**：主行动区。核心导航信息（转向箭头、距离、路名）与当前速度。这是视线余光最易捕捉的区域，字体最大，对比度最强。
    - **左下角**：次要信息/媒体区。显示音乐播放信息（如歌名）或组队通话状态。使用低饱和度色彩。
    - **中央区域**：保留为**瞬时弹层区**，仅用于最高优先级的ADAS预警和关键系统通知（如来电）。平时为空。
- 此布局回应了约束 **[CDO 1, 3]** 和 **[CMO 2]**，确保信息层级清晰，余光可扫视。

**2. 色彩语义与视觉冲击力**
- [偏好·--] 建立严格的全彩功能色体系：
    - **速度白 (#FFFFFF)**：用于速度、时间等核心读数，确保最高亮度和可读性。
    - **导航蓝 (#4A90E2)**：用于所有导航相关元素（箭头、路线、距离）。
    - **预警红 (#FF3B30)**：专用于ADAS碰撞预警（FCW、BSD、开门杀）。触发时，在危险方向对应的视野边缘（左/右/中央）迸发高频率闪烁的红色弧光或警示图标，并伴随模拟的震动提示音效（通过CSS动画模拟）。这是视觉冲击力的最高点。
    - **组队青 (#5AC8FA)**：用于Mesh组队通信状态与语音波形。
    - **音乐紫 (#AF52DE)**：用于音乐播放控制与信息。
- 此体系回应了约束 **[CDO 5, 6]** 和 **[CMO 1]**，用色彩建立直觉，并确保预警具有压倒性的注意力捕获能力。

**3. 状态机与页面态设计**
- [判断·高] 定义 **7个核心页面态** 及其优先级（数字越小优先级越高）：
    1.  **紧急预警态** (优先级1)：全屏或半屏红色闪烁预警，覆盖一切。
    2.  **主动导航态** (优先级2)：右下主行动区显示详细导航信息。
    3.  **来电/通信态** (优先级3)：中央弹层显示来电者，提供语音接听/挂断选项。
    4.  **音乐播放态** (优先级4)：左下角显示音乐信息，中央可轻量显示播放控制（短暂）。
    5.  **系统通知态** (优先级5)：顶部轻提示条显示低优先级通知（如“头盔已连接”）。
    6.  **待机态** (优先级6)：仅显示时间、速度、状态等基础信息。
    7.  **启动/自检态** (优先级7)：开机时的品牌动画与系统自检界面。
- [判断·高] 采用 **“全局状态管理 + 优先级仲裁”** 架构。一个中央状态机管理当前`activeState`。任何触发事件（来自自动剧本或键盘）都会提交一个目标态及其优先级。状态机比较优先级，决定是否切换。这避免了多个布尔变量管理的混乱，回应了约束 **[CTO 2, 7, 8]**。

**4. 交互与自动剧本叙事**
- **键盘交互全覆盖**：设计直观的键盘映射（如 `1-7` 键直接切换7个状态，`A` 键触发自动剧本，`空格` 模拟语音指令确认，方向键模拟预警方向）。回应 **[CTO 3]**。
- **自动剧本时序驱动**：编写一个预设的、约90秒的剧本，模拟一次完整的骑行：
    1.  (0-5s) **启动与自检**：展示简洁的开机动画与系统检查，建立可靠感。
    2.  (5-20s) **巡航待机**：进入待机态，展示基础信息。
    3.  (20-35s) **语音发起导航**：通过模拟语音输入和确认，切换到主动导航态，展示“30秒内完成操作”。
    4.  (35-50s) **音乐播放**：自动播放音乐，展示左下角音乐信息。
    5.  (50-65s) **Mesh组队加入**：展示组队青色的连接提示。
    6.  **(高潮 65-80s) ADAS预警连环演示**：
        - 先触发**右侧盲区预警**（右侧红色闪烁）。
        - 紧接着模拟**前车碰撞预警**（中央红色闪烁+距离快速减少）。
        - 最后演示**左侧开门杀预警**（左侧红色闪烁）。
    7.  (80-90s) **恢复巡航**：预警解除，系统恢复至导航态，最后淡出至待机态。
- 剧本设计回应了 **[CDO 8]**、**[CMO 3, 7]** 和 **[CTO 4]**，并强化了核心卖点。
- [判断·中] **自动剧本与手动操作的互斥**：当用户按下任何交互键时，自动剧本立即暂停并重置。按下`A`键则重新开始剧本。这解决了人机冲突，回应了 **[CTO 9]**。

**5. 技术实现与视觉模拟**
- 使用纯HTML/CSS/JS，所有资源内联，确保双击运行。回应 **[CTO 1]**。
- 采用分段生成策略：先构建HTML骨架和基础CSS，再注入JS状态机逻辑，最后注入剧本数据。回应 **[CTO 6, 10]**。
- 通过CSS `filter: brightness()` 和半透明黑色遮罩层，模拟 **A/B光学方案在强光下的表现**（如亮度不足时信息变淡）和**夜间模式**（整体降低亮度，提高对比度）。这回应了 **[CDO 4, 7]** 和 **[CMO 4]**。
- 所有动态效果（预警闪烁、导航箭头跳动、音乐波形）均使用CSS `@keyframes` 实现，性能高效且无需外部依赖。

## 决策点回应
- **决策1：采用“四象限锚点”布局而非完全对称布局** → 回应约束：[CDO 1, 3]（置信度：[判断·高]）。基于摩托车头盔HUD的竞品研究，右下角是主信息的最佳余光捕捉区。
- **决策2：将ADAS预警设为最高优先级(1)并赋予最强视觉表现** → 回应约束：[CDO 6], [CMO 1]（置信度：[判断·高]）。这是产品“被保护着”核心理念的直接体现，必须在演示中给予最震撼的呈现。
- **决策3：使用中央状态机与优先级仲裁管理7个页面态** → 回应约束：[CTO 2, 7, 8]（置信度：[判断·高]）。这是满足状态机完整性、避免逻辑混乱的唯一工程合理路径。
- **决策4：自动剧本以ADAS预警连环演示为高潮** → 回应约束：[CDO 8], [CMO 3]（置信度：[判断·中]）。基于市场演示需要，必须在短时间内给观众留下最深刻的差异化印象。
- **决策5：手动操作打断并重置自动剧本** → 回应约束：[CTO 9]（置信度：[判断·中]）。基于交互演示的常见问题类推，这是保证演示可控性的有效规则。

## 自评：最大风险
**最大风险在于“视觉冲击力”与“一瞥可读性”的平衡可能被技术实现所削弱。**
- [判断·中] 在单HTML文件内，仅用CSS和JS模拟HUD在真实光学镜片上的亮度、通透感、虚像距离和强光下的可视性，存在巨大挑战。过度复杂的闪烁和动画可能在实际浏览器渲染中导致卡顿，反而破坏沉浸感和流畅感，这与“安全可靠”的品牌感知相悖。
- 缓解措施：严格控制动画的复杂度和数量，优先保证性能；在演示开始前以文字说明“此为概念模拟，实际光学效果以硬件为准”。
