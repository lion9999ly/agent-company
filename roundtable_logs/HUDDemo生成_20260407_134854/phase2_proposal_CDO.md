## 方案描述

本方案旨在生成一个零依赖、单HTML文件的HUD演示系统，用于向供应商和投资人直观展示智能骑行座舱的核心体验。方案严格遵循“消灭停车掏手机”的使命，聚焦于HUD在骑行中的信息呈现、ADAS安全预警与状态管理。

**1. 整体架构与状态机设计**
- [判断·高] 采用“单页应用 + 内置状态机 + 定时器剧本引擎”的架构。所有代码（HTML/CSS/JS）内置于一个文件，确保通过`file://`协议双击即可运行 — 回应约束：[CTO 1, CTO 9]。
- [判断·高] 状态机模型严格区分为三层：**基础态**（巡航、导航、组队）、**瞬时覆盖态**（来电、语音、音乐）、**告警抢占态**（ADAS预警）。这七种状态共同构成一个完整的优先级系统，通过一个全局的`currentState`对象进行管理，并由统一的`render()`函数根据优先级和《7态 x 4角 渲染映射矩阵表》驱动四角UI更新 — 回应约束：[CDO 3, CTO 2, CTO 10]。
- [偏好·--] 使用原生JavaScript手写状态机逻辑，不引入任何外部框架，以最大化代码透明度和可控性 — 回应约束：[CTO 15]。

**2. 视觉与交互设计**
- [事实·高] **四角信息骨架**：严格遵循LT（速度）、RT（设备/通知）、LB（组队/录制）、RB（导航）的布局。这是建立肌肉记忆的基础，所有状态渲染均以此为准 — 回应约束：[CDO 1]。
- [事实·高] **中央视野留空**：所有信息仅出现在四角，中央视野区域保持绝对干净，确保骑行安全 — 回应约束：[CDO 7]。
- [判断·高] **信息极简与一瞥可读**：所有文本使用大字号、高对比度。例如，速度数字使用超大字体，导航信息仅显示下一个转向箭头和距离。避免任何复杂图表或地图 — 回应约束：[CMO 2, CTO 14]。
- [判断·高] **全彩功能色语义**：
    - **速度白** (`#FFFFFF`): LT区域主速度显示。
    - **导航蓝** (`#00A3FF`): RB区域导航箭头、路名。
    - **预警红** (`#FF3B30`): ADAS预警图标与边框闪烁。
    - **组队青** (`#00D4AA`): LB区域组队成员状态。
    - **音乐紫** (`#AF52DE`): RT区域音乐通知图标。
    色彩用于快速区分信息类型，增强余光识别效率 — 回应约束：[议题专属指导]。
- [事实·高] **A/B光学方案切换**：通过一个全局CSS滤镜实现。默认全彩模式。当切换到“单绿光波导”模式时，应用`filter: sepia(1) saturate(1000%) hue-rotate(60deg)`，模拟单绿色显示效果，以展示硬件适配能力 — 回应约束：[CDO 9]。
- [判断·高] **ADAS预警方位感**：预警UI不仅变色，更通过**方位激活**强化空间直觉。例如，“左后方盲区车辆”预警会**强烈激活并闪烁LB区域**的红色边框与图标；“前向碰撞预警”则同时激活LT和RT区域。方位与UI区域严格对应 — 回应约束：[CDO 2]。

**3. 控制与演示系统**
- [事实·高] **双层键盘映射系统**：
    - **全局控制层**：例如，`Tab`键呼出/隐藏右侧调试面板，`D`键切换A/B光学方案。
    - **场景动作层**：数字键`1-7`直接切换七种状态，字母键模拟事件（如`C`来电，`B`急刹预警）。所有映射关系在《键盘事件-状态机映射表》中明确定义 — 回应约束：[CDO 5, CTO 3, CTO 11]。
- [事实·高] **自动剧本引擎**：预置“前车急刹”、“左后方超车”、“右后方盲区”三个紧急场景剧本。每个剧本是一个时间轴事件序列（例如：`{time: 2000, action: 'trigger', payload: {state: 'cruise'}}`）。
    - 引擎支持`start()`、`pause()`、`resume()`、`reset()`、`step()`、`skip()`六个核心控制动作。
    - 时间轴UI直观展示剧本进度，并支持拖拽Seek（拖拽时自动暂停，防止状态混乱） — 回应约束：[CDO 10, CTO 4, CTO 12]。
- [事实·高] **沙盒控制面板**：在界面右侧绝对定位一个半透明调试面板。面板分为三部分：
    1. **状态机控制**：显示当前状态栈，并提供按钮手动触发七种状态。
    2. **剧本控制**：三大剧本的切换按钮与播放控制（开始/暂停/重置等）。
    3. **键位说明**：实时显示当前生效的键盘映射 — 回应约束：[CDO 4, CDO 12]。

**4. 核心体验流程（开机至剧本演示）**
1.  **开机动画**：打开页面，模拟头盔启动。四角边框依次以品牌主色调点亮，并显示“系统自检...OK”，随后进入默认的“巡航(S0)”状态 — 回应约束：[CDO 12]。
2.  **基础功能演示**：操作者通过键盘依次触发“导航”、“组队”等基础态，展示四角信息如何平稳切换。
3.  **中断与恢复演示**：在导航中，触发“来电”。RT区域显示来电通知，导航信息（RB）保持但视觉降级（如变淡）。挂断后，导航信息恢复原貌。此过程展示覆盖态的打断与恢复逻辑 — 回应约束：[CDO 8]。
4.  **ADAS预警核心演示**：启动“左后方超车”自动剧本。剧本将先建立巡航状态，随后模拟左后方车辆快速接近。LB区域瞬间触发红色高强度闪烁预警图标和边框，并伴有轻微震动模拟（CSS动画）。预警持续数秒后消失，状态自动恢复至巡航。此过程直观传达“被保护着”的核心价值 — 回应约束：[CMO 1, CMO 3]。
5.  **方案切换**：在演示过程中，随时按`D`键，在“全彩OLED”与“单绿光波导”两种视觉方案间切换，展示产品对不同硬件方案的适配能力。

## 决策点回应
- **决策1：采用三层七态状态机模型** → 回应约束：[CDO 3, CTO 2, CTO 10]（置信度：[判断·高]）。这是实现复杂打断、恢复与优先级仲裁的唯一可行架构，避免了平铺状态带来的逻辑混乱。
- **决策2：严格执行四角骨架与中央留空** → 回应约束：[CDO 1, CDO 7, CMO 2]（置信度：[事实·高] + [判断·高]）。这是HUD安全设计的铁律，任何侵入中央视野的设计都会在真实骑行中造成危险，必须在Demo中彻底贯彻以建立信任。
- **决策3：通过CSS滤镜实现A/B光学方案切换** → 回应约束：[CDO 9]（置信度：[事实·高]）。这是一种轻量、直观且完全在浏览器中实现的技术方案，完美满足了Demo的展示需求，无需复杂模拟。
- **决策4：将ADAS预警与方位区域强绑定** → 回应约束：[CDO 2, CMO 1]（置信度：[事实·高] + [判断·高]）。预警的方位感是安全功能生效的心理基础，将威胁方向与UI位置耦合，能最快建立骑手的条件反射，是“被保护着”体验的关键视觉表达。
- **决策5：内置完整的调试与控制面板** → 回应约束：[CDO 4, CDO 12, CTO 8]（置信度：[判断·高]）。对于投资人/供应商演示，可控、可复现、可解释比模拟完全真实环境更重要。控制面板让演示者能灵活应对现场提问，并深入展示状态机工作原理。

## 自评：最大风险
- [判断·高] **最大风险在于单文件内代码复杂度与可维护性的平衡** — 基于：本方案需在一个HTML文件内实现状态机、渲染引擎、剧本引擎、UI层和调试层，代码量极易超过500行。虽然通过模块化（JS内分区块）管理，但在后续迭代中，如果没有清晰的注释和结构，修改一个状态或优先级规则可能会产生难以预料的副作用，导致调试困难。这直接呼应了[CTO 5, CTO 6]的约束。缓解措施是：在代码中提供极其详细的常量定义、状态映射表和函数说明，并将《7态优先级与中断恢复路由表》以注释形式直接嵌入代码开头，作为“活的文档”。
