## 方案描述

本方案旨在生成一个面向供应商与投资人的 HUD 演示文件，核心目标是**稳定、清晰、可控地复现**智能骑行座舱的核心体验。方案严格遵循“极简、安全优先”的HUD设计范式，并满足所有CTO提出的技术交付约束。

**1. 整体架构与状态机设计 (回应CTO约束 2, 5, 9, 10)**
- [偏好·--] 采用“单页应用 + 内置状态机 + 定时器剧本引擎”的架构。所有逻辑、样式、内容均封装于单个HTML文件内，双击即可运行。
- [判断·高] 将7个页面态建模为三层结构，以清晰表达打断与抢占逻辑：
    1.  **基础态 (Base Layer)**：`IDLE`（待机）、`NAVIGATING`（导航中）、`GROUP_COMM`（组队通信中）、`MUSIC`（音乐播放中）。这些状态可共存或平缓切换。
    2.  **瞬时通知态 (Notification Layer)**：`INCOMING_CALL`（来电）。此层可短暂覆盖基础态，需用户确认或忽略。
    3.  **告警抢占态 (Alert Layer)**：`ADAS_ALERT`（ADAS预警）。此层拥有最高优先级，可打断任何其他状态，使用强视觉冲击力，并通常自动超时或需紧急确认。
- [判断·高] 状态机负责管理当前激活的状态、处理状态切换（包括打断、恢复）、并驱动UI渲染。这确保了优先级逻辑的清晰和代码的可维护性。

**2. HUD视觉与布局方案 (回应CDO约束 1, 3, 4, 5, 7; CMO约束 2, 5)**
- [判断·高] 视觉骨架严格采用研究推荐的“右下主信息区 + 顶部状态条 + 中央瞬时弹层”布局，确保信息一瞥可读。
    - **右下主信息区**：用于承载当前基础态的核心信息。例如，`NAVIGATING`态下，显示大号白色距离、蓝色导航箭头与路名；`GROUP_COMM`态下，显示青色队友状态图标与语音波形。
    - **顶部状态条**：固定显示时速（大号白色）、时间、连接状态等最关键的全局信息。
    - **中央瞬时弹层**：用于`ADAS_ALERT`和`INCOMING_CALL`。告警层占据视觉中心，使用高饱和红色、脉冲动画与大号图标/文字，确保冲击力。
- [偏好·--] 建立全彩功能色语义系统，并在CSS中定义为变量，便于维护与A/B方案调整：
    - `--color-speed: #ffffff;` (白)
    - `--color-nav: #4dabf7;` (蓝)
    - `--color-alert: #ff6b6b;` (红)
    - `--color-group: #38d9a9;` (青)
    - `--color-music: #9c36b5;` (紫)
- [判断·中] 所有字体采用无衬线体，字号层级分明（时速最大，导航信息次之，状态提示最小），并确保在模拟的“透明背景”上有足够对比度，以应对[CDO约束6]中提到的亮度挑战。

**3. 交互与演示控制 (回应CTO约束 3, 4, 8, 11, 12; CMO约束 3, 4)**
- [判断·高] 设计完整的键盘映射，支持演示与调试：
    - **数字键1-7**：直接切换到对应页面态（用于快速功能展示）。
    - **字母键**：模拟真实事件。例如，`B`触发BSD预警，`O`触发开门杀预警，`C`模拟来电，`N`开始导航。
    - **控制键**：`Space`开始/暂停自动剧本，`R`重置Demo，`Right Arrow`单步推进剧本。
- [判断·高] **自动剧本引擎**：预设一个约90秒的骑行叙事剧本，按时间线自动触发状态切换，模拟真实骑行场景（如：开始导航 → 听音乐 → 组队通话 → 被ADAS预警打断 → 处理来电）。引擎支持暂停、继续、重置，确保现场演示可控。
- [判断·中] 剧本设计将突出[CMO约束1]的ADAS差异化，安排至少两次不同类型的高冲击力预警（如盲区车辆快速接近、右侧开门杀），并确保在[CMO约束4]的3分钟时长内完成核心价值循环。

**4. 代码组织与实现策略 (回应CTO约束 1, 5, 6, 7, 13)**
- [判断·高] 在单个HTML文件内，通过`<script>`标签组织JS代码为清晰模块：
    1.  **CONFIG & STATE_DEF**：定义状态枚举、优先级映射、颜色变量、剧本事件序列。
    2.  **STATE_MACHINE**：核心状态机，包含状态切换、优先级判断、历史状态恢复逻辑。
    3.  **RENDERER**：根据当前状态机输出，操作DOM更新HUD各区域内容与样式。避免直接操作DOM的业务逻辑。
    4.  **SCRIPT_ENGINE**：管理定时器，按剧本触发事件，并与状态机交互。
    5.  **EVENT_HANDLER**：集中处理键盘事件，映射到状态切换或剧本控制命令。
    6.  **DEBUG_PANEL (可选)**：在开发模式显示当前状态、剧本进度，便于调试。
- [判断·高] 采用分段生成策略：先构建基础框架（状态机、渲染器），再填充具体状态UI和剧本事件，最后整合优化，以避免单次生成超过500行代码导致的质量下降。

## 决策点回应
- **决策1：采用三层状态机架构** → 回应约束：[CTO 9, CTO 10]（置信度：[判断·高]）。这是实现7个页面态间复杂优先级（尤其是告警打断）的唯一清晰、可维护方式。
- **决策2：视觉上严格遵循“右下主信息区+顶部状态条+中央弹层”骨架** → 回应约束：[CDO 1, CDO 3, CDO 4, CMO 2, CMO 5]（置信度：[判断·高]）。这是经过竞品验证、最适合摩托车余光扫视的安全设计范式。
- **决策3：实现包含暂停、单步控制的自动剧本引擎** → 回应约束：[CTO 4, CTO 8, CTO 12, CMO 4]（置信度：[判断·高]）。对于投资人演示，稳定、可控的叙事比完全自由的交互更重要。
- **决策4：在单HTML文件内采用模块化JS编码** → 回应约束：[CTO 5, CTO 13, CTO 6]（置信度：[判断·高]）。在“单文件”硬约束下保证后续可维护性的必要设计。
- **决策5：使用强对比色与脉冲动画表现ADAS预警** → 回应约束：[CDO 5, CMO 1, CMO 3]（置信度：[判断·中]）。这是传达“被保护着”这一核心差异化的最直接视觉手段。

## 自评：最大风险
- [判断·高] **最大风险在于光学模拟的失真**。本Demo在普通显示器上通过CSS模拟HUD的透明、悬浮、高对比度效果，这与真实光波导在强光下的实际亮度、色彩表现、虚像质量可能存在较大差距。供应商（尤其是光学方案商）可能会质疑其视觉保真度，认为Demo过于理想化。这需要我们在演示时明确说明当前为逻辑与交互演示，并辅以光学样机的实物展示作为补充。
- 次要风险是剧本节奏与键盘控制的现场操作复杂度，需要演示者充分练习，以避免在关键环节操作失误。
