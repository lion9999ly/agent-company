# HUD Demo 视觉与交互设计方案

## 方案描述

本方案旨在生成一个单HTML文件，用于向供应商和投资人展示智能骑行座舱HUD的核心体验。方案严格遵循“一瞥可读”原则，聚焦于通过视觉和交互设计传达“被保护着”的产品核心差异化，并消灭“停车掏手机”的核心痛点。

**1. 视觉架构与布局**
- [偏好·--] 采用“右下主信息区 + 顶部状态条 + 中央瞬时弹层”的固定布局骨架。右下区为信息锚点，用于显示导航、速度等核心信息；顶部状态条用于显示连接状态、电量、时间等轻量信息；中央区域仅用于高优先级、高冲击力的瞬时安全预警。此布局回应了约束[CDO-2]和[CTO-10]，确保信息结构稳定，避免认知过载。
- [判断·高] 严格遵循“低层级、低元素、大字号”原则。每个信息区域同一时间仅显示1-2个核心数据元素（如：导航箭头+距离，或速度+档位），主信息字号模拟在2-3米外观看7英寸屏幕的视觉大小。这直接回应了约束[CDO-1]和[CDO-4]。
- [偏好·--] 建立全彩功能色语义系统：
    - **速度/基础信息：白色** - 高对比度，常驻信息。
    - **导航指引：蓝色** - 冷静、指引性。
    - **ADAS安全预警：红色** - 高饱和度，配合动态效果（如呼吸、闪烁），形成强烈视觉冲击。这是“被保护着”体验的核心视觉表达，回应约束[CMO-1]、[CMO-4]和[CTO-8]。
    - **Mesh组队通信：青色** - 科技感与连接感。
    - **音乐/语音交互：紫色** - 区别于安全色，代表娱乐与交互。
    此系统回应约束[CDO-6]，通过颜色加速信息分类识别。

**2. 状态机与交互设计**
- [判断·高] 设计一个基于优先级仲裁的显式状态机，包含7个页面态。状态机模型如下：
    1.  **基础巡航态** (低优先级)：显示速度、时间、电量。默认状态。
    2.  **导航任务态** (中优先级)：右下区显示导航箭头、距离、下一道路名。
    3.  **组队通信态** (中优先级)：右下区显示队友状态、语音通话指示。
    4.  **音乐播放态** (中优先级)：右下区显示歌曲名、播放控制简示。
    5.  **行车记录仪态** (低优先级)：顶部状态条显示录制图标。
    6.  **ADAS预警态** (高优先级 - 瞬时中断)：中央弹层显示红色预警图标与简短文本（如“左侧盲区！”），并覆盖其他所有状态。
    7.  **语音交互态** (中优先级 - 瞬时)：中央弹层显示语音波形与识别结果反馈。
    此设计回应了约束[CTO-2]、[CTO-8]、[CTO-9]和[CTO-15]，确保安全预警能强制中断其他任何状态，体现产品核心价值。
- [判断·高] 实现全覆盖的键盘事件映射，模拟真实交互：
    - `N`键：切换导航开启/关闭。
    - `C`键：切换组队通信开启/关闭。
    - `M`键：切换音乐播放/暂停。
    - `R`键：切换行车记录仪录制。
    - `1/2/3`键：分别触发左侧盲区预警、开门杀预警、前车碰撞预警（ADAS）。
    - `V`键：模拟语音指令触发（如“导航去外滩”）。
    - `Space`键：暂停/继续自动剧本。
    这满足了约束[CTO-3]和[CTO-13]，使演示完全可控。

**3. 自动剧本叙事设计**
- [判断·中] 设计一个约90秒的自动演示剧本，节奏张弛有度，旨在3分钟内讲清故事：
    1.  **(0-10s) 开机与品牌展示**：简洁动画，建立“可靠科技”感，回应约束[CMO-5]。
    2.  **(11-30s) 基础巡航**：展示速度、电量等信息，体现常态。
    3.  **(31-50s) 语音发起导航**：模拟“导航去东方明珠”，HUD切换至导航态，展示蓝色指引。
    4.  **(51-70s) 组队与音乐**：展示社交与娱乐功能，体现场景覆盖。
    5.  **(71-85s) 安全预警高潮**：连续触发1-2个ADAS预警（红色中央弹层），视觉冲击最大化，突出“被保护着”。
    6.  **(86-90s) 恢复巡航**：预警结束，状态平滑恢复，体现系统稳定性。
    剧本设计需与状态机深度集成，支持随时被键盘操作打断，并在暂停后可选择继续或停止，满足约束[CTO-4]、[CTO-13]和[CMO-2]。

**4. 技术实现与视觉模拟**
- [判断·高] 使用纯原生HTML/CSS/JS实现单文件零依赖交付，代码结构分为清晰区块（样式、状态定义、状态机逻辑、事件监听、剧本配置），确保可维护性，回应约束[CTO-1]、[CTO-5]、[CTO-11]和[CTO-16]。
- [判断·低] 通过CSS滤镜和半透明层模拟两种光学方案（A: OLED， B: 光波导）的视觉差异。例如，通过切换类名，模拟“方案A”色彩更饱和、对比度高，“方案B”亮度更均匀、视场角更广的视觉效果。这旨在辅助技术讨论，回应约束[CDO-7]。具体实现为在页面角落提供“A/B方案切换”按钮，点击后全局应用不同的CSS滤镜。

## 决策点回应
- **决策1：采用固定“右下+顶部+中央”布局，并严格限制信息密度** → 回应约束：[CDO-1], [CDO-2], [CDO-4], [CTO-10]（置信度：[判断·高]）
- **决策2：建立“红-蓝-青-紫-白”功能色语义系统，并将红色预警置于中央弹层** → 回应约束：[CDO-6], [CTO-8], [CMO-1], [CMO-4]（置信度：[偏好·--] + [判断·高]）
- **决策3：设计7态优先级状态机，确保ADAS预警为最高优先级中断源** → 回应约束：[CTO-2], [CTO-8], [CTO-9], [CTO-12], [CTO-15]（置信度：[判断·高]）
- **决策4：实现全覆盖键盘交互与可中断的自动剧本** → 回应约束：[CTO-3], [CTO-4], [CTO-13], [CMO-2]（置信度：[判断·高]）
- **决策5：使用原生技术栈并结构化代码，模拟A/B光学效果** → 回应约束：[CTO-1], [CTO-5], [CTO-11], [CTO-16], [CDO-7]（置信度：[判断·高] + [判断·低]）

## 自评：最大风险
**最大风险在于视觉模拟的真实性与技术可行性的认知偏差。** 方案中通过CSS模拟A/B光学方案差异（[CDO-7]），这完全基于设计侧的推测。实际光波导与OLED在亮度、均匀性、眼动范围（Eyebox）、色彩表现上的差异可能远超CSS滤镜所能模拟的范畴。若供应商或懂行的投资人深究，此模拟可能显得过于简陋甚至产生误导。这源于我（CDO）对“技术可行性、工程复杂度”的盲区。**缓解措施**：在Demo中明确标注此为“视觉示意差异，非最终光学效果”，并将讨论引导至功能体验和用户价值（[CMO-7]），而非技术参数对比。同时，此风险点也凸显了与CTO及光学供应商早期紧密协同的必要性。
