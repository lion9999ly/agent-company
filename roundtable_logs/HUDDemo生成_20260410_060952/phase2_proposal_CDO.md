# HUD Demo HTML 方案

## 方案描述

本方案将生成一个单HTML文件，模拟智能骑行头盔HUD的核心体验。方案严格遵循“极简固定骨架”原则，视觉设计聚焦于“一瞥可读”与“被保护着”的情感传递。

**1. 视觉与信息架构设计**
- [偏好·--] **固定视觉骨架**：采用“右下主信息区 + 顶部状态条 + 中央瞬时弹层”的三层结构。此布局稳定，符合骑手余光扫视习惯，避免认知过载。主信息区默认显示速度、导航箭头和距离；顶部状态条显示连接状态、电量、时间等轻量信息；中央弹层仅用于最高优先级的ADAS预警。
    - 回应约束：[CDO-1], [CDO-2], [CTO-5], [CMO-2]
- [判断·中] **全彩功能色语义系统**：
    - **速度/基础信息**：白色 (#FFFFFF) 或浅灰色 (#CCCCCC)，高对比度确保基础可读性。
    - **导航**：蓝色 (#4A90E2)，用于转向箭头、路名、剩余距离。
    - **ADAS预警**：高饱和度红色 (#FF3B30)，配合脉冲动画和中央弹层，实现最高视觉冲击力。
    - **Mesh组队**：青色 (#5AC8FA)，用于队友状态、通话指示。
    - **音乐/通话**：紫色 (#AF52DE)，用于歌名、来电显示。
    - 回应约束：[CDO-4]
- [判断·高] **ADAS预警作为视觉高潮**：预警（如BSD盲区碰撞、FCW前车碰撞）将触发以下组合效果：
    1. 中央弹层瞬间出现，占据视觉中心。
    2. 高饱和度红色背景，配合白色图标（如感叹号、车辆侧影）和简明文字（如“左侧危险！”）。
    3. 背景红色执行“脉冲”动画（明暗交替），并伴随整个界面边缘的红色光晕闪烁。
    4. 预警结束后，界面平滑恢复至之前状态（如导航界面）。
    - 回应约束：[CDO-3], [CDO-5], [CMO-1], [CMO-3], [CTO-11]

**2. 交互与状态管理**
- [判断·高] **七态优先级状态机**：定义七个核心页面状态及其抢占优先级（数字越小优先级越高）：
    1. **紧急告警** (ADAS碰撞预警)
    2. **安全提示** (LDW车道偏离、交通标志识别)
    3. **导航引导** (转弯、变道)
    4. **主动通话** (Mesh组队来电)
    5. **音乐控制** (播放/暂停/切歌触发)
    6. **巡航信息** (默认态：速度、基础导航)
    7. **系统状态** (开机、连接中)
    - 状态机统一管理渲染，确保高优先级状态能打断低优先级状态，并在结束后能正确恢复。
    - 回应约束：[CTO-5], [CTO-6], [CTO-11]
- [判断·中] **键盘事件抽象层**：键盘事件（如数字键1-7触发不同状态，空格键触发自动剧本）将被映射为“状态切换意图”，由状态机统一处理，而非直接操作DOM。
    - 回应约束：[CTO-7]
- [判断·中] **自动剧本叙事节奏**：剧本将模拟一次完整的骑行旅程，时长约90秒，节奏如下：
    1. **(0-10s) 开机与巡航**：展示品牌开机动画（简洁可靠风格）后，进入默认巡航态，显示速度、时间。
    2. **(11-30s) 导航介入**：触发导航态，右下角出现蓝色转向箭头和路名，模拟语音指令“导航去西山公园”。
    3. **(31-50s) 组队与音乐**：叠加组队态（顶部出现青色队友图标），然后触发音乐态（底部出现紫色音乐控件）。
    4. **(51-70s) 安全提示**：触发LDW提示（顶部状态条黄色闪烁，显示“车道偏离”）。
    5. **(71-90s) 高潮与收尾**：触发**紧急BSD预警**（中央红色脉冲弹层，强烈视觉冲击），预警结束后，界面恢复至导航态，最后淡出结束。
    - 回应约束：[CDO-6], [CMO-3], [CMO-5], [CTO-8]

**3. 技术实现与交付**
- [判断·高] **单HTML零依赖**：所有代码（HTML结构、CSS样式、JavaScript逻辑）将整合进一个HTML文件。CSS使用`@keyframes`实现脉冲、闪烁等动画；JS使用原生`setInterval`和`requestAnimationFrame`管理剧本时间轴和动画。
- [判断·中] **模块化代码结构**：在单文件内，通过注释清晰分隔以下模块：
    - `<!-- CONSTANTS & STATE -->`：定义状态枚举、优先级、颜色常量。
    - `<!-- DOM & RENDERING -->`：定义固定的DOM槽位和渲染函数。
    - `<!-- STATE MACHINE -->`：状态机核心逻辑。
    - `<!-- INPUT HANDLER -->`：键盘事件监听与意图映射。
    - `<!-- AUTO-SCRIPT -->`：自动剧本的时间轴调度器。
    - `<!-- DEBUG PANEL -->`：一个可隐藏的调试面板，显示当前状态和快捷键说明。
    - 回应约束：[CTO-1], [CTO-3], [CTO-9], [CTO-12]
- [判断·中] **规避高风险技术**：不使用Canvas/WebGL/音视频，所有视觉效果通过CSS实现。虚像的“悬浮感”通过轻微模糊和阴影模拟。为表达A/B光学方案差异，可在调试面板用文字备注说明（如“方案A：FOV 12°，虚像距离10m”）。
    - 回应约束：[CTO-10], [CDO-7]

## 自评：最大风险
- [判断·中] **最大风险：视觉冲击力与真实光学效果的差距**。方案完全依赖CSS模拟HUD的亮度、透明度和“悬浮于道路”的景深感。在强光环境模拟、不同光学方案（如光波导的色散、FOV限制）的视觉差异表达上会非常薄弱，可能让技术背景的供应商或投资人质疑最终量产效果。这是由“零依赖、单HTML、规避Canvas”的技术约束与“需要表达光学差异”的设计需求之间的根本矛盾导致的。
    - 缓解措施：在Demo的调试面板中，以文字形式清晰标注当前模拟的是“理想光学条件”，并预留备注说明A/B方案的关键参数差异，引导观众关注信息架构和交互逻辑本身。
