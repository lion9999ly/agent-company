# Web-Rooter 工具使用策略指南

**版本**: v2.1
**更新日期**: 2026-03-08

---

## 一、概述

Web-Rooter 提供了一套完整的网页访问和信息获取工具。本指南旨在帮助 AI 理解**何时使用哪个工具**，以及**如何组合使用多个工具**来完成复杂任务。

### 核心原则

1. **没有万能工具** - 每个工具都有其适用场景
2. **避免局部最优** - 不要习惯性使用单一工具
3. **组合优于单一** - 复杂任务需要多工具协作
4. **效率优先** - 选择最快达成目标的路径

## 二、信息来源管理与标注规范 ⭐重要

### 2.1 第一手资料优先原则

**核心要求**: 所有回答必须严格基于 web-rooter 搜索到的第一手资料，不得编造或臆测。

```
优先级排序:
1. ✅ web-rooter 直接获取的网页内容 (第一手资料)
2. ✅ web-rooter 搜索结果的摘要 (第二手资料)
3. ⚠️ AI 训练数据中的知识 (需明确标注)
4. ❌ 编造或未经验证的信息 (禁止使用)
```

### 2.2 信息来源标注规范

**在回答中必须明确标注信息来源**:

| 来源类型 | 标注方式 | 示例 |
|----------|----------|------|
| web-rooter 搜索结果 | `[web-rooter]` 或 `[搜索]` | "根据 web-rooter 搜索结果显示..." |
| 具体网页内容 | `[来源：URL]` | "[来源：https://example.com]" |
| AI 知识库 | `[AI 知识库]` 或 `[训练数据]` | "根据我的训练数据（截至 2025 年）..." |
| 未找到信息 | `[未找到]` | "web-rooter 未找到相关信息" |

### 2.3 回答模板

**推荐使用的回答结构**:

```markdown
## 搜索结果总结

根据 web-rooter 搜索，找到以下信息：

### [主题]
[内容] [来源：URL 或 工具名]

---

## 补充说明（可选）

以下内容来自我的训练数据，仅供参考：
[内容] [标注：AI 知识库]

---

## 信息来源说明
- 主要内容来源：web-rooter 搜索（X 个来源）
- 补充信息来源：AI 训练知识库
- 未找到确认的信息：[列出]
```

### 2.4 禁止行为

❌ **禁止将 AI 知识库内容伪装成搜索结果**
```
错误示例："根据搜索结果显示..."（实际是 AI 知识库）
正确示例："根据我的训练数据...（web-rooter 未找到相关信息）"
```

❌ **禁止混合来源而不标注**
```
错误示例：混合搜索结果和 AI 知识，不做区分
正确示例：明确标注"以下为搜索结果"和"以下为 AI 知识库补充"
```

❌ **禁止编造 URL 或引用**
```
错误示例：编造不存在的网页链接
正确示例：只引用 web-rooter 实际找到的 URL
```

### 2.5 信息冲突处理

当 web-rooter 搜索结果与 AI 知识库不一致时：

1. **优先信任 web-rooter 获取的第一手资料**
2. 在回答中说明差异：
   ```
   "web-rooter 搜索结果显示 X，但我训练数据中记录的是 Y。
   建议以最新搜索结果为准（我的训练数据截至 2025 年）。"
   ```
3. 如遇重大差异，建议用户核实

## 三、各工具详细使用场景

### 3.1 信息搜索类工具

#### `web` / `web_search_internet` - 互联网搜索

**用途**: 在多个搜索引擎中搜索信息

**适用场景**:
- ✅ 需要了解某个主题的概况
- ✅ 寻找最新新闻或事件
- ✅ 验证某个事实或数据
- ✅ 寻找相关网站和资源

**不适用场景**:
- ❌ 需要特定网站的深层内容
- ❌ 需要社交媒体上的实时讨论
- ❌ 需要技术代码或 API 文档

**示例**:
```
用户：帮我了解一下最近 AI 领域的最新进展

AI 思考：这是需要了解概况的请求，使用 web 搜索获取多个来源的信息
操作：web_search_internet("AI 人工智能 最新进展 2025")
```

**返回内容**:
- 搜索结果列表（标题、URL、摘要）
- 来源搜索引擎
- 可选：爬取的详细内容

---

#### `deep` / `web_deep_search` - 深度搜索

**用途**: 并行使用多个搜索引擎，支持中英文双语搜索

**适用场景**:
- ✅ 需要全面的信息覆盖
- ✅ 主题较复杂，需要多角度信息
- ✅ 希望同时获取中英文资源
- ✅ 需要爬取搜索结果页面获取详细内容

**与 `web` 的区别**:
| 特性 | web | deep |
|------|-----|------|
| 引擎数量 | 1-2 个 | 4 个并行 |
| 语言 | 单一 | 中英文双语 |
| 速度 | 快 | 稍慢（但并行） |
| 结果数量 | 较少 | 更多 |
| 爬取 | 可选 | 强烈推荐 |

**示例**:
```
用户：我想全面了解 Transformer 架构的原理和应用

AI 思考：这是需要深入全面信息的请求，使用 deep 搜索获取多引擎、多语言结果
操作：web_deep_search("Transformer architecture principle application",
                       num_results=20, use_english=True, crawl_top=5)
```

---

#### `social` / `web_search_social` - 社交媒体搜索

**用途**: 在社交媒体平台搜索用户讨论和评价

**适用场景**:
- ✅ 了解用户真实评价和反馈
- ✅ 监控舆情和口碑
- ✅ 寻找使用体验和心得分享
- ✅ 了解热门话题和趋势

**支持平台**:
- Bilibili（B 站）- 视频评论
- Zhihu（知乎）- 专业讨论
- Weibo（微博）- 实时舆情
- Reddit - 国际讨论
- Twitter - 实时动态

**示例**:
```
用户：iPhone 17 用户反馈怎么样？

AI 思考：需要了解真实用户评价，使用 social 搜索社交媒体
操作：web_search_social("iPhone 17 用户评价", platforms=["zhihu", "bilibili"])
```

---

#### `tech` / `web_search_tech` - 技术社区搜索

**用途**: 在技术社区搜索代码项目和技术讨论

**适用场景**:
- ✅ 寻找开源项目和代码
- ✅ 解决技术问题和 bug
- ✅ 学习最佳实践
- ✅ 了解技术趋势和讨论

**支持来源**:
- GitHub - 代码项目
- Stack Overflow - 技术问答
- Medium - 技术文章
- Hacker News - 技术热点

**示例**:
```
用户：有没有好的 Python Web 框架推荐？

AI 思考：需要技术项目推荐，使用 tech 搜索技术社区
操作：web_search_tech("python web framework comparison", sources=["github", "stackoverflow"])
```

---

#### `academic` / `web_search_academic` - 学术搜索

**用途**: 搜索学术论文和代码项目

**适用场景**:
- ✅ 研究学术主题
- ✅ 查找相关论文
- ✅ 寻找研究代码实现
- ✅ 了解学术前沿

**支持来源**:
- arXiv - 预印本论文
- Google Scholar - 学术搜索
- Semantic Scholar - 语义学术
- GitHub/Gitee - 代码项目

**示例**:
```
用户：我想找一些关于大语言模型的论文

AI 思考：需要学术论文，使用 academic 搜索
操作：web_search_academic("large language model survey", include_code=True)
```

---

#### `research` / `web_research` - 深度研究

**用途**: 多步骤深度研究，自动搜索 + 爬取 + 分析

**适用场景**:
- ✅ 需要 comprehensive 研究报告
- ✅ 主题复杂，需要多轮搜索
- ✅ 需要综合分析和总结
- ✅ 时间充裕，追求深度

**与 `deep` 的区别**:
| 特性 | deep | research |
|------|------|----------|
| 搜索轮次 | 单轮 | 多轮迭代 |
| 爬取深度 | 浅层 | 深层 |
| 分析程度 | 基础 | 深度综合 |
| 耗时 | 较短 | 较长 |
| 输出 | 结果列表 | 综合报告 |

**示例**:
```
用户：帮我研究一下量子计算的发展现状和未来趋势

AI 思考：这是复杂的研究性任务，需要多轮搜索和深度分析，使用 research
操作：web_research("量子计算 发展现状 未来趋势", max_pages=15)
```

### 3.2 网页访问类工具

#### `visit` / `web_fetch` - 访问网页

**用途**: 获取单个网页内容

**适用场景**:
- ✅ 已有明确 URL，需要获取内容
- ✅ 搜索结果中的链接需要查看详情
- ✅ 需要提取页面结构化信息
- ✅ 需要检查页面是否存在

**与搜索工具的区别**:
- 搜索工具：不知道有什么 → 发现内容
- visit：知道 URL → 获取内容

**示例**:
```
用户：帮我看看这个页面的内容：https://example.com/article

AI 思考：已有明确 URL，直接访问
操作：web_fetch("https://example.com/article")
```

---

#### `fetch_js` / `web_fetch_js` - 浏览器访问（JavaScript 支持）

**用途**: 使用真实浏览器获取渲染后的页面

**适用场景**:
- ✅ 页面需要 JavaScript 渲染
- ✅ 普通 fetch 获取内容为空
- ✅ 需要等待特定元素加载
- ✅ 需要模拟用户交互

**与普通 `visit` 的区别**:
| 特性 | visit/fetch | fetch_js |
|------|-------------|----------|
| 速度 | 快 | 慢 |
| 资源消耗 | 低 | 高 |
| JS 支持 | 无 | 完整 |
| 适用场景 | 静态页面 | 动态页面 |

**示例**:
```
用户：这个页面我访问了但内容是空的

AI 思考：可能是 JS 渲染的页面，需要使用 browser 模式
操作：web_fetch_js("https://example.com", wait_for=".content")
```

---

#### `crawl` / `web_crawl` - 爬取网站

**用途**: 自动爬取整个网站的多个页面

**适用场景**:
- ✅ 需要获取网站多个页面的内容
- ✅ 需要追踪链接深入网站
- ✅ 需要批量获取相似页面
- ✅ 需要网站地图或结构

**参数说明**:
- `max_pages`: 最大页面数（默认 10）
- `max_depth`: 最大深度（默认 3）

**示例**:
```
用户：帮我爬取这个博客的所有文章

AI 思考：需要获取网站多页内容，使用 crawl
操作：web_crawl("https://blog.example.com", max_pages=50, max_depth=3)
```

---

#### `site` / `web_search_site` - 站内搜索

**用途**: 使用网站内部的搜索功能

**适用场景**:
- ✅ 需要在特定网站内搜索
- ✅ 网站有内部搜索框
- ✅ 需要了解网站对某个主题的内容

**与 `web` 的区别**:
- `web site:example.com query` - 使用 Google 搜索该网站
- `site example.com query` - 使用网站自己的搜索功能

**示例**:
```
用户：在 GitHub 上搜索机器学习的 Python 项目

AI 思考：需要在特定网站内搜索，使用 site
操作：web_search_site("https://github.com", "machine learning python")
```

### 3.3 信息处理类工具

#### `extract` / `web_extract` - 信息提取

**用途**: 从网页提取特定信息

**适用场景**:
- ✅ 需要从页面提取结构化数据
- ✅ 需要特定字段（价格、评价、规格等）
- ✅ 需要总结页面内容

**示例**:
```
用户：从这个页面提取产品名称、价格和库存状态

AI 思考：需要提取结构化信息，使用 extract
操作：web_extract("https://example.com/product", "产品名称、价格、库存状态")
```

---

#### `parse_html` - 解析 HTML

**用途**: 解析 HTML 内容，提取结构化数据

**适用场景**:
- ✅ 已有 HTML 内容需要解析
- ✅ 需要获取链接、标题、文本
- ✅ 需要生成 CSS/XPath 选择器

**示例**:
```
用户：帮我解析这段 HTML，提取所有链接

AI 思考：已有 HTML 需要解析，使用 parse_html
操作：parse_html(html_content, url="https://example.com")
```

---

#### `get_links` - 获取页面链接

**用途**: 获取页面所有链接

**适用场景**:
- ✅ 需要了解网站结构
- ✅ 需要追踪相关页面
- ✅ 需要批量处理链接

**示例**:
```
用户：这个页面有哪些内部链接？

AI 思考：需要获取页面链接，使用 get_links
操作：get_links("https://example.com", internal_only=True)
```

### 4.2 典型任务的工作流

## 五、避免局部最优的策略

### 5.1 常见局部最优陷阱

#### 陷阱 1: 只用 `web` 搜索

```
❌ 错误模式：
用户：帮我找一下 GitHub 上的机器学习项目
AI: web_search("GitHub machine learning")  # 只能用 web

✅ 正确模式：
用户：帮我找一下 GitHub 上的机器学习项目
AI: web_search_tech("machine learning", sources=["github"])  # tech 更专业
   或者：web_search_site("https://github.com", "machine learning")  # 站内搜索
```

---

#### 陷阱 2: 搜索后不爬取

```
❌ 错误模式：
用户：帮我了解这个产品的详细信息
AI: web_search("产品名")  # 只返回搜索结果列表

✅ 正确模式：
用户：帮我了解这个产品的详细信息
AI: web_search_combined("产品名", crawl_top=5)  # 搜索 + 爬取
   或者：result = web_search(...)
        for url in top_urls: web_fetch(url)
```

## 六、最佳实践

### 6.1 搜索策略

1. **先宽后深**: 先用 `web` 或 `deep` 了解概况，再深入特定来源
2. **多源验证**: 重要信息用多个工具交叉验证
3. **中英文结合**: 技术内容使用 `deep --en` 获取英文资源
4. **时效性优先**: 最新信息用 `web` + `social`

### 6.2 爬取策略

1. **先搜索后爬取**: 不要盲目爬取，先搜索找到目标
2. **限制深度**: 设置合理的 `max_pages` 和 `max_depth`
3. **动态页面检测**: 内容为空时尝试 `fetch_js`
4. **错误处理**: 爬取失败时重试或更换方法

### 6.3 组合策略

```
复杂任务 = 搜索 (发现) + 爬取 (获取) + 提取 (结构化) + 分析 (综合)

示例工作流:
1. deep_search()  # 发现相关信息
2. fetch(top_urls)  # 获取详细内容
3. extract()  # 提取关键信息
4. 综合分析返回用户
```

## 八、常见问题解答

### Q1: 什么时候用 `web` 什么时候用 `deep`？

**A**:
- `web`: 快速搜索，了解概况
- `deep`: 全面搜索，多引擎并行，建议默认使用

### Q2: 搜索结果为空怎么办？

**A**:
1. 尝试更换关键词
2. 使用英文搜索（`deep --en`）
3. 更换工具（如 `web` → `tech` 或 `social`）
4. 检查网络连接

### Q3: 页面内容为空怎么办？

**A**:
1. 尝试 `fetch_js` 代替 `fetch`
2. 添加 `wait_for` 参数等待元素
3. 检查 URL 是否正确

### Q4: 如何选择合适的爬取深度？

**A**:
- 单页面信息：`crawl_top=0`（不爬取）
- 获取摘要：`crawl_top=3-5`
- 详细内容：`crawl_top=10+`
- 完整研究：使用 `research` 自动决定

### Q5: 多个工具返回不同结果怎么办？

**A**:
1. 交叉验证，优先选择多个来源一致的信息
2. 注明来源，让用户判断
3. 解释差异原因（如时效性、来源不同）

## 九、总结

本指南的核心思想：

1. **理解工具特性** - 每个工具都有其优势和局限
2. **根据任务选择** - 不同任务需要不同工具组合
3. **避免思维定式** - 不要习惯性地只用一个工具
4. **灵活组合使用** - 复杂任务需要多工具协作

记住：**最好的工具是能帮助最高效完成任务的工具**，而不是你最熟悉的工具。
