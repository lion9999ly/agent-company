# HUD Demo v2 Scorer 校准文档

> 来源：Leo 亲自评审 + Claude Chat 审查，2026-04-13
> 用途：作为 EvoSkill scorer 的 few-shot 校准样本，对齐 Leo 的品味标准

---

## 一、Leo 直接评审（截图标注，权重最高）

### R1: 单绿模式 emoji 彩色 — 评分 0/10
**截图**：单绿波导模式下，导航箭头 emoji ⬅️ 显示为蓝色
**Leo 判断**：NOK
**根因**：emoji 是 Unicode 字符，自带颜色，CSS 无法强制变色
**正确做法**：单绿模式下禁止一切 emoji，全部用 SVG 线条或纯文本符号
**scorer 规则**：HTML 源码在 .optics-green 相关渲染路径中不得包含任何 emoji Unicode 字符（U+1F000-U+1FFFF 范围）

### R2: ADAS 预警内容溢出 — 评分 2/10
**截图**：FCW 预警的距离+速度+TTC 内容超出 zone-lb 的可视边界
**Leo 判断**：NOK
**根因**：zone-lb 高度固定 90px，预警内容（图标+类型+方向+距离+速度+TTC）至少需要 150px
**正确做法**：预警信息紧凑排列（一行显示"FCW front 15m 45km/h"，TTC 单独大字），或 zone 动态扩展
**scorer 规则**：任何 zone 的内容不得出现 overflow，所有信息必须在可视区域内完整可见

### R3: 连续预警堆叠上移 — 评分 1/10
**截图**：连续按 W/A/D，预警信息不断追加导致内容向上移出视野
**Leo 判断**：NOK
**根因**：activeWarnings 数组无上限，push 不清理旧预警
**正确做法**：新预警替换旧预警（保留 TTC 最短的 1-2 个），不追加堆叠
**scorer 规则**：activeWarnings.length 永远不超过 2

### R4: touring 剧本空洞 — 评分 2/10
**截图**：touring 剧本运行中，没有语音交互字幕，没有交通标志识别
**Leo 判断**：NOK
**根因**：touring 剧本事件列表过于简陋（60秒只有7个事件），缺少 showVoiceSubtitle 和 setTrafficSign 调用
**正确做法**：touring 是最重要的场景（目标用户是摩旅），必须最丰富——至少 15 个事件，含语音标记精彩瞬间、多次交通标志切换、DVR 生活记录、导航切换
**scorer 规则**：touring 剧本事件数 ≥ 15，包含 ≥ 2 次 showVoiceSubtitle 和 ≥ 2 次 setTrafficSign

## 二、Claude Chat 审查（对照 SKILL.md 逐条检查）

### C1: 路面背景图片上传缺失 — 评分 0/10
SKILL 要求"用户可导入自定义图片替换"，sandbox 里没有上传按钮
**正确做法**：sandbox 中提供 input type=file，上传后替换 #road-bg 为用户图片
**优先级**：P0（Leo 明确提出的需求，superpowers brainstorming 中确认过）

### C2: 开机视野确认缺失 — 评分 0/10
Leo 要求"开机时有个视野位置确认"
**正确做法**：开机流程中显示四角测试图案（四个角各一个标记），用户确认四个角都可见后进入正常模式。预留接口，具体交互待定
**优先级**：P1

### C3: LDW 预警完全缺失 — 评分 0/10
emergency 剧本中没有 LDW（车道偏离）场景
**正确做法**：emergency 剧本必须包含至少一次 LDW 左侧压线和一次 LDW 右侧压线
**优先级**：P0

### C4: 多重预警并发未实现 — 评分 0/10
emergency 剧本所有预警串行——一个清除后才触发下一个
**正确做法**：至少一次 FCW + BSD 同时存在（TTC 短的大显示，另一个弱化）
**优先级**：P0

### C5: FreeForm vs 全彩波导无视觉差异 — 评分 1/10
两种模式看起来完全一样，只有左上角标签不同
**正确做法**：全彩波导整体亮度降低 30-40%（模拟到眼 <1000nits vs FreeForm ~1500nits），透过率更高意味着背景更清晰（背景 opacity 默认更高）
**优先级**：P0

### C6: FOV 滑块无实际效果 — 评分 1/10
滑块改变数值显示但不改变显示区域大小
**正确做法**：FOV 越大→竖条可显示区域越宽，内容可以更丰富；FOV 越小→竖条越窄，内容必须更精简
**优先级**：P1

### C7: 右侧竖条几乎全空 — 评分 2/10
四张截图中右侧只有电池+信号两个 emoji 图标，大面积空白
**正确做法**：RT zone 显示交通标志（持续）+ 系统状态；RB zone 根据模式显示 Mesh 组队/预警详情/DVR 状态
**优先级**：P0

### C8: Mesh 组队只有文字 — 评分 2/10
"队友: 前方 200m" 一行文字
**正确做法**：SVG 简图显示 2-3 个队友的相对位置（前方/后方+距离标注）
**优先级**：P2

### C9: 导航地图是假的 — 评分 2/10
"简易地图"和"全信息地图"都是空框+文字说明
**正确做法**：简易地图用 SVG 画路口简图（十字路口+转向箭头+当前位置标记），全信息地图用 SVG 画从起点到终点的折线路线
**优先级**：P1

### C10: 来电显示太弱 — 评分 3/10
来电只在 zone-lb 一小块区域，不像是一个高优先级事件
**正确做法**：来电抢占时，左侧竖条主要区域显示来电信息（名称+接听/拒绝），导航缩小到角落
**优先级**：P1

### C11: 单绿模式用 rgba alpha 而非 opacity — 评分 3/10
实现用 rgba(0,255,0,0.5) 等变体
**正确做法**：所有元素 color: #00FF00，通过 opacity 属性控制亮度
**scorer 规则**：.optics-green 相关 CSS 中不得出现 rgba(0,255,0, 非1的值)，只能用 opacity

### C12: S2/S3 速度简化不完整 — 评分 3/10
速度到 80km/h（S2）时音乐和组队信息仍完整显示
**正确做法**：S2 时音乐/组队只保留小图标，S3 时只保留速度+ADAS+箭头导航
**优先级**：P1

### C13: 预警视觉效果粗糙 — 评分 3/10
只有红色边框闪烁，方向用文字 "front/left/right"
**正确做法**：方向用 SVG 箭头图形，预警框用脉冲 glow 效果（box-shadow 动画），紧急程度通过闪烁频率直观区分
**优先级**：P2

### C14: 所有图标用 emoji — 评分 2/10
电池🔋、信号📶、温度🌡、警告⚠、标志🚸 全是 emoji
**正确做法**：用 SVG 线条图标，风格统一为 HUD 科技感（细线条、无填充、统一描边宽度）
**优先级**：P1

### C15: superpowers brainstorming 的视觉方案未被使用 — 评分 1/10
CC 用 brainstorming 对齐了布局方案（有截图确认），但写代码时完全做了自己的版本
**根因**：brainstorming 产出的 spec 在 docs/superpowers/specs/ 目录，CC 重写时没有主动加载
**正确做法**：spec 文件应该被 SKILL.md 引用，或者直接内联到 skill 的 references/ 目录

### C16: 物理布局模型不够准确 — 评分 4/10
截图显示两条完整矩形竖条，中间全部填满内容。真实效果应该是：竖条的上端一块信息（LT）、下端一块信息（LB），中间部分大多时候是空的或只有导航地图。不是从上到下铺满
**正确做法**：zone 之间有明确间隔，非活跃 zone 透明，不填充背景色
**优先级**：P1

## 三、校准基准

### Leo 的品味标准总结
1. **物理真实性优先**：Demo 必须反映真实硬件约束（FOV、亮度、单绿限制），不能只是"看起来像 HUD"
2. **信息不能溢出**：任何内容超出可视区域 = 不合格，没有商量余地
3. **颜色约束是硬约束**：单绿说只有 #00FF00 就只有 #00FF00，不能有任何变通
4. **剧本要有故事感**：不是数字在变，是骑行场景的还原，要能让看的人想象自己在骑车
5. **每个功能都要真正实现**：空框+文字标签不叫实现
6. **细节决定可信度**：emoji 图标、粗糙闪烁、空白区域——这些让 Demo 看起来像 hackathon 作品而非产品原型

### 评分标准
- 10/10：完全符合 SKILL.md 和对齐需求，视觉效果可直接用于投资人/供应商演示
- 7-9/10：功能完整但视觉粗糙，需要打磨
- 4-6/10：功能缺失但架构正确，需要补齐
- 1-3/10：功能缺失且实现错误
- 0/10：完全缺失或严重违反约束
