# HUD 信息显示原则 & 优先级体系（草案 v0.1）

> 定位：本文档既是 HUD Demo 和布局分析的输入约束，也是 agent 需要用人因工程研究验证和修正的对象
> 部署路径：`docs/specs/hud_display_principles.md`
> 标注说明：[待验证] = 需要 agent 通过研究确认或修正的数值/判断

## 一、核心设计原则

### 1.1 安全第一原则
HUD 存在的唯一理由是让骑手更安全，不是更酷。任何信息如果会分散骑手对路面的注意力，就不应该显示。

具体规则：
- **中央视野零遮挡**：骑手正前方视野的核心区域不放任何 HUD 信息
- **一瞥可读**：任何 HUD 信息必须在单次余光扫视中完成理解，不需要持续注视 [待验证：扫视时间上限建议 0.5 秒以内]
- **最少信息量**：任意时刻 HUD 上同时显示的信息元素不超过 3 个 [待验证：认知负荷研究是否支持这个上限]
- **宁可不显示**：当系统无法确定信息是否需要显示时，默认不显示

### 1.2 渐进披露原则
信息按需出现，不按需消失。

- 平时：只显示最低限度的常驻信息（速度 + 连接状态图标）
- 事件触发：相关信息出现在对应区域
- 事件结束：信息在合理延迟后消失（不是瞬间消失，给骑手确认时间）
- 高危场景：信息主动推送到骑手视野，不等骑手去找

### 1.3 速度自适应原则
骑手速度越快，可处理的附加信息越少、对路面注意力的需求越高。HUD 显示策略必须随速度动态调整。

### 1.4 一致性原则
- 同类信息始终出现在同一个区域位置
- 同一颜色始终代表同一语义（绿=正常，黄=注意，红=危险）
- 骑手不需要学习「这个信息今天在哪」

### 1.5 容错与降级原则
- HUD 故障时，骑手必须仍能安全骑行（HUD 不是刹车，不能成为关键依赖）
- 低电量时优先保留安全类信息，砍掉娱乐类
- 强光下读不清时，降级为高对比图标模式

## 二、速度分级信息策略

将骑行速度分为四档，每档对应不同的信息显示策略。

### 2.1 速度分档定义

| 速度档 | 速度范围 | 典型场景 | 认知特征 |
|--------|---------|---------|---------|
| S0 静止 | 0 km/h | 等红灯、停车查看 | 注意力充裕，可处理复杂信息 |
| S1 低速 | 1-30 km/h | 城市拥堵、小区内、起步 | 注意力较充裕，需关注行人和近距障碍 |
| S2 中速 | 31-80 km/h | 城市主干道、国道 | 注意力中等，视线需频繁扫描路况 |
| S3 高速 | 81 km/h+ | 高速公路、快速路 | 注意力高度集中于前方，余光处理能力有限 |

[待验证：分档阈值是否合理。参考汽车 HUD 行业的速度分级标准和摩托车人因工程研究]

### 2.2 各速度档的信息显示规则

#### S0 静止（0 km/h）

**可显示内容：**
- 所有常驻信息 + 所有事件信息
- 完整导航地图（如有）
- 消息内容预览
- 设置菜单可交互
- 队友详细信息
- OTA 升级提示和进度

**显示特征：**
- 信息密度上限：5-6 个元素 [待验证]
- 可使用小号字体和详细文本
- 动画和过渡效果完整展示
- 这是唯一允许骑手「阅读」而非「扫视」的档位

#### S1 低速（1-30 km/h）

**可显示内容：**
- 速度
- 导航转向提示（简化：箭头 + 距离）
- 来电通知（来电者 + 接听/拒接）
- 安全预警（全部）
- 组队状态（简化：队友数 + 语音图标）
- 录制状态（仅 REC 图标）
- 设备状态告警（电量低、蓝牙断连等）

**不显示：**
- 消息内容预览 → 只显示「1 条新消息」图标
- 音乐详细信息 → 只显示播放状态图标
- OTA 进度
- 设置菜单

**显示特征：**
- 信息密度上限：4 个元素 [待验证]
- 字体放大，高对比度
- 非紧急信息的显示时间缩短

#### S2 中速（31-80 km/h）

**可显示内容：**
- 速度
- 导航转向提示（进一步简化：箭头 + 距离数字）
- 来电通知（来电者名 + 操作提示）
- 安全预警（全部，且预警信息升级为最高优先级独占显示）
- 录制状态（仅 REC 小图标）
- 关键告警（电量极低、蓝牙断连）

**不显示：**
- 音乐任何信息（切歌通过语音反馈，不占 HUD）
- 消息通知
- 队友详细位置 → 只保留掉队告警
- 非关键设备状态
- OTA 任何信息

**显示特征：**
- 信息密度上限：3 个元素 [待验证]
- 只用大号字体和图标
- 非安全信息的显示时长进一步缩短（3 秒后自动消失）[待验证]
- 安全预警可以打断所有其他显示

#### S3 高速（81 km/h+）

**可显示内容：**
- 速度（常驻但最小化）
- 导航仅保留：转向箭头图标（无文字）
- 安全预警（全部，独占 HUD，所有其他信息让路）
- SOS 紧急信息
- 电量极低告警（<10%）

**不显示：**
- 来电通知 → 静默处理，通过语音播报来电者（不占 HUD）
- 音乐
- 消息
- 组队详情
- 录制状态
- 设备状态（除极低电量）
- 所有非安全类信息

**显示特征：**
- 信息密度上限：2 个元素（速度 + 最多 1 个事件）[待验证]
- 仅使用图标和大号数字，零文字
- 安全预警使用全区域闪烁 + 方向指示
- 高速下的显示驻留时间更短（2 秒）[待验证：高速下余光扫视能力]

## 三、全局信息优先级体系

### 3.1 优先级分级（P-Level）

将所有 HUD 信息分为 5 个优先级层，高优先级可抢占低优先级的显示区域。

| 优先级 | 名称 | 含义 | 能否被打断 | 显示行为 |
|--------|------|------|----------|---------|
| **P-CRITICAL** | 生命安全级 | 不看可能死 | 不能 | 强制独占显示，闪烁+全区域 |
| **P-URGENT** | 紧急操作级 | 需要立即操作 | 仅被 CRITICAL 打断 | 抢占主显示区，保留速度 |
| **P-ACTIVE** | 主动任务级 | 骑手正在进行的任务 | 被 URGENT 和 CRITICAL 打断 | 占据指定区域，正常显示 |
| **P-INFO** | 状态信息级 | 知道就好，不需要操作 | 被以上所有打断 | 小图标或短暂弹出 |
| **P-BG** | 背景环境级 | 锦上添花 | 被以上所有打断 | 最小化图标，或不显示 |

| 72 | 充电中 45% | 充电中 | S0 | 小图标+百分比 |

## 四、优先级评判规则

### 4.1 评判流程

当一条新信息需要在 HUD 上显示时，系统按以下流程判断：

```
步骤 1：判断当前速度档位（S0/S1/S2/S3）
    ↓
步骤 2：查询该信息在当前速度档下是否允许显示
    → 不允许 → 不显示（转语音或 App 推送）
    → 允许 → 继续
    ↓
步骤 3：查询该信息的优先级（P-CRITICAL ~ P-BG）
    ↓
步骤 4：检查目标显示区域当前是否有内容
    → 无内容 → 直接显示
    → 有内容 → 比较优先级
    ↓
步骤 5：优先级比较
    → 新信息优先级更高 → 抢占区域，旧信息降级处理
    → 新信息优先级相同 → 按时间排队，或分区域并存
    → 新信息优先级更低 → 排队等待，或缩小显示
```

### 4.2 抢占规则

当高优先信息需要抢占正在显示的低优先信息时：

| 被抢占信息的去向 | 规则 |
|----------------|------|
| 完全隐藏 | 被 P-CRITICAL 抢占时，所有其他信息隐藏 |
| 缩小到图标 | 被 P-URGENT 抢占时，P-ACTIVE 信息缩小为角标图标 |
| 移到次要区域 | 如果有空闲区域，低优先信息平移过去 |
| 恢复 | 高优先信息消失后，被抢占的信息自动恢复原位 |

### 4.3 同优先级竞争规则

同一优先级内多条信息同时需要显示时：

| 场景 | 处理方式 |
|------|---------|
| 2 条 P-CRITICAL 同时触发 | 合并显示（如同时有前方碰撞+盲区车辆 → 两个方向同时闪烁）|
| 多条 P-URGENT 同时存在 | 按子优先级排序：安全类 > 操作类 > 告警类，轮流显示或分区 |
| 导航 + 来电同时存在 | 来电占据一个区域，导航保持在另一个区域，两者共存 |
| 多条 P-INFO 堆积 | 只显示最新一条，其余在 App 端查看 |

### 4.4 信息消失规则

| 信息类型 | 消失条件 | 默认显示时长 |
|---------|---------|------------|
| P-CRITICAL 安全预警 | 威胁解除 | 持续到威胁消除 |
| P-CRITICAL SOS | 用户确认或流程结束 | 持续到流程完成 |
| P-URGENT 来电 | 用户处理（接/拒/超时）| 持续到处理 |
| P-URGENT 导航转向 | 完成转向 + 2秒确认 | 完成转向后 2秒 |
| P-URGENT 告警 | 状况恢复或用户确认 | 持续到恢复 |
| P-ACTIVE 任务确认 | 固定时间 | 2-3秒 [待验证] |
| P-ACTIVE 常驻状态 | 任务结束 | 任务期间持续 |
| P-INFO | 固定时间 | 3-5秒 [待验证] |
| P-BG | 固定时间 | 2-3秒 |

|------|------|---------|---------|

以下标注 [待验证] 的内容需要 agent 在研究 A/B 中用人因工程文献和竞品数据验证或修正：

# 头盔 HUD 光学参数对标综合报告  
**主题：OLED + Free Form 自由曲面 / Micro LED + 树脂衍射光波导量产级参数收集与默认值建议**  
**用途：直接用于 HUD Demo 参数滑块默认值设定**

## 一、执行结论

本次对标的核心结论非常明确：

1. **V1 HUD Demo 默认技术路线建议采用：Micro-OLED + Free-form combiner（自由曲面反射/合像方案）**
2. **V2 或高阶轻薄化路线建议继续验证：Micro-LED + 树脂衍射光波导**
3. **默认参数不应从“参数上限”出发，而应从 HUD 显示原则反推“最小可行光学需求”**
4. **在当前 spec 约束下，头盔 HUD 本质不是 AR 眼镜，而是“极简余光提示器”**，因此首代应优先追求：
   - 白天可读
   - 高速低干扰
   - 头盔可集成
   - 功耗与热可控
   - 供应链与成本可落地

综合团队分析后，最终建议是：  
**V1 采用 Micro-OLED + Free-form，配合中小 FoV、图标级分辨率、较远虚像距离、宽动态亮度、准单色/低色数高对比 UI。**  
这组定义最符合“量产现实 + Demo 可用 + 安全优先”的目标。

---

## 二、为什么不能只看光学先进性

根据 `hud_display_principles.md`，HUD 的唯一理由是提升骑行安全，而不是增加炫技感。因此，光学参数不能脱离显示策略单独讨论。  
Spec 已经明确了几个关键约束：

- **中央视野零遮挡**
- **一瞥可读**
- **最少信息量**
- **速度越高，显示越少**
- **高危信息优先级最高，可抢占其他内容**

尤其在 S2（31–80 km/h）和 S3（81 km/h+）档位下，HUD 显示内容被严格压缩：

- S2：信息密度上限 **3 个元素**
- S3：信息密度上限 **2 个元素**
- S3 仅保留：**速度、导航箭头、安全预警、SOS、电量极低告警**

这意味着首代 HUD 不需要承载复杂地图、长文本、多卡片并列，也不需要追求沉浸式大视场。  
真正关键的是：  
**在 0.3–0.5 秒扫视内，骑手能否识别箭头方向、速度数字和告警等级。**

因此，光学目标应从“显示更多”转向“以最低打扰成本稳定显示最关键的信息”。

---

## 三、两条主路线量产级参数对标

### 1. 路线 A：Micro-OLED + Free-form combiner

这是当前最现实、最适合作为 V1 Demo 默认值的方案。团队修复后的量产级参数如下：

- **分辨率：1024×768**
- **FOV：18° 对角**
- **虚像距离：2.5 m**
- **入眼亮度：1200 nits**
- **Eye box：8×8 mm**
- **透过率：85%**
- **光机体积：8–15 cc**
- **光学效率：15%–30%**
- **BOM（光学+显示）：$80–150**

#### 优势
- 光学效率高，**15%–30%** 明显优于波导路线
- 白天可读性更容易达标，**1200 nits 入眼亮度**更适合户外强光
- 供应链成熟，6–12 个月内更容易打样、迭代、认证
- 调试路径短，系统风险低
- 在中小 FoV 下更容易获得稳定像质和亮度

#### 劣势
- **Eye box 8×8 mm** 仍偏紧，对佩戴姿态和装调公差敏感
- 光机体积 **8–15 cc** 对头盔侧边/眉部集成仍有压力
- 机械结构、重心和震动稳定性需要重点优化

#### 结论
如果目标是“最接近量产现实的 HUD Demo 默认值”，A 路线是当前最优解。

---

### 2. 路线 B：Micro-LED + 树脂衍射光波导

这是更轻薄、更前沿的路线，但当前更适合作为 V2 验证方向。量产级参数如下：

- **分辨率：640×480**
- **FOV：16°**
- **虚像距离：2.0 m**
- **入眼亮度：1000 nits**
- **Eye box：10×8 mm**
- **透过率：82%**
- **光学效率：0.1%–1%**
- **BOM：$180–400**

#### 优势
- 结构更薄，更有未来感
- **Eye box 10×8 mm** 略优于 OLED 路线
- 更适合长期轻薄化、一体化外观方向

#### 劣势
- **光学效率仅 0.1%–1%**，亮度链路压力极大
- 即使入眼亮度目标为 **1000 nits**，系统端往往需要更高发光功率，带来热和功耗问题
- 彩虹纹、均匀性、耦合/出耦公差、良率风险高
- BOM **$180–400**，显著高于 OLED 路线
- 供应链成熟度和量产稳定性仍弱于自由曲面方案

#### 结论
B 路线不是不能做，而是**不适合成为 V1 默认值**。  
但如果显示策略足够克制，即严格遵守 S2/S3 极简显示原则，波导路线会获得新的可行窗口：小 FoV、低分辨率、单色/有限色、图标化显示，都能显著降低其量产难度。因此它应被定义为 **V2/高阶版重点验证路线**，而不是被彻底否定。

## 四、被排除方案及排除理由

为了证明推荐结论不是先验偏好，而是筛选后的结果，团队也补充了其他路线比较。

### 1. LCoS + Free-form
问题在于：
- 需要偏振器、PBS 等附加器件，光机更复杂
- 系统链路损耗不低
- 功耗和热负担更高
- 小型化和头盔集成不如 Micro-OLED 顺手

**结论：**在“尽快做出可读、可装进头盔的 Demo”目标下，不优于 Micro-OLED + Free-form。

### 2. DLP + combiner
问题在于：
- 体积大
- 功耗高
- 散热压力大
- 对电池头盔极不友好
- 更适合车载/工业，不适合轻量头盔

**结论：**可做实验室样机，但不适合作为量产默认参数基线。

### 3. 单色 Micro-OLED / 单色显示方案
这不是被排除，而是被重新定义为 **A 路线的配置收敛版本**。

优点：
- 亮度利用率更高
- 功耗更低
- UI 语义更简单
- 白天可读性更稳

缺点：
- 表达能力受限
- 产品观感不如全彩

**结论：**非常适合作为 V1 默认显示策略。  
即：**光学路线选 A，但内容层优先采用准单色高对比方案。**

### 4. 单色 waveguide / visor combiner 极简方案
- 单色 waveguide 虽比全彩波导低风险，但仍受效率、均匀性、公差和供应链限制
- visor combiner 极简反射方案虽然成本低、出样快，但虚像质量、鬼像、外观一致性和 visor 曲率适配问题明显

**结论：**
- 单色 waveguide：仍不如 A 稳妥
- visor combiner：适合 EVT 概念样机，不适合作为量产默认值基线

## 五、从显示原则反推光学参数下限

这是本次整合中最重要的修正：  
**HUD 光学参数不是先验固定值，而应由显示策略反推最小可行需求。**

### 1. FoV：取小不取大
在 S2/S3 档位下，HUD 只需容纳：
- 一个主箭头
- 一组速度数字
- 一个告警/状态区

因此首代不需要沉浸式大视场，而应采用**中小 FoV**。  
这会带来直接收益：
- 光机更小
- 畸变和边缘像质压力下降
- 波导耦出效率压力下降
- 佩戴适配更容易

结合现有对标数据，**18° 对角（OLED）和 16°（波导）**都落在合理区间内。

### 2. 分辨率：以图标和粗字可读为准
如果不显示长文本、地图细节、复杂菜单，则分辨率目标可明显下调。  
首代标准应是：
- 数字一瞥可辨
- 箭头方向零歧义
- 告警等级不误判

因此：
- **1024×768** 对 OLED 路线是稳妥值
- **640×480** 对波导路线在极简 UI 下也可接受

### 3. 虚像距离：可更远
HUD 只承担提示，不承担精细阅读，因此更远虚像距离反而更优：
- 更接近道路观察焦点
- 减少频繁调焦
- 降低近眼压迫感

现有对标中：
- OLED 路线 **2.5 m**
- 波导路线 **2.0 m**

从安全和舒适性看，**2.5 m 更适合作为 V1 默认值**。

### 4. Eye-box：不能过度压缩
这里不能因为极简显示就放松要求。骑行头部姿态变化、头盔佩戴差异、是否戴眼镜，都会放大眼位偏移问题。  
因此真正不该牺牲的是：
- 眼盒宽容度
- 出瞳一致性
- 轻微偏头时的可见性

现有数据中：
- OLED：**8×8 mm**
- 波导：**10×8 mm**

虽然波导略优，但考虑整体系统风险，V1 仍应接受 **8×8 mm** 作为现实基线，同时在机械设计和佩戴调节上补偿。

### 5. 亮度：看动态范围，不只看峰值
Spec 已明确“强光下读不清时，降级为高对比图标模式”。  
因此亮度设计重点不是单纯堆峰值，而是：
- 白天足够亮
- 夜间足够暗
- 自动调光平滑
- 不闪烁、不突兀

现有对标值：
- OLED：**1200 nits 入眼亮度**
- 波导：**1000 nits 入眼亮度**

结合效率差异，**1200 nits + 高对比 UI + 昼夜双模式**是更稳妥的 V1 设定。

---

## 六、与 HUD 信息显示原则的匹配结论

结合 spec 中速度分级和优先级体系，可以得出以下判断：

1. **S2/S3 极简显示原则显著降低了首代 HUD 对大 FoV、高分辨率的需求**
2. **高速场景并不是最苛刻的光学场景**，因为显示内容最少
3. **真正的系统瓶颈不是“显示多少”，而是“在 0.3–0.5 秒扫视内稳定识别”**
4. **昼夜双模式是必须项**，且应与高对比图标化 UI 联动
5. **首代产品应优先冻结导航箭头、速度、危险告警三类核心内容，再反推光学参数**

这意味着：  
**A 路线不仅在器件层面更成熟，也更符合当前 HUD 显示原则的产品定义。**

## 八、最终结论

如果目标是“**最接近量产现实的头盔 HUD Demo 默认值**”，那么经过技术路线、显示原则、系统风险、成本和集成难度的综合筛选后，明确结论如下：

**首代 V1 应选择 Micro-OLED + Free-form，而不是波导、DLP、LCoS 或极简 visor 方案。**  
同时，**默认参数必须服从 HUD 的安全显示原则**，即以 S2/S3 极简显示为核心约束，采用中小 FoV、图标级分辨率、较远虚像距离、宽动态亮度和准单色高对比 UI。

一句话总结：  
**V1 做“最简但可靠”的 HUD，选 OLED + Free-form；V2 再做“更薄更未来”的波导验证。**
