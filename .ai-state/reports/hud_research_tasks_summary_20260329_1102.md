# 头盔 HUD 光学参数对标综合报告  
**主题：OLED + Free Form 自由曲面 / Micro LED + 树脂衍射光波导量产级参数收集与默认值建议**  
**用途：直接用于 HUD Demo 参数滑块默认值设定**

## 一、执行结论

本次对标的核心结论非常明确：

1. **V1 HUD Demo 默认技术路线建议采用：Micro-OLED + Free-form combiner（自由曲面反射/合像方案）**
2. **V2 或高阶轻薄化路线建议继续验证：Micro-LED + 树脂衍射光波导**
3. **默认参数不应从“参数上限”出发，而应从 HUD 显示原则反推“最小可行光学需求”**
4. **在当前 spec 约束下，头盔 HUD 本质不是 AR 眼镜，而是“极简余光提示器”**，因此首代应优先追求：
   - 白天可读
   - 高速低干扰
   - 头盔可集成
   - 功耗与热可控
   - 供应链与成本可落地

综合团队分析后，最终建议是：  
**V1 采用 Micro-OLED + Free-form，配合中小 FoV、图标级分辨率、较远虚像距离、宽动态亮度、准单色/低色数高对比 UI。**  
这组定义最符合“量产现实 + Demo 可用 + 安全优先”的目标。

---

## 二、为什么不能只看光学先进性

根据 `hud_display_principles.md`，HUD 的唯一理由是提升骑行安全，而不是增加炫技感。因此，光学参数不能脱离显示策略单独讨论。  
Spec 已经明确了几个关键约束：

- **中央视野零遮挡**
- **一瞥可读**
- **最少信息量**
- **速度越高，显示越少**
- **高危信息优先级最高，可抢占其他内容**

尤其在 S2（31–80 km/h）和 S3（81 km/h+）档位下，HUD 显示内容被严格压缩：

- S2：信息密度上限 **3 个元素**
- S3：信息密度上限 **2 个元素**
- S3 仅保留：**速度、导航箭头、安全预警、SOS、电量极低告警**

这意味着首代 HUD 不需要承载复杂地图、长文本、多卡片并列，也不需要追求沉浸式大视场。  
真正关键的是：  
**在 0.3–0.5 秒扫视内，骑手能否识别箭头方向、速度数字和告警等级。**

因此，光学目标应从“显示更多”转向“以最低打扰成本稳定显示最关键的信息”。

---

## 三、两条主路线量产级参数对标

### 1. 路线 A：Micro-OLED + Free-form combiner

这是当前最现实、最适合作为 V1 Demo 默认值的方案。团队修复后的量产级参数如下：

- **分辨率：1024×768**
- **FOV：18° 对角**
- **虚像距离：2.5 m**
- **入眼亮度：1200 nits**
- **Eye box：8×8 mm**
- **透过率：85%**
- **光机体积：8–15 cc**
- **光学效率：15%–30%**
- **BOM（光学+显示）：$80–150**

#### 优势
- 光学效率高，**15%–30%** 明显优于波导路线
- 白天可读性更容易达标，**1200 nits 入眼亮度**更适合户外强光
- 供应链成熟，6–12 个月内更容易打样、迭代、认证
- 调试路径短，系统风险低
- 在中小 FoV 下更容易获得稳定像质和亮度

#### 劣势
- **Eye box 8×8 mm** 仍偏紧，对佩戴姿态和装调公差敏感
- 光机体积 **8–15 cc** 对头盔侧边/眉部集成仍有压力
- 机械结构、重心和震动稳定性需要重点优化

#### 结论
如果目标是“最接近量产现实的 HUD Demo 默认值”，A 路线是当前最优解。

---

### 2. 路线 B：Micro-LED + 树脂衍射光波导

这是更轻薄、更前沿的路线，但当前更适合作为 V2 验证方向。量产级参数如下：

- **分辨率：640×480**
- **FOV：16°**
- **虚像距离：2.0 m**
- **入眼亮度：1000 nits**
- **Eye box：10×8 mm**
- **透过率：82%**
- **光学效率：0.1%–1%**
- **BOM：$180–400**

#### 优势
- 结构更薄，更有未来感
- **Eye box 10×8 mm** 略优于 OLED 路线
- 更适合长期轻薄化、一体化外观方向

#### 劣势
- **光学效率仅 0.1%–1%**，亮度链路压力极大
- 即使入眼亮度目标为 **1000 nits**，系统端往往需要更高发光功率，带来热和功耗问题
- 彩虹纹、均匀性、耦合/出耦公差、良率风险高
- BOM **$180–400**，显著高于 OLED 路线
- 供应链成熟度和量产稳定性仍弱于自由曲面方案

#### 结论
B 路线不是不能做，而是**不适合成为 V1 默认值**。  
但如果显示策略足够克制，即严格遵守 S2/S3 极简显示原则，波导路线会获得新的可行窗口：小 FoV、低分辨率、单色/有限色、图标化显示，都能显著降低其量产难度。因此它应被定义为 **V2/高阶版重点验证路线**，而不是被彻底否定。

---

## 四、被排除方案及排除理由

为了证明推荐结论不是先验偏好，而是筛选后的结果，团队也补充了其他路线比较。

### 1. LCoS + Free-form
问题在于：
- 需要偏振器、PBS 等附加器件，光机更复杂
- 系统链路损耗不低
- 功耗和热负担更高
- 小型化和头盔集成不如 Micro-OLED 顺手

**结论：**在“尽快做出可读、可装进头盔的 Demo”目标下，不优于 Micro-OLED + Free-form。

### 2. DLP + combiner
问题在于：
- 体积大
- 功耗高
- 散热压力大
- 对电池头盔极不友好
- 更适合车载/工业，不适合轻量头盔

**结论：**可做实验室样机，但不适合作为量产默认参数基线。

### 3. 单色 Micro-OLED / 单色显示方案
这不是被排除，而是被重新定义为 **A 路线的配置收敛版本**。

优点：
- 亮度利用率更高
- 功耗更低
- UI 语义更简单
- 白天可读性更稳

缺点：
- 表达能力受限
- 产品观感不如全彩

**结论：**非常适合作为 V1 默认显示策略。  
即：**光学路线选 A，但内容层优先采用准单色高对比方案。**

### 4. 单色 waveguide / visor combiner 极简方案
- 单色 waveguide 虽比全彩波导低风险，但仍受效率、均匀性、公差和供应链限制
- visor combiner 极简反射方案虽然成本低、出样快，但虚像质量、鬼像、外观一致性和 visor 曲率适配问题明显

**结论：**
- 单色 waveguide：仍不如 A 稳妥
- visor combiner：适合 EVT 概念样机，不适合作为量产默认值基线

## 五、从显示原则反推光学参数下限

这是本次整合中最重要的修正：  
**HUD 光学参数不是先验固定值，而应由显示策略反推最小可行需求。**

### 1. FoV：取小不取大
在 S2/S3 档位下，HUD 只需容纳：
- 一个主箭头
- 一组速度数字
- 一个告警/状态区

因此首代不需要沉浸式大视场，而应采用**中小 FoV**。  
这会带来直接收益：
- 光机更小
- 畸变和边缘像质压力下降
- 波导耦出效率压力下降
- 佩戴适配更容易

结合现有对标数据，**18° 对角（OLED）和 16°（波导）**都落在合理区间内。

### 2. 分辨率：以图标和粗字可读为准
如果不显示长文本、地图细节、复杂菜单，则分辨率目标可明显下调。  
首代标准应是：
- 数字一瞥可辨
- 箭头方向零歧义
- 告警等级不误判

因此：
- **1024×768** 对 OLED 路线是稳妥值
- **640×480** 对波导路线在极简 UI 下也可接受

### 3. 虚像距离：可更远
HUD 只承担提示，不承担精细阅读，因此更远虚像距离反而更优：
- 更接近道路观察焦点
- 减少频繁调焦
- 降低近眼压迫感

现有对标中：
- OLED 路线 **2.5 m**
- 波导路线 **2.0 m**

从安全和舒适性看，**2.5 m 更适合作为 V1 默认值**。

### 4. Eye-box：不能过度压缩
这里不能因为极简显示就放松要求。骑行头部姿态变化、头盔佩戴差异、是否戴眼镜，都会放大眼位偏移问题。  
因此真正不该牺牲的是：
- 眼盒宽容度
- 出瞳一致性
- 轻微偏头时的可见性

现有数据中：
- OLED：**8×8 mm**
- 波导：**10×8 mm**

虽然波导略优，但考虑整体系统风险，V1 仍应接受 **8×8 mm** 作为现实基线，同时在机械设计和佩戴调节上补偿。

### 5. 亮度：看动态范围，不只看峰值
Spec 已明确“强光下读不清时，降级为高对比图标模式”。  
因此亮度设计重点不是单纯堆峰值，而是：
- 白天足够亮
- 夜间足够暗
- 自动调光平滑
- 不闪烁、不突兀

现有对标值：
- OLED：**1200 nits 入眼亮度**
- 波导：**1000 nits 入眼亮度**

结合效率差异，**1200 nits + 高对比 UI + 昼夜双模式**是更稳妥的 V1 设定。

## 六、与 HUD 信息显示原则的匹配结论

结合 spec 中速度分级和优先级体系，可以得出以下判断：

1. **S2/S3 极简显示原则显著降低了首代 HUD 对大 FoV、高分辨率的需求**
2. **高速场景并不是最苛刻的光学场景**，因为显示内容最少
3. **真正的系统瓶颈不是“显示多少”，而是“在 0.3–0.5 秒扫视内稳定识别”**
4. **昼夜双模式是必须项**，且应与高对比图标化 UI 联动
5. **首代产品应优先冻结导航箭头、速度、危险告警三类核心内容，再反推光学参数**

这意味着：  
**A 路线不仅在器件层面更成熟，也更符合当前 HUD 显示原则的产品定义。**

## 八、最终结论

如果目标是“**最接近量产现实的头盔 HUD Demo 默认值**”，那么经过技术路线、显示原则、系统风险、成本和集成难度的综合筛选后，明确结论如下：

**首代 V1 应选择 Micro-OLED + Free-form，而不是波导、DLP、LCoS 或极简 visor 方案。**  
同时，**默认参数必须服从 HUD 的安全显示原则**，即以 S2/S3 极简显示为核心约束，采用中小 FoV、图标级分辨率、较远虚像距离、宽动态亮度和准单色高对比 UI。

一句话总结：  
**V1 做“最简但可靠”的 HUD，选 OLED + Free-form；V2 再做“更薄更未来”的波导验证。**

## HUD 显示布局方案深度分析

# HUD 显示布局方案深度分析综合报告

## 一、执行摘要

本次分析目标，是在四种 HUD 信息布局方案及其动态组合中，选出能够直接指导 HUD Demo 视觉架构的推荐方案。结论非常明确：

**最终产品目标方案推荐 D：自适应混合布局；HUD Demo / V1 推荐采用 B：三分区固定布局作为工程骨架，并按 D 的规则预留演进能力。**

这是一个“双层结论”：

- **架构最优解：D 自适应混合布局**
- **当前阶段最优工程解：B 三分区固定布局**
- **交互上位原则：必须叠加多模态分流，不把所有信息都留在 HUD 上**

换句话说，**B 是 V1 最稳妥的可交付方案，D 是中期必须演进到的正式架构。**

原因有三点：

1. **从量化评分看，D 总分最高 4.34，B 次高 4.02**，明显优于 A 的 2.82 和 C 的 3.16。
2. **从硬件约束看，15–20°FOV、小视场、720p 级微显示条件下，纯四角分布式 C 风险过高**；而 A 虽简单，但并发冲突严重。
3. **从骑行安全看，HUD 优化不能只讨论“怎么排”，还必须讨论“哪些信息不该显示”**。因此最终方案必须结合速度分级、优先级抢占和语音/触觉分流。

---

## 二、研究前提与约束

本次分析严格基于 `hud_display_principles.md` 的原则体系展开，核心约束包括：

- **中央视野零遮挡**
- **一瞥可读**
- **最少信息量**
- **速度自适应**
- **一致性**
- **容错与降级**

同时采用以下量化硬指标作为裁决基线：

### 1. 速度分档
- **S1**：低速/城市慢行
- **S2**：中速巡航
- **S3**：高速/高风险

### 2. 四个硬指标
1. **单次扫视理解时间**：S1 ≤0.5s，S3 ≤0.35s  
2. **同时可感知元素数**：S1 ≤4，S2 ≤3，S3 ≤2  
3. **中心遮挡面积**：核心前视区 0%  
4. **高优先级抢占切换**：P-CRITICAL / P-URGENT <150ms

这些指标与约束文件中的待验证项高度一致，也与速度分级策略形成闭环：  
- S1 信息密度上限 4 个元素  
- S2 信息密度上限 3 个元素  
- S3 信息密度上限 2 个元素  

因此，本次方案评估不是单纯视觉审美比较，而是**围绕安全、认知负荷、并发调度、FOV 适配和工程可落地性**的综合裁决。

---

## 三、四种布局方案定义

### A 单卡片中心下偏轮播
所有主要信息在单一区域轮播显示，扫视路径最短，但所有事件竞争同一显示位。

### B 三分区固定布局
将 HUD 划分为主区、次区、侧区：
- 主区：导航
- 次区：速度 + 极简状态
- 侧区：来电/组队/录制等短时事件

### C 四角分布式布局
信息分布在四角，中央尽量留空，方向表达能力强，但搜索成本高。

### D 自适应混合布局
平时采用 B 的固定骨架；危险时切入方向化预警；S3 高速时自动极简。  
本质上是 **B + 方向化预警层 + 速度自适应极简策略**。

---

## 四、量化对比结果

### 1. 关键指标对比

| 指标 | A 单区轮播 | B 三分区固定 | C 四角分布 | D 自适应混合 |
|---|---:|---:|---:|---:|
| S1元素容量 | 3 | 4 | 4 | 4 |
| S2元素容量 | 2 | 3 | 3 | 3 |
| S3元素容量 | 1–2 | 2 | 2 | 2 |
| 平均扫视路径长度（相对值） | 1.0 | 1.2 | 1.6 | 1.1/1.4 |
| 并发冲突次数（每10事件） | 6 | 3 | 2 | 1 |
| 抢占恢复次数（每10事件） | 6 | 4 | 3 | 2 |
| 实现复杂度（1低5高） | 1 | 2 | 3 | 5 |
| 测试复杂度（1低5高） | 1 | 2 | 4 | 5 |
| 20°FOV适配风险（1低5高） | 1 | 2 | 5 | 3 |
| 方向表达能力（1低5高） | 1 | 2 | 5 | 5 |

其中 D 的平均扫视路径长度写作 **1.1/1.4**，含义是：
- 正常态接近 B，为 **1.1**
- 预警态接近 C，为 **1.4**
- 但方向化只短时启用，因此综合扫视成本仍低于纯 C 的 **1.6**

### 2. 数据解读

#### A 的问题
A 的优势是扫视路径最短，平均值 **1.0**，实现复杂度和测试复杂度都只有 **1**。  
但它的致命问题也最明显：
- 并发冲突次数 **每10事件 6 次**
- 抢占恢复次数 **每10事件 6 次**

这意味着在真实骑行中，用户会频繁面对“同一区域不断被不同事件抢占”的情况。骑手首先要判断“现在弹出来的是什么”，而不是“看到了就懂”。这与“一瞥可读”原则冲突。  
**结论：A 适合最小 Demo，不适合作为正式架构。**

#### B 的优势
B 在各项指标上最均衡：
- S1/S2/S3 元素容量分别为 **4 / 3 / 2**
- 平均扫视路径长度 **1.2**
- 并发冲突次数 **每10事件 3 次**
- 抢占恢复次数 **每10事件 4 次**
- 实现复杂度 **2**
- 测试复杂度 **2**
- 20°FOV 适配风险 **2**

这说明 B 在小视场、有限研发资源下，能够较好平衡安全、可读性和工程成本。  
**结论：B 是当前阶段最稳的基线方案。**

#### C 的问题
C 的方向表达能力最高，为 **5**，并发冲突次数也较低，为 **每10事件 2 次**。  
但它的问题同样突出：
- 平均扫视路径长度 **1.6**，四方案中最高
- 20°FOV 适配风险 **5**，四方案中最差
- 测试复杂度 **4**

在 20°FOV 小视场下，边角信息可读性和稳定性都存在明显风险，高速场景下认知负荷也最大。  
**结论：C 不能作为主架构，只适合作为方向化预警能力的局部借鉴。**

#### D 的优势与代价
D 的综合表现最好：
- 并发冲突次数 **每10事件 1 次**
- 抢占恢复次数 **每10事件 2 次**
- 方向表达能力 **5**
- 20°FOV 适配风险 **3**
- 元素容量满足 S1/S2/S3 的 **4 / 3 / 2**

但代价也最大：
- 实现复杂度 **5**
- 测试复杂度 **5**

这意味着 D 是正确方向，但不适合在 V1 直接全量落地，否则资源会被状态机、动画、异常恢复和验证闭环大量消耗。  
**结论：D 是正式架构最优解，但不应作为 V1 的一次性交付目标。**

## 六、为什么最终不是“只选一个方案”

如果只从视觉布局内部比较，D 显然是最优。  
但从产品落地角度，必须同时考虑：

1. **当前硬件 FOV 限制**
2. **V1 研发与测试资源**
3. **速度分级下的信息迁移**
4. **HUD 不应承担全部信息表达**

因此，最终决策不能是“只选 D”或“只选 B”，而应是：

### 结论一：V1 采用 B 三分区固定布局
作为 HUD Demo 和首版视觉架构骨架。

### 结论二：V2 演进到 D 自适应混合布局
在 B 的基础上增加：
- 方向化预警层
- 高速极简模式
- 更完整的优先级抢占与恢复机制

### 结论三：必须引入多模态分流作为上位策略
这点是 CDO 修正后补足的关键。  
真正更优的方案，不是把 HUD 做得更满，而是让骑手在关键场景下**少看一眼，也能更早做对判断**。

## 七、多模态分流：哪些信息不该继续留在 HUD

这是本次综合报告必须明确补充的部分。  
HUD 视觉布局只是“怎么显示”，但产品还必须回答“哪些信息根本不该显示”。

### 1. 导航信息迁移规则
- **S0/S1**：HUD 保留转向箭头、距离、简单车道提示
- **S2**：HUD 仅保留“下一动作箭头 + 距离”，复杂路口细节转语音
- **S3**：**约 60%-80% 导航信息应转出 HUD**，只留一个方向化主提示；提前量、路名、复杂分叉说明交给语音，必要时辅以触觉节奏提示

### 2. 来电信息迁移规则
- **S0/S1**：HUD 可显示来电人 + 接听/拒接提示
- **S2**：HUD 只显示来电图标与接听状态，联系人名改语音播报
- **S3**：**约 80%-100% 来电信息应转出 HUD**，默认仅语音播报“某某来电”，HUD 最多闪现接听状态，不应要求视觉决策

这与约束文件中 S3 “来电通知不显示，语音播报替代”的原则一致，结论是：  
**不是过于激进，而是合理且必要。**

### 3. 预警信息迁移规则
- **S0/S1**：HUD 可显示图标 + 简短文本
- **S2**：HUD 保留方向化危险符号，文本尽量取消，语音补充危险类型
- **S3**：**约 50%-70% 预警解释信息应转出 HUD**，HUD 只保留最高优先级方向化警示；真正的抢占由触觉/语音完成

因此，高速场景下 HUD 的职责不是“解释危险”，而是“让危险被立刻感知”。

## 八、推荐落地方案

## V1：B 三分区固定布局
建议直接用于 HUD Demo。

### 布局结构
- **主区**：导航
- **次区**：速度 + 极简状态
- **侧区**：来电/组队/录制等短时事件
- **P-CRITICAL**：全局抢占
- **S3**：自动退化为“速度 + 导航箭头”两元素

### 采用原因
- 在 15–20°FOV、720p 级微显示条件下最稳
- 实现复杂度 **2**，测试复杂度 **2**
- 20°FOV 适配风险 **2**
- 加权总分 **4.02**

## V2：D 自适应混合布局
在 B 骨架上升级，不推翻重做。

### 增强能力
- 正常态沿用 B
- 预警态切入方向化警示层
- 高速态自动极简
- 来电/消息在 S3 仅保留语音，不占 HUD

### 采用原因
- 综合评分最高 **4.34**
- 并发冲突最低 **每10事件 1 次**
- 抢占恢复最低 **每10事件 2 次**
- 方向表达能力 **5**

## 九、关键风险

1. **20°FOV 小视场风险**  
纯 C 不适合作主布局；D 也必须限制方向化元素只在预警短时出现。

2. **亮度不等于可读性**  
即使达到 **3000 nits**，仍必须坚持高对比、粗字重、少文字。

3. **高速认知负荷**  
S3 必须严格执行 **2 个元素上限**，并满足 **S3 ≤0.35s** 的单次扫视理解时间。

4. **方向映射正确率**  
左/右闪烁、左上/右上方向映射是否能被稳定理解，必须通过封闭场地验证。

5. **状态机复杂度**  
D 的实现复杂度和测试复杂度均为 **5**，必须在 B 骨架稳定后再逐步引入。

## 十、最终结论

本次综合分析的明确结论如下：

**第一，HUD Demo 与 V1 视觉架构，推荐采用 B：三分区固定布局。**  
因为它在安全性、可读性、20°FOV 适配和工程成本之间达到最佳平衡，是当前阶段最稳妥的可交付方案。

**第二，中期正式架构，推荐演进到 D：自适应混合布局。**  
因为 D 在量化评分中以 **4.34** 位列第一，明显优于 B 的 **4.02**，是长期最优解。

**第三，产品策略上必须引入多模态分流。**  
HUD 不是信息总入口。尤其在 S2/S3 场景下，应主动把导航细节、来电身份、预警解释迁移到语音与触觉：
- S3 导航转出 HUD **60%-80%**
- S3 来电转出 HUD **80%-100%**
- S3 预警解释转出 HUD **50%-70%**

**第四，A 和 C 均不应作为正式主架构。**
- A：总分 **2.82**，并发冲突过高
- C：总分 **3.16**，20°FOV 风险过大

因此，最终建议可以浓缩为一句话：

**V1 用 B 做骨架，V2 升级到 D，并以多模态分流作为上位原则。**  
这既满足当前 Demo 和工程交付要求，也保证后续产品架构不会走向“HUD 越做越满、骑手越看越累”的错误方向。
