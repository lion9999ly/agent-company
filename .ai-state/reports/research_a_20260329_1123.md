# HUD 信息显示原则 & 优先级体系（草案 v0.1）

> 定位：本文档既是 HUD Demo 和布局分析的输入约束，也是 agent 需要用人因工程研究验证和修正的对象
> 部署路径：`docs/specs/hud_display_principles.md`
> 标注说明：[待验证] = 需要 agent 通过研究确认或修正的数值/判断

## 一、核心设计原则

### 1.1 安全第一原则
HUD 存在的唯一理由是让骑手更安全，不是更酷。任何信息如果会分散骑手对路面的注意力，就不应该显示。

具体规则：
- **中央视野零遮挡**：骑手正前方视野的核心区域不放任何 HUD 信息
- **一瞥可读**：任何 HUD 信息必须在单次余光扫视中完成理解，不需要持续注视 [待验证：扫视时间上限建议 0.5 秒以内]
- **最少信息量**：任意时刻 HUD 上同时显示的信息元素不超过 3 个 [待验证：认知负荷研究是否支持这个上限]
- **宁可不显示**：当系统无法确定信息是否需要显示时，默认不显示

### 1.2 渐进披露原则
信息按需出现，不按需消失。

- 平时：只显示最低限度的常驻信息（速度 + 连接状态图标）
- 事件触发：相关信息出现在对应区域
- 事件结束：信息在合理延迟后消失（不是瞬间消失，给骑手确认时间）
- 高危场景：信息主动推送到骑手视野，不等骑手去找

### 1.3 速度自适应原则
骑手速度越快，可处理的附加信息越少、对路面注意力的需求越高。HUD 显示策略必须随速度动态调整。

### 1.4 一致性原则
- 同类信息始终出现在同一个区域位置
- 同一颜色始终代表同一语义（绿=正常，黄=注意，红=危险）
- 骑手不需要学习「这个信息今天在哪」

### 1.5 容错与降级原则
- HUD 故障时，骑手必须仍能安全骑行（HUD 不是刹车，不能成为关键依赖）
- 低电量时优先保留安全类信息，砍掉娱乐类
- 强光下读不清时，降级为高对比图标模式

## 二、速度分级信息策略

将骑行速度分为四档，每档对应不同的信息显示策略。

### 2.1 速度分档定义

| 速度档 | 速度范围 | 典型场景 | 认知特征 |
|--------|---------|---------|---------|
| S0 静止 | 0 km/h | 等红灯、停车查看 | 注意力充裕，可处理复杂信息 |
| S1 低速 | 1-30 km/h | 城市拥堵、小区内、起步 | 注意力较充裕，需关注行人和近距障碍 |
| S2 中速 | 31-80 km/h | 城市主干道、国道 | 注意力中等，视线需频繁扫描路况 |
| S3 高速 | 81 km/h+ | 高速公路、快速路 | 注意力高度集中于前方，余光处理能力有限 |

[待验证：分档阈值是否合理。参考汽车 HUD 行业的速度分级标准和摩托车人因工程研究]

### 2.2 各速度档的信息显示规则

#### S0 静止（0 km/h）

**可显示内容：**
- 所有常驻信息 + 所有事件信息
- 完整导航地图（如有）
- 消息内容预览
- 设置菜单可交互
- 队友详细信息
- OTA 升级提示和进度

**显示特征：**
- 信息密度上限：5-6 个元素 [待验证]
- 可使用小号字体和详细文本
- 动画和过渡效果完整展示
- 这是唯一允许骑手「阅读」而非「扫视」的档位

#### S1 低速（1-30 km/h）

**可显示内容：**
- 速度
- 导航转向提示（简化：箭头 + 距离）
- 来电通知（来电者 + 接听/拒接）
- 安全预警（全部）
- 组队状态（简化：队友数 + 语音图标）
- 录制状态（仅 REC 图标）
- 设备状态告警（电量低、蓝牙断连等）

**不显示：**
- 消息内容预览 → 只显示「1 条新消息」图标
- 音乐详细信息 → 只显示播放状态图标
- OTA 进度
- 设置菜单

**显示特征：**
- 信息密度上限：4 个元素 [待验证]
- 字体放大，高对比度
- 非紧急信息的显示时间缩短

#### S2 中速（31-80 km/h）

**可显示内容：**
- 速度
- 导航转向提示（进一步简化：箭头 + 距离数字）
- 来电通知（来电者名 + 操作提示）
- 安全预警（全部，且预警信息升级为最高优先级独占显示）
- 录制状态（仅 REC 小图标）
- 关键告警（电量极低、蓝牙断连）

**不显示：**
- 音乐任何信息（切歌通过语音反馈，不占 HUD）
- 消息通知
- 队友详细位置 → 只保留掉队告警
- 非关键设备状态
- OTA 任何信息

**显示特征：**
- 信息密度上限：3 个元素 [待验证]
- 只用大号字体和图标
- 非安全信息的显示时长进一步缩短（3 秒后自动消失）[待验证]
- 安全预警可以打断所有其他显示

#### S3 高速（81 km/h+）

**可显示内容：**
- 速度（常驻但最小化）
- 导航仅保留：转向箭头图标（无文字）
- 安全预警（全部，独占 HUD，所有其他信息让路）
- SOS 紧急信息
- 电量极低告警（<10%）

**不显示：**
- 来电通知 → 静默处理，通过语音播报来电者（不占 HUD）
- 音乐
- 消息
- 组队详情
- 录制状态
- 设备状态（除极低电量）
- 所有非安全类信息

**显示特征：**
- 信息密度上限：2 个元素（速度 + 最多 1 个事件）[待验证]
- 仅使用图标和大号数字，零文字
- 安全预警使用全区域闪烁 + 方向指示
- 高速下的显示驻留时间更短（2 秒）[待验证：高速下余光扫视能力]

## 三、全局信息优先级体系

### 3.1 优先级分级（P-Level）

将所有 HUD 信息分为 5 个优先级层，高优先级可抢占低优先级的显示区域。

| 优先级 | 名称 | 含义 | 能否被打断 | 显示行为 |
|--------|------|------|----------|---------|
| **P-CRITICAL** | 生命安全级 | 不看可能死 | 不能 | 强制独占显示，闪烁+全区域 |
| **P-URGENT** | 紧急操作级 | 需要立即操作 | 仅被 CRITICAL 打断 | 抢占主显示区，保留速度 |
| **P-ACTIVE** | 主动任务级 | 骑手正在进行的任务 | 被 URGENT 和 CRITICAL 打断 | 占据指定区域，正常显示 |
| **P-INFO** | 状态信息级 | 知道就好，不需要操作 | 被以上所有打断 | 小图标或短暂弹出 |
| **P-BG** | 背景环境级 | 锦上添花 | 被以上所有打断 | 最小化图标，或不显示 |

| 72 | 充电中 45% | 充电中 | S0 | 小图标+百分比 |

## 四、优先级评判规则

### 4.1 评判流程

当一条新信息需要在 HUD 上显示时，系统按以下流程判断：

```
步骤 1：判断当前速度档位（S0/S1/S2/S3）
    ↓
步骤 2：查询该信息在当前速度档下是否允许显示
    → 不允许 → 不显示（转语音或 App 推送）
    → 允许 → 继续
    ↓
步骤 3：查询该信息的优先级（P-CRITICAL ~ P-BG）
    ↓
步骤 4：检查目标显示区域当前是否有内容
    → 无内容 → 直接显示
    → 有内容 → 比较优先级
    ↓
步骤 5：优先级比较
    → 新信息优先级更高 → 抢占区域，旧信息降级处理
    → 新信息优先级相同 → 按时间排队，或分区域并存
    → 新信息优先级更低 → 排队等待，或缩小显示
```

### 4.2 抢占规则

当高优先信息需要抢占正在显示的低优先信息时：

| 被抢占信息的去向 | 规则 |
|----------------|------|
| 完全隐藏 | 被 P-CRITICAL 抢占时，所有其他信息隐藏 |
| 缩小到图标 | 被 P-URGENT 抢占时，P-ACTIVE 信息缩小为角标图标 |
| 移到次要区域 | 如果有空闲区域，低优先信息平移过去 |
| 恢复 | 高优先信息消失后，被抢占的信息自动恢复原位 |

### 4.3 同优先级竞争规则

同一优先级内多条信息同时需要显示时：

| 场景 | 处理方式 |
|------|---------|
| 2 条 P-CRITICAL 同时触发 | 合并显示（如同时有前方碰撞+盲区车辆 → 两个方向同时闪烁）|
| 多条 P-URGENT 同时存在 | 按子优先级排序：安全类 > 操作类 > 告警类，轮流显示或分区 |
| 导航 + 来电同时存在 | 来电占据一个区域，导航保持在另一个区域，两者共存 |
| 多条 P-INFO 堆积 | 只显示最新一条，其余在 App 端查看 |

### 4.4 信息消失规则

| 信息类型 | 消失条件 | 默认显示时长 |
|---------|---------|------------|
| P-CRITICAL 安全预警 | 威胁解除 | 持续到威胁消除 |
| P-CRITICAL SOS | 用户确认或流程结束 | 持续到流程完成 |
| P-URGENT 来电 | 用户处理（接/拒/超时）| 持续到处理 |
| P-URGENT 导航转向 | 完成转向 + 2秒确认 | 完成转向后 2秒 |
| P-URGENT 告警 | 状况恢复或用户确认 | 持续到恢复 |
| P-ACTIVE 任务确认 | 固定时间 | 2-3秒 [待验证] |
| P-ACTIVE 常驻状态 | 任务结束 | 任务期间持续 |
| P-INFO | 固定时间 | 3-5秒 [待验证] |
| P-BG | 固定时间 | 2-3秒 |

# 头盔 HUD 光学参数对标综合报告  
## —— 面向 Demo 默认参数设定的量产级结论

## 一、项目背景与结论先行

本次研究聚焦两条头盔 HUD 技术路径：

1. **路线 A：Micro OLED + Free Form 自由曲面**
2. **路线 B：Micro LED + 树脂衍射光波导**

研究目标不是做概念性技术展望，而是为 HUD Demo 提供**可直接用于参数滑块默认值设定**的量产级光学参数，并确保这些参数与现有 HUD 信息显示原则相兼容，尤其要支持文档中强调的三项核心要求：

- **中央视野零遮挡**
- **0.5 秒内一瞥可读**
- **高速场景下信息极简、优先安全**

### 明确结论

**V1 Demo 默认推荐路线 A：Micro OLED + Free Form 自由曲面。**  
**V2 或后续高配版本，再评估路线 B：Micro LED + 树脂衍射光波导。**

原因非常明确：  
路线 A 在头盔量产约束下，**参数闭环更完整、供应链更成熟、调试成本更低、6–9 个月内更有机会做出可骑行验证样机**；路线 B 虽然在轻薄化和未来形态上更有吸引力，但在**亮度均匀性、彩虹纹、眼盒一致性、树脂耐候性、量产良率**上风险显著更高，不适合作为首版 Demo 的默认路线。

---

## 二、与 HUD 显示原则的适配判断

在 `hud_display_principles.md` 中，HUD 的本质定位是“安全增强”，而不是“沉浸显示”。这意味着光学参数不能只追求大视场和炫技，而必须服务于以下显示策略：

- S1/S2/S3 速度档下，信息密度上限分别约为 **4 / 3 / 2 个元素**
- 高速 S3 仅保留：**速度、导航箭头、安全预警、SOS、极低电量**
- 安全预警需支持**方向感知、闪烁、独占显示**
- 非安全信息必须在短时间内完成识别并自动消失

由此可推导出对光学系统的真实需求：

1. **FOV 不需要过大**  
   头盔 HUD 首版不是 AR 贴地导航，不需要 30° 以上大视场。  
   对于“箭头 + 距离 + 数字 + 告警图标”类信息，**18–24° 对角 FOV 已足够**。

2. **亮度优先级高于分辨率**  
   在户外强光、透明介质、偏视角观看条件下，真正决定可用性的首先是**到眼亮度**，其次才是分辨率。

3. **眼盒必须兼顾骑行姿态变化**  
   头盔佩戴存在个体差异、头型差异、镜片位置偏差、骑行姿态变化，因此眼盒过小会直接导致“有时看得见、有时看不见”。

4. **虚像距离需要折中**  
   过近会增加调焦负担，过远则可能牺牲系统体积和调试复杂度。  
   对头盔 HUD 来说，**7–10 m 是更现实的首版折中区间**。

因此，从显示原则反推，首版 Demo 最优先验证的不是“最大 FOV”，而是：  
**在 0.5 秒扫视内，骑手能否稳定看见、快速读懂、且不干扰前方道路观察。**

---

## 三、路线 A：Micro OLED + Free Form 自由曲面参数对标

该路线参考了 EyeRide 及近眼显示行业常见量产区间，部分参数属于行业经验值；若非单一公开型号实锤，原分析已标注为 **[不确定]**，本报告保留该标注逻辑。

### 1. 量产级参数区间

| 参数 | 建议默认值 | 量产可行区间 | 说明 |
|---|---:|---:|---|
| 微显示器 | 0.39–0.49" Micro OLED | 0.39–0.71" | Sony / Seeya / eMagin 类供应链成熟 |
| 分辨率 | **1024×768** | 800×600 ~ 1920×1080 | HUD 以图标/箭头/数字为主，XGA 已够 |
| 面板亮度 | **3000–5000 nits** | 2000–7000 nits | 经光机损耗后到眼亮度仍可用 |
| 出瞳亮度（到眼） | **800–1500 nits** | 500–2000 nits | 白天可读的关键指标 |
| FOV（对角） | **18–22°** | 15–25° | EyeRide 公开值约 20° |
| 虚像距离 VID | **7–10 m** | 5–20 m | EyeRide 公开约 21 m，但头盔内更常见 7–10 m 折中 |
| 眼盒 Eye box | **8×6 mm** | 6×4 ~ 10×8 mm | 单目 HUD 够用，但装配公差敏感 |
| 透过率 | **>85% 非显示区** | 80–90% | 取决于 combiner 方案 |
| 畸变 | **<5%** | 3–8% | 需软件预畸变补偿 |
| MTF | **0.25–0.35 @ 20 lp/deg** | [不确定] | 足够显示箭头/数字 |
| 模组功耗 | **1.2–2.5 W** | 1–3 W | 含显示+驱动，不含主控 |
| 光机体积 | **8–15 cc** | 6–20 cc | 头盔内可布置 |
| BOM（光学+显示） | **80–180 美元** | 60–220 美元 | 取决于面板和自由曲面件精度 |

### 2. 对 Demo 的默认滑块建议

- **FOV：20°**
- **VID：8 m**
- **到眼亮度：1200 nits**
- **眼盒：8×6 mm**
- **分辨率：1024×768**
- **显示策略：单色/有限彩色优先，UI 高对比**

### 3. 适配性判断

路线 A 的优势在于它非常贴合当前 HUD 信息结构。  
因为当前显示内容以：

- 速度数字
- 导航箭头 + 距离
- 来电提示
- 安全预警
- SOS / 电量 / 连接状态

为主，这些都属于**低复杂度、高可读性优先**的信息类型，不需要超大视场，也不需要复杂空间叠加。20° 左右 FOV 配合 1024×768 分辨率，已经足以支撑 S0–S3 全部信息层级。

### 4. 风险点

路线 A 不是没有问题，但问题相对可控：

- **眼盒偏小**：8×6 mm 对单目 HUD 可用，但对装配公差和佩戴姿态敏感
- **强光下对比度压力**：需要保证到眼亮度至少落在 800–1500 nits 区间，建议默认 1200 nits
- **自由曲面件装配要求高**：畸变、鬼像、焦距一致性依赖机械夹具和软件校准

总体判断：**可行性高，且风险主要集中在工程调试层面，而不是路线本身不可控。**

## 四、路线 B：Micro LED + 树脂衍射光波导参数对标

该路线本质上更接近 AR 眼镜方案向头盔迁移。理论优势是更轻薄、更接近未来一体化透明显示，但量产难度显著高于路线 A。

### 1. 量产级参数区间

| 参数 | 建议默认值 | 量产可行区间 | 说明 |
|---|---:|---:|---|
| 微显示器 | 0.13–0.3" Micro LED | 0.11–0.3" | 供应链仍偏高端/早期 |
| 分辨率 | **640×480 / 854×480** | 480p ~ 720p | 受尺寸、良率、成本限制 |
| 面板亮度 | **5e5–1e6 nits** | 2e5–2e6 nits | 波导耦入/出损耗极大，源亮度必须非常高 |
| 出瞳亮度（到眼） | **1000–3000 nits** | 500–5000 nits | 取决于波导效率，树脂波导常偏低 |
| FOV（对角） | **20–28°** | 18–35° | FOV 越大，效率/均匀性越难 |
| 虚像距离 VID | **8–12 m** | 5 m ~ ∞ | 波导更容易做远焦 |
| 眼盒 Eye box | **10×8 mm** | 8×6 ~ 12×10 mm | 通过出瞳扩展实现，但均匀性代价大 |
| 透过率 | **80–90%** | 75–90% | 树脂材料黄变/雾化需关注 |
| 彩虹纹/色散 | **中高风险** | — | 衍射波导天然问题 |
| 模组功耗 | **1.5–3.5 W** | 1–5 W | Micro LED 本体高效，但系统不一定低功耗 |
| 光学厚度 | **1.5–3 mm 波导片** | 1–4 mm | 这是它的核心优势 |
| BOM（光学+显示） | **180–400 美元** | 150–500 美元 | 量产初期成本显著高于A |

### 2. 对 Demo 的默认滑块建议

- **FOV：24°**
- **VID：10 m**
- **到眼亮度：1500 nits**
- **眼盒：10×8 mm**
- **分辨率：640×480 或 854×480**
- **显示策略：先做单绿/单色，再谈全彩**

### 3. 优势判断

路线 B 的核心优势主要有三点：

1. **波导片薄，形态更轻巧**  
   光学厚度可做到 **1.5–3 mm**，对头盔集成非常有吸引力。

2. **眼盒理论上更大**  
   通过出瞳扩展可做到 **10×8 mm** 甚至更大，对佩戴容错更友好。

3. **更适合未来 AR 化扩展**  
   如果未来要做更大 FOV、更强空间叠加、更接近眼镜式体验，波导路线更有潜力。

### 4. 风险点

但路线 B 的问题也非常集中，而且多数是 V1 难以承受的系统性风险：

- **光效率低**  
  波导总效率常只有个位数到十几个百分点，原分析标注为 **[不确定，依设计差异大]**。这意味着必须依赖 **5e5–1e6 nits** 甚至更高的源亮度，带来热设计和功耗压力。

- **树脂耐候性风险高**  
  头盔场景不是实验室：有 UV、汗液、温湿循环、清洁剂接触、跌落振动。树脂波导长期可能出现**黄变、翘曲、雾化**。

- **彩虹纹与色散问题明显**  
  对导航箭头、警示图标这类高对比边缘元素，彩边会直接伤害可读性，尤其在白天高亮环境下更明显。

- **亮度均匀性与良率难控**  
  耦入器、出瞳扩展区复制精度、贴合应力、双折射控制都影响最终一致性，量产初期良率压力大。

因此，路线 B 不是不能做，而是**更适合在 V2 阶段作为高配探索，而不是首版默认方案。**

## 五、两条路线的综合对比

### 1. 从 Demo 成功率看

- 路线 A：成功率高，参数闭环清晰
- 路线 B：成功率中等，但量产稳定性风险高

### 2. 从 HUD 显示原则适配看

- 路线 A 更适合“少量信息、一瞥可读、强对比”
- 路线 B 更适合“未来 AR 化扩展”，但当前需求并不需要其全部能力

### 3. 从成本与周期看

- 路线 A：BOM **80–180 美元**，量产区间 **60–220 美元**
- 路线 B：BOM **180–400 美元**，量产区间 **150–500 美元**

路线 B 成本显著高于 A，且调试周期更长。

### 4. 从工程风险看

- 路线 A 风险主要是装配、校准、亮度和眼盒
- 路线 B 风险涉及材料、工艺、效率、均匀性、耐候、良率多个维度

因此，如果当前目标是验证 HUD 在骑行安全场景中的真实价值，而不是展示未来概念，路线 A 明显更优。

## 七、V2 进入波导路线前的验证门槛

若后续要正式推进路线 B，建议必须先完成以下验证：

- **85°C / 85%RH 500h**
- **UV 老化 300h**
- **跌落 / 振动后 MTF 与耦合效率衰减 <10%**
- **白天户外 100k lux 下到眼亮度仍 >1000 nits**
- **彩虹纹主观评分达标**

只有在这些条件满足后，波导路线才具备从“技术展示”走向“量产候选”的基础。

---

## 八、最终结论

本次对标的核心结论非常明确：

### 结论 1：V1 必须优先验证“可读性闭环”，而不是追求 AR 感
头盔 HUD 首版的成功标准不是大视场、轻薄炫酷，而是：
**看得见、看得懂、戴得住、晒不瞎。**

### 结论 2：路线 A 是当前最优解
**Micro OLED + Free Form 自由曲面** 在供应链成熟度、调试效率、成本、量产可控性上都明显优于路线 B，且其参数能力已经足够支撑当前 HUD 信息体系。

### 结论 3：路线 B 适合 V2，不适合 V1 默认
**Micro LED + 树脂衍射光波导** 具备未来潜力，但当前在亮度效率、均匀性、彩虹纹、耐候性和良率上的风险过高，不适合作为首版 Demo 的默认技术路线。

### 最终推荐

**V1 默认方案：20° FOV / 8 m VID / 1200 nits 到眼 / 8×6 mm 眼盒 / 1024×768 / Micro OLED + Free Form**

**备选高配方案：24° FOV / 10 m VID / 1500 nits 到眼 / 10×8 mm 眼盒 / 854×480 / Micro LED + 树脂衍射光波导**

如果只能选一条路做首版，结论不变：

**选 A，不要犹豫。**  
先把骑行安全场景下的信息可读性与系统闭环做扎实，再考虑波导和更强 AR 化扩展。
