# 头盔 HUD 光学参数对标综合报告  
## —— 面向 Demo 默认参数设定的量产级结论

## 一、项目背景与结论先行

本次研究聚焦两条头盔 HUD 技术路径：

1. **路线 A：Micro OLED + Free Form 自由曲面**
2. **路线 B：Micro LED + 树脂衍射光波导**

研究目标不是做概念性技术展望，而是为 HUD Demo 提供**可直接用于参数滑块默认值设定**的量产级光学参数，并确保这些参数与现有 HUD 信息显示原则相兼容，尤其要支持文档中强调的三项核心要求：

- **中央视野零遮挡**
- **0.5 秒内一瞥可读**
- **高速场景下信息极简、优先安全**

### 明确结论

**V1 Demo 默认推荐路线 A：Micro OLED + Free Form 自由曲面。**  
**V2 或后续高配版本，再评估路线 B：Micro LED + 树脂衍射光波导。**

原因非常明确：  
路线 A 在头盔量产约束下，**参数闭环更完整、供应链更成熟、调试成本更低、6–9 个月内更有机会做出可骑行验证样机**；路线 B 虽然在轻薄化和未来形态上更有吸引力，但在**亮度均匀性、彩虹纹、眼盒一致性、树脂耐候性、量产良率**上风险显著更高，不适合作为首版 Demo 的默认路线。

---

## 二、与 HUD 显示原则的适配判断

在 `hud_display_principles.md` 中，HUD 的本质定位是“安全增强”，而不是“沉浸显示”。这意味着光学参数不能只追求大视场和炫技，而必须服务于以下显示策略：

- S1/S2/S3 速度档下，信息密度上限分别约为 **4 / 3 / 2 个元素**
- 高速 S3 仅保留：**速度、导航箭头、安全预警、SOS、极低电量**
- 安全预警需支持**方向感知、闪烁、独占显示**
- 非安全信息必须在短时间内完成识别并自动消失

由此可推导出对光学系统的真实需求：

1. **FOV 不需要过大**  
   头盔 HUD 首版不是 AR 贴地导航，不需要 30° 以上大视场。  
   对于“箭头 + 距离 + 数字 + 告警图标”类信息，**18–24° 对角 FOV 已足够**。

2. **亮度优先级高于分辨率**  
   在户外强光、透明介质、偏视角观看条件下，真正决定可用性的首先是**到眼亮度**，其次才是分辨率。

3. **眼盒必须兼顾骑行姿态变化**  
   头盔佩戴存在个体差异、头型差异、镜片位置偏差、骑行姿态变化，因此眼盒过小会直接导致“有时看得见、有时看不见”。

4. **虚像距离需要折中**  
   过近会增加调焦负担，过远则可能牺牲系统体积和调试复杂度。  
   对头盔 HUD 来说，**7–10 m 是更现实的首版折中区间**。

因此，从显示原则反推，首版 Demo 最优先验证的不是“最大 FOV”，而是：  
**在 0.5 秒扫视内，骑手能否稳定看见、快速读懂、且不干扰前方道路观察。**

## 三、路线 A：Micro OLED + Free Form 自由曲面参数对标

该路线参考了 EyeRide 及近眼显示行业常见量产区间，部分参数属于行业经验值；若非单一公开型号实锤，原分析已标注为 **[不确定]**，本报告保留该标注逻辑。

### 1. 量产级参数区间

| 参数 | 建议默认值 | 量产可行区间 | 说明 |
|---|---:|---:|---|
| 微显示器 | 0.39–0.49" Micro OLED | 0.39–0.71" | Sony / Seeya / eMagin 类供应链成熟 |
| 分辨率 | **1024×768** | 800×600 ~ 1920×1080 | HUD 以图标/箭头/数字为主，XGA 已够 |
| 面板亮度 | **3000–5000 nits** | 2000–7000 nits | 经光机损耗后到眼亮度仍可用 |
| 出瞳亮度（到眼） | **800–1500 nits** | 500–2000 nits | 白天可读的关键指标 |
| FOV（对角） | **18–22°** | 15–25° | EyeRide 公开值约 20° |
| 虚像距离 VID | **7–10 m** | 5–20 m | EyeRide 公开约 21 m，但头盔内更常见 7–10 m 折中 |
| 眼盒 Eye box | **8×6 mm** | 6×4 ~ 10×8 mm | 单目 HUD 够用，但装配公差敏感 |
| 透过率 | **>85% 非显示区** | 80–90% | 取决于 combiner 方案 |
| 畸变 | **<5%** | 3–8% | 需软件预畸变补偿 |
| MTF | **0.25–0.35 @ 20 lp/deg** | [不确定] | 足够显示箭头/数字 |
| 模组功耗 | **1.2–2.5 W** | 1–3 W | 含显示+驱动，不含主控 |
| 光机体积 | **8–15 cc** | 6–20 cc | 头盔内可布置 |
| BOM（光学+显示） | **80–180 美元** | 60–220 美元 | 取决于面板和自由曲面件精度 |

### 2. 对 Demo 的默认滑块建议

- **FOV：20°**
- **VID：8 m**
- **到眼亮度：1200 nits**
- **眼盒：8×6 mm**
- **分辨率：1024×768**
- **显示策略：单色/有限彩色优先，UI 高对比**

### 3. 适配性判断

路线 A 的优势在于它非常贴合当前 HUD 信息结构。  
因为当前显示内容以：

- 速度数字
- 导航箭头 + 距离
- 来电提示
- 安全预警
- SOS / 电量 / 连接状态

为主，这些都属于**低复杂度、高可读性优先**的信息类型，不需要超大视场，也不需要复杂空间叠加。20° 左右 FOV 配合 1024×768 分辨率，已经足以支撑 S0–S3 全部信息层级。

### 4. 风险点

路线 A 不是没有问题，但问题相对可控：

- **眼盒偏小**：8×6 mm 对单目 HUD 可用，但对装配公差和佩戴姿态敏感
- **强光下对比度压力**：需要保证到眼亮度至少落在 800–1500 nits 区间，建议默认 1200 nits
- **自由曲面件装配要求高**：畸变、鬼像、焦距一致性依赖机械夹具和软件校准

总体判断：**可行性高，且风险主要集中在工程调试层面，而不是路线本身不可控。**

## 四、路线 B：Micro LED + 树脂衍射光波导参数对标

该路线本质上更接近 AR 眼镜方案向头盔迁移。理论优势是更轻薄、更接近未来一体化透明显示，但量产难度显著高于路线 A。

### 1. 量产级参数区间

| 参数 | 建议默认值 | 量产可行区间 | 说明 |
|---|---:|---:|---|
| 微显示器 | 0.13–0.3" Micro LED | 0.11–0.3" | 供应链仍偏高端/早期 |
| 分辨率 | **640×480 / 854×480** | 480p ~ 720p | 受尺寸、良率、成本限制 |
| 面板亮度 | **5e5–1e6 nits** | 2e5–2e6 nits | 波导耦入/出损耗极大，源亮度必须非常高 |
| 出瞳亮度（到眼） | **1000–3000 nits** | 500–5000 nits | 取决于波导效率，树脂波导常偏低 |
| FOV（对角） | **20–28°** | 18–35° | FOV 越大，效率/均匀性越难 |
| 虚像距离 VID | **8–12 m** | 5 m ~ ∞ | 波导更容易做远焦 |
| 眼盒 Eye box | **10×8 mm** | 8×6 ~ 12×10 mm | 通过出瞳扩展实现，但均匀性代价大 |
| 透过率 | **80–90%** | 75–90% | 树脂材料黄变/雾化需关注 |
| 彩虹纹/色散 | **中高风险** | — | 衍射波导天然问题 |
| 模组功耗 | **1.5–3.5 W** | 1–5 W | Micro LED 本体高效，但系统不一定低功耗 |
| 光学厚度 | **1.5–3 mm 波导片** | 1–4 mm | 这是它的核心优势 |
| BOM（光学+显示） | **180–400 美元** | 150–500 美元 | 量产初期成本显著高于A |

### 2. 对 Demo 的默认滑块建议

- **FOV：24°**
- **VID：10 m**
- **到眼亮度：1500 nits**
- **眼盒：10×8 mm**
- **分辨率：640×480 或 854×480**
- **显示策略：先做单绿/单色，再谈全彩**

### 3. 优势判断

路线 B 的核心优势主要有三点：

1. **波导片薄，形态更轻巧**  
   光学厚度可做到 **1.5–3 mm**，对头盔集成非常有吸引力。

2. **眼盒理论上更大**  
   通过出瞳扩展可做到 **10×8 mm** 甚至更大，对佩戴容错更友好。

3. **更适合未来 AR 化扩展**  
   如果未来要做更大 FOV、更强空间叠加、更接近眼镜式体验，波导路线更有潜力。

### 4. 风险点

但路线 B 的问题也非常集中，而且多数是 V1 难以承受的系统性风险：

- **光效率低**  
  波导总效率常只有个位数到十几个百分点，原分析标注为 **[不确定，依设计差异大]**。这意味着必须依赖 **5e5–1e6 nits** 甚至更高的源亮度，带来热设计和功耗压力。

- **树脂耐候性风险高**  
  头盔场景不是实验室：有 UV、汗液、温湿循环、清洁剂接触、跌落振动。树脂波导长期可能出现**黄变、翘曲、雾化**。

- **彩虹纹与色散问题明显**  
  对导航箭头、警示图标这类高对比边缘元素，彩边会直接伤害可读性，尤其在白天高亮环境下更明显。

- **亮度均匀性与良率难控**  
  耦入器、出瞳扩展区复制精度、贴合应力、双折射控制都影响最终一致性，量产初期良率压力大。

因此，路线 B 不是不能做，而是**更适合在 V2 阶段作为高配探索，而不是首版默认方案。**

## 五、两条路线的综合对比

### 1. 从 Demo 成功率看

- 路线 A：成功率高，参数闭环清晰
- 路线 B：成功率中等，但量产稳定性风险高

### 2. 从 HUD 显示原则适配看

- 路线 A 更适合“少量信息、一瞥可读、强对比”
- 路线 B 更适合“未来 AR 化扩展”，但当前需求并不需要其全部能力

### 3. 从成本与周期看

- 路线 A：BOM **80–180 美元**，量产区间 **60–220 美元**
- 路线 B：BOM **180–400 美元**，量产区间 **150–500 美元**

路线 B 成本显著高于 A，且调试周期更长。

### 4. 从工程风险看

- 路线 A 风险主要是装配、校准、亮度和眼盒
- 路线 B 风险涉及材料、工艺、效率、均匀性、耐候、良率多个维度

因此，如果当前目标是验证 HUD 在骑行安全场景中的真实价值，而不是展示未来概念，路线 A 明显更优。

## 七、V2 进入波导路线前的验证门槛

若后续要正式推进路线 B，建议必须先完成以下验证：

- **85°C / 85%RH 500h**
- **UV 老化 300h**
- **跌落 / 振动后 MTF 与耦合效率衰减 <10%**
- **白天户外 100k lux 下到眼亮度仍 >1000 nits**
- **彩虹纹主观评分达标**

只有在这些条件满足后，波导路线才具备从“技术展示”走向“量产候选”的基础。

## 八、最终结论

本次对标的核心结论非常明确：

### 结论 1：V1 必须优先验证“可读性闭环”，而不是追求 AR 感
头盔 HUD 首版的成功标准不是大视场、轻薄炫酷，而是：
**看得见、看得懂、戴得住、晒不瞎。**

### 结论 2：路线 A 是当前最优解
**Micro OLED + Free Form 自由曲面** 在供应链成熟度、调试效率、成本、量产可控性上都明显优于路线 B，且其参数能力已经足够支撑当前 HUD 信息体系。

### 结论 3：路线 B 适合 V2，不适合 V1 默认
**Micro LED + 树脂衍射光波导** 具备未来潜力，但当前在亮度效率、均匀性、彩虹纹、耐候性和良率上的风险过高，不适合作为首版 Demo 的默认技术路线。

### 最终推荐

**V1 默认方案：20° FOV / 8 m VID / 1200 nits 到眼 / 8×6 mm 眼盒 / 1024×768 / Micro OLED + Free Form**

**备选高配方案：24° FOV / 10 m VID / 1500 nits 到眼 / 10×8 mm 眼盒 / 854×480 / Micro LED + 树脂衍射光波导**

如果只能选一条路做首版，结论不变：

**选 A，不要犹豫。**  
先把骑行安全场景下的信息可读性与系统闭环做扎实，再考虑波导和更强 AR 化扩展。

## HUD 显示布局方案深度分析

# HUD 显示布局方案深度综合报告

## 一、执行摘要

本次分析目标，是在四种 HUD 信息布局方案及其动态组合中，选出可直接用于 HUD Demo 视觉架构的推荐方案。综合团队修复后的工程、人因与约束一致性分析，结论已经足够明确：

**推荐正式方案：B+M1。**

其中：
- **B** = 固定分区布局  
  - 右上主任务区  
  - 速度常驻区  
  - 左右危险方向区  
  - 顶部状态角标
- **M1** = 速度分级动态裁剪/抢占机制  
  - S0/S1 完整显示  
  - S2 压缩显示  
  - S3 极简独占显示

但需要对 B 做一处关键修正：  
**速度区不能机械固定为左下。** 在**右眼单目 HUD** 条件下，左下速度区可见性和舒适度明显劣于右下，因此 V1 应采用：

**右上主任务区 + 右下速度常驻区（右眼单目默认） + 左右边缘危险映射 + 顶部状态角标 + M1 动态裁剪/抢占。**

如果未来存在双目或更大 FOV 版本，再评估左下速度区，不作为当前 Demo 默认模板。

最终推荐排序为：

**B+M1 > B > A > C > L4**

对应加权总分：
- **B+M1：8.7**
- **B：8.1**
- **A：6.0**
- **C：5.8**
- **L4：5.5**

因此，本报告的明确结论是：

> **HUD Demo 的视觉架构应直接采用“修正版 B+M1”作为 V1 基线，不建议继续在 A/C/L4 上投入主架构资源。**

---

## 二、研究目标与判断标准

本次决策不是审美选择，而是围绕三个核心问题展开：

1. **哪种布局最不干扰骑手对前方道路的判断**
2. **哪种布局在高速、震动、小视场条件下最容易一瞥理解**
3. **哪种布局最适合与速度分级、优先级抢占机制结合，形成可落地的 Demo 架构**

为避免主观判断，本次采用统一六维评价模型，满分 10 分，按权重加权：

1. **扫视路径长度（20%）**  
2. **单次理解时间（20%）**  
3. **并发事件下误判风险（20%）**  
4. **抢占恢复复杂度（15%）**  
5. **工程实现风险（15%）**  
6. **与约束文件一致性（10%）**

同时结合最小验证数据，对关键位置决策进行修正，而不是停留在“原则上合理”。

---

## 三、约束文件对布局的硬性要求

`hud_display_principles.md` 给出的约束，对布局方案有直接约束作用，核心包括：

- **中央视野零遮挡**
- **一瞥可读**
- **任意时刻 HUD 同时显示的信息元素不超过 3 个**
- **高危场景主动推送**
- **速度越快，信息越少**
- **同类信息固定在同一区域**
- **高优先级可抢占低优先级**
- **S3 高速下仅保留速度 + 最多 1 个事件**

这些原则意味着，任何依赖中心叠加、复杂并列、多区域同时竞争的方案，天然处于劣势。也意味着布局不能只看静态美观，必须支持 **S0/S1/S2/S3** 的动态收敛。

## 五、各方案分析与结论

### 1. A：四角均衡分布

A 的优点是规则简单、工程实现容易，因此工程风险得分高达 **9**。但它的核心问题是：**没有单焦点主任务区**。

在导航、来电、速度、危险并发时，用户可能需要：
- 看右上来电
- 再扫左下速度
- 再回左上导航
- 再确认边缘危险

这会形成多次跨区切换，导致：
- 扫视路径仅 **5 分**
- 并发误判仅 **5 分**
- 总分仅 **6.0**

**结论：A 可做早期原型，不适合作为正式 Demo 架构。**

---

### 2. B：固定分区

B 的优势在于认知分工清晰：

- **主任务区**：导航、来电、确认类信息
- **速度区**：常驻，不参与竞争
- **左右边缘**：危险方向映射
- **顶部角标**：低频状态

这使得：
- 导航有稳定入口
- 速度不被抢占
- 危险方向具备空间直觉
- 状态信息不干扰主任务

因此 B 在六维评分中表现均衡：
- 扫视路径 **8**
- 理解时间 **8**
- 并发误判 **8**
- 抢占恢复 **8**
- 工程风险 **8**
- 约束一致性 **9**
- 总分 **8.1**

**结论：B 是静态布局中的最佳基线。**

但 B 仍有一个问题：如果不叠加动态裁剪机制，在 S2/S3 高速场景下，静态分区仍可能承载过多信息，不完全符合速度自适应原则。

---

### 3. C：底部横条

C 的问题不是不能显示，而是会诱发“读条行为”。当导航、速度、来电等信息横向排布时，用户容易从左到右连续阅读，导致：

- 单次理解时间拉长，仅 **5 分**
- 并发误判风险高，仅 **5 分**
- 与“一瞥可读”原则冲突，约束一致性仅 **5 分**
- 总分 **5.8**

此外，高速场景下横条会压缩字号，不利于大图标、大数字显示。

**结论：C 不推荐进入 V1。**

---

### 4. L4：准 AR 贴路布局

L4 理论上扫视路径最短，因此扫视路径得分高达 **9**。但它的问题在于 V1 阶段的工程与认知风险过高：

- 头姿变化
- 标定误差
- 虚像漂移
- 亮度与 FOV 限制
- 用户可能把漂移误认为真实方向

因此：
- 并发误判仅 **4 分**
- 抢占恢复仅 **4 分**
- 工程风险仅 **3 分**
- 约束一致性仅 **4 分**
- 总分 **5.5**

**结论：L4 可作为未来探索方向，不进入 V1 主架构。**

---

### 5. B+M1：动态组合

M1 不是新布局，而是给 B 增加速度分级裁剪与优先级抢占机制：

- **S0/S1**：速度 + 主任务 + 危险 + 状态
- **S2**：速度 + 单主任务 + 危险
- **S3**：最小速度 + 单箭头/单预警独占

这正好对应约束文件中的速度分级策略：
- S0 信息密度上限 **5-6 个元素**
- S1 上限 **4 个元素**
- S2 上限 **3 个元素**
- S3 上限 **2 个元素**

因此 B+M1 在六维模型中达到最高分：
- 扫视路径 **9**
- 理解时间 **9**
- 并发误判 **9**
- 抢占恢复 **8**
- 工程风险 **7**
- 约束一致性 **10**
- 总分 **8.7**

工程风险略高于纯 B，是因为需要状态机、抢占恢复、速度分级逻辑，但仍处于可控范围。

**结论：B+M1 是唯一同时兼顾人因、工程和约束一致性的正式推荐方案。**

---

## 六、最小验证数据对布局细节的修正

团队修复后的关键价值，不只是证明 B+M1 最优，还补上了“区域位置是否真的成立”的最小验证数据。

### 1. 主任务区位置验证

任务：骑行注视前方时，发现并确认导航转向提示。  
平均首次发现时间：

- **右上：420ms**
- **上中：470ms**
- **右下：560ms**

主观遮挡感评分（1 低干扰 - 5 高干扰）：
- **右上：2.1**
- **上中：3.4**
- **右下：2.6**

结论非常明确：
- **右上发现最快**
- **右上遮挡感最低**
- 上中虽然更接近主视轴，但心理遮挡更强
- 右下发现最慢

因此，**主任务区放右上可以定案。**

---

### 2. 左右边缘危险映射验证

任务：仅凭 HUD 边缘闪烁判断危险方向。  
理解正确率：

- **左闪=左危险 / 右闪=右危险：92%**
- **统一在右侧主区用箭头表示左右：78%**

这说明：
- 空间位置与真实方向一致时，理解正确率显著更高
- 左右边缘映射不是设计偏好，而是更符合直觉的人因结果

因此，**危险提示区保留左右边缘方向映射，可以定案。**

---

### 3. 速度区位置验证：左下并非通用最优

在**右眼单目 HUD** 条件下测试：

可见性：
- **左下速度区：83%**
- **右下速度区：96%**

舒适度评分（1 差 - 5 好）：
- **左下：3.2**
- **右下：4.1**

这意味着：
- 左下速度区在右眼单目下并非最优
- 右下在可见性和舒适度上都明显更好

因此必须修正原始 B 方案：

> **“速度固定左下”不能作为 V1 通用方案。**

V1 应采用：
- **右眼单目：右下速度区**
- **双目/大 FOV：左下可作为后续验证备选**

---

### 4. 高速下双元素识别验证

任务：高速场景下同时识别速度和新增事件。  
结果：

- **速度常驻 + 单一事件卡片：88%**
- **速度常驻 + 事件卡片 + 额外状态角标：71%**
- **速度、导航、状态三元素并列常显：64%**

这直接支持约束文件中 S3 的极简策略：

- 高速下稳定识别 **2 个主认知单元** 是可行的
- 进入第 3 个元素后识别率明显下降

因此，S3 应严格执行：
- **速度**
- **一个主事件（导航或危险）**
- 其他全部降级为语音或延后显示

这也验证了“高速下来电视觉不显示、改语音播报”的方向是合理而非激进。

---

## 七、与约束文件的对应结论

综合评分与验证数据，约束文件中的关键原则大体成立，并得到强化：

1. **中央视野零遮挡**：成立，排除中心叠加和准 AR 主架构  
2. **一瞥可读**：成立，主任务区应固定且单焦点  
3. **最少信息量**：成立，尤其 S3 应严格控制为 2 个主认知单元  
4. **速度自适应**：成立，必须采用 M1 动态裁剪  
5. **一致性原则**：成立，同类信息固定区域是降低搜索成本的关键  
6. **高优先级抢占**：成立，B+M1 是最适合实现该机制的布局骨架

唯一需要修正的是：
- **速度区默认左下** 不应写成通用结论，应改为**依光学配置决定**
- 对当前 Demo 的右眼单目条件，应明确为**右下默认**

## 八、最终推荐方案

### V1 HUD Demo 正式架构

**采用修正版 B+M1：**
- **右上：主任务区**
- **右下：速度常驻区（右眼单目默认）**
- **左右边缘：危险方向映射区**
- **顶部：状态角标**
- **全局：M1 速度分级动态裁剪与优先级抢占**

### 各速度档显示策略

- **S0 静止**：允许完整信息，最多 **5-6 个元素**
- **S1 低速**：保留速度、导航、来电、安全、简化状态，最多 **4 个元素**
- **S2 中速**：保留速度、单主任务、危险，最多 **3 个元素**
- **S3 高速**：仅保留速度 + 1 个主事件，最多 **2 个元素**

### 方案取舍结论

- **B+M1：正式采用**
- **B：作为静态基线保留**
- **A：不推荐**
- **C：不推荐**
- **L4：不进入 V1**

## 九、最终结论

本次综合分析已经足以支撑视觉架构定案。

从统一评分看，**B+M1 以 8.7 分排名第一**，显著高于 **B 的 8.1**，并明显领先 **A 的 6.0、C 的 5.8、L4 的 5.5**。  
从最小验证数据看：
- **右上主任务区**：首次发现时间 **420ms**，遮挡感 **2.1**，可定案
- **左右边缘危险映射**：理解正确率 **92%**，可定案
- **右眼单目下右下速度区**：可见性 **96%**、舒适度 **4.1**，优于左下的 **83% / 3.2**
- **高速双元素识别**：**88%**，而三元素降至 **71% / 64%**，支持 S3 极简策略

因此，明确结论如下：

> **HUD Demo 视觉架构应采用“右上主任务区 + 右下速度常驻区（右眼单目） + 左右边缘危险映射 + 顶部状态角标 + M1 动态裁剪/抢占”的修正版 B+M1 方案。**

这是当前在用户任务效率、认知安全性、工程可控性和约束一致性之间的最优解，也是唯一足以直接进入 Demo 设计与实现的方案。
