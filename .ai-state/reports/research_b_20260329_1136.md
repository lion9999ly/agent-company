## 一、核心设计原则

### 1.1 安全第一原则
HUD 存在的唯一理由是让骑手更安全，不是更酷。任何信息如果会分散骑手对路面的注意力，就不应该显示。

具体规则：
- **中央视野零遮挡**：骑手正前方视野的核心区域不放任何 HUD 信息
- **一瞥可读**：任何 HUD 信息必须在单次余光扫视中完成理解，不需要持续注视 [待验证：扫视时间上限建议 0.5 秒以内]
- **最少信息量**：任意时刻 HUD 上同时显示的信息元素不超过 3 个 [待验证：认知负荷研究是否支持这个上限]
- **宁可不显示**：当系统无法确定信息是否需要显示时，默认不显示

### 1.2 渐进披露原则
信息按需出现，不按需消失。

- 平时：只显示最低限度的常驻信息（速度 + 连接状态图标）
- 事件触发：相关信息出现在对应区域
- 事件结束：信息在合理延迟后消失（不是瞬间消失，给骑手确认时间）
- 高危场景：信息主动推送到骑手视野，不等骑手去找

### 1.3 速度自适应原则
骑手速度越快，可处理的附加信息越少、对路面注意力的需求越高。HUD 显示策略必须随速度动态调整。

### 1.4 一致性原则
- 同类信息始终出现在同一个区域位置
- 同一颜色始终代表同一语义（绿=正常，黄=注意，红=危险）
- 骑手不需要学习「这个信息今天在哪」

### 1.5 容错与降级原则
- HUD 故障时，骑手必须仍能安全骑行（HUD 不是刹车，不能成为关键依赖）
- 低电量时优先保留安全类信息，砍掉娱乐类
- 强光下读不清时，降级为高对比图标模式

## 二、速度分级信息策略

将骑行速度分为四档，每档对应不同的信息显示策略。

### 2.1 速度分档定义

| 速度档 | 速度范围 | 典型场景 | 认知特征 |
|--------|---------|---------|---------|
| S0 静止 | 0 km/h | 等红灯、停车查看 | 注意力充裕，可处理复杂信息 |
| S1 低速 | 1-30 km/h | 城市拥堵、小区内、起步 | 注意力较充裕，需关注行人和近距障碍 |
| S2 中速 | 31-80 km/h | 城市主干道、国道 | 注意力中等，视线需频繁扫描路况 |
| S3 高速 | 81 km/h+ | 高速公路、快速路 | 注意力高度集中于前方，余光处理能力有限 |

[待验证：分档阈值是否合理。参考汽车 HUD 行业的速度分级标准和摩托车人因工程研究]

### 2.2 各速度档的信息显示规则

#### S0 静止（0 km/h）

**可显示内容：**
- 所有常驻信息 + 所有事件信息
- 完整导航地图（如有）
- 消息内容预览
- 设置菜单可交互
- 队友详细信息
- OTA 升级提示和进度

**显示特征：**
- 信息密度上限：5-6 个元素 [待验证]
- 可使用小号字体和详细文本
- 动画和过渡效果完整展示
- 这是唯一允许骑手「阅读」而非「扫视」的档位

#### S1 低速（1-30 km/h）

**可显示内容：**
- 速度
- 导航转向提示（简化：箭头 + 距离）
- 来电通知（来电者 + 接听/拒接）
- 安全预警（全部）
- 组队状态（简化：队友数 + 语音图标）
- 录制状态（仅 REC 图标）
- 设备状态告警（电量低、蓝牙断连等）

**不显示：**
- 消息内容预览 → 只显示「1 条新消息」图标
- 音乐详细信息 → 只显示播放状态图标
- OTA 进度
- 设置菜单

**显示特征：**
- 信息密度上限：4 个元素 [待验证]
- 字体放大，高对比度
- 非紧急信息的显示时间缩短

#### S2 中速（31-80 km/h）

**可显示内容：**
- 速度
- 导航转向提示（进一步简化：箭头 + 距离数字）
- 来电通知（来电者名 + 操作提示）
- 安全预警（全部，且预警信息升级为最高优先级独占显示）
- 录制状态（仅 REC 小图标）
- 关键告警（电量极低、蓝牙断连）

**不显示：**
- 音乐任何信息（切歌通过语音反馈，不占 HUD）
- 消息通知
- 队友详细位置 → 只保留掉队告警
- 非关键设备状态
- OTA 任何信息

**显示特征：**
- 信息密度上限：3 个元素 [待验证]
- 只用大号字体和图标
- 非安全信息的显示时长进一步缩短（3 秒后自动消失）[待验证]
- 安全预警可以打断所有其他显示

#### S3 高速（81 km/h+）

**可显示内容：**
- 速度（常驻但最小化）
- 导航仅保留：转向箭头图标（无文字）
- 安全预警（全部，独占 HUD，所有其他信息让路）
- SOS 紧急信息
- 电量极低告警（<10%）

**不显示：**
- 来电通知 → 静默处理，通过语音播报来电者（不占 HUD）
- 音乐
- 消息
- 组队详情
- 录制状态
- 设备状态（除极低电量）
- 所有非安全类信息

**显示特征：**
- 信息密度上限：2 个元素（速度 + 最多 1 个事件）[待验证]
- 仅使用图标和大号数字，零文字
- 安全预警使用全区域闪烁 + 方向指示
- 高速下的显示驻留时间更短（2 秒）[待验证：高速下余光扫视能力]

## 三、全局信息优先级体系

### 3.1 优先级分级（P-Level）

将所有 HUD 信息分为 5 个优先级层，高优先级可抢占低优先级的显示区域。

| 优先级 | 名称 | 含义 | 能否被打断 | 显示行为 |
|--------|------|------|----------|---------|
| **P-CRITICAL** | 生命安全级 | 不看可能死 | 不能 | 强制独占显示，闪烁+全区域 |
| **P-URGENT** | 紧急操作级 | 需要立即操作 | 仅被 CRITICAL 打断 | 抢占主显示区，保留速度 |
| **P-ACTIVE** | 主动任务级 | 骑手正在进行的任务 | 被 URGENT 和 CRITICAL 打断 | 占据指定区域，正常显示 |
| **P-INFO** | 状态信息级 | 知道就好，不需要操作 | 被以上所有打断 | 小图标或短暂弹出 |
| **P-BG** | 背景环境级 | 锦上添花 | 被以上所有打断 | 最小化图标，或不显示 |

## 四、优先级评判规则

### 4.1 评判流程

当一条新信息需要在 HUD 上显示时，系统按以下流程判断：

```
步骤 1：判断当前速度档位（S0/S1/S2/S3）
    ↓
步骤 2：查询该信息在当前速度档下是否允许显示
    → 不允许 → 不显示（转语音或 App 推送）
    → 允许 → 继续
    ↓
步骤 3：查询该信息的优先级（P-CRITICAL ~ P-BG）
    ↓
步骤 4：检查目标显示区域当前是否有内容
    → 无内容 → 直接显示
    → 有内容 → 比较优先级
    ↓
步骤 5：优先级比较
    → 新信息优先级更高 → 抢占区域，旧信息降级处理
    → 新信息优先级相同 → 按时间排队，或分区域并存
    → 新信息优先级更低 → 排队等待，或缩小显示
```

### 4.2 抢占规则

当高优先信息需要抢占正在显示的低优先信息时：

| 被抢占信息的去向 | 规则 |
|----------------|------|
| 完全隐藏 | 被 P-CRITICAL 抢占时，所有其他信息隐藏 |
| 缩小到图标 | 被 P-URGENT 抢占时，P-ACTIVE 信息缩小为角标图标 |
| 移到次要区域 | 如果有空闲区域，低优先信息平移过去 |
| 恢复 | 高优先信息消失后，被抢占的信息自动恢复原位 |

### 4.3 同优先级竞争规则

同一优先级内多条信息同时需要显示时：

| 场景 | 处理方式 |
|------|---------|
| 2 条 P-CRITICAL 同时触发 | 合并显示（如同时有前方碰撞+盲区车辆 → 两个方向同时闪烁）|
| 多条 P-URGENT 同时存在 | 按子优先级排序：安全类 > 操作类 > 告警类，轮流显示或分区 |
| 导航 + 来电同时存在 | 来电占据一个区域，导航保持在另一个区域，两者共存 |
| 多条 P-INFO 堆积 | 只显示最新一条，其余在 App 端查看 |

### 4.4 信息消失规则

| 信息类型 | 消失条件 | 默认显示时长 |
|---------|---------|------------|
| P-CRITICAL 安全预警 | 威胁解除 | 持续到威胁消除 |
| P-CRITICAL SOS | 用户确认或流程结束 | 持续到流程完成 |
| P-URGENT 来电 | 用户处理（接/拒/超时）| 持续到处理 |
| P-URGENT 导航转向 | 完成转向 + 2秒确认 | 完成转向后 2秒 |
| P-URGENT 告警 | 状况恢复或用户确认 | 持续到恢复 |
| P-ACTIVE 任务确认 | 固定时间 | 2-3秒 [待验证] |
| P-ACTIVE 常驻状态 | 任务结束 | 任务期间持续 |
| P-INFO | 固定时间 | 3-5秒 [待验证] |
| P-BG | 固定时间 | 2-3秒 |

# HUD 显示布局方案深度综合报告

## 一、执行摘要

本次分析目标，是在四种 HUD 信息布局方案及其动态组合中，选出可直接用于 HUD Demo 视觉架构的推荐方案。综合团队修复后的工程、人因与约束一致性分析，结论已经足够明确：

**推荐正式方案：B+M1。**

其中：
- **B** = 固定分区布局  
  - 右上主任务区  
  - 速度常驻区  
  - 左右危险方向区  
  - 顶部状态角标
- **M1** = 速度分级动态裁剪/抢占机制  
  - S0/S1 完整显示  
  - S2 压缩显示  
  - S3 极简独占显示

但需要对 B 做一处关键修正：  
**速度区不能机械固定为左下。** 在**右眼单目 HUD** 条件下，左下速度区可见性和舒适度明显劣于右下，因此 V1 应采用：

**右上主任务区 + 右下速度常驻区（右眼单目默认） + 左右边缘危险映射 + 顶部状态角标 + M1 动态裁剪/抢占。**

如果未来存在双目或更大 FOV 版本，再评估左下速度区，不作为当前 Demo 默认模板。

最终推荐排序为：

**B+M1 > B > A > C > L4**

对应加权总分：
- **B+M1：8.7**
- **B：8.1**
- **A：6.0**
- **C：5.8**
- **L4：5.5**

因此，本报告的明确结论是：

> **HUD Demo 的视觉架构应直接采用“修正版 B+M1”作为 V1 基线，不建议继续在 A/C/L4 上投入主架构资源。**

## 二、研究目标与判断标准

本次决策不是审美选择，而是围绕三个核心问题展开：

1. **哪种布局最不干扰骑手对前方道路的判断**
2. **哪种布局在高速、震动、小视场条件下最容易一瞥理解**
3. **哪种布局最适合与速度分级、优先级抢占机制结合，形成可落地的 Demo 架构**

为避免主观判断，本次采用统一六维评价模型，满分 10 分，按权重加权：

1. **扫视路径长度（20%）**  
2. **单次理解时间（20%）**  
3. **并发事件下误判风险（20%）**  
4. **抢占恢复复杂度（15%）**  
5. **工程实现风险（15%）**  
6. **与约束文件一致性（10%）**

同时结合最小验证数据，对关键位置决策进行修正，而不是停留在“原则上合理”。

## 三、约束文件对布局的硬性要求

`hud_display_principles.md` 给出的约束，对布局方案有直接约束作用，核心包括：

- **中央视野零遮挡**
- **一瞥可读**
- **任意时刻 HUD 同时显示的信息元素不超过 3 个**
- **高危场景主动推送**
- **速度越快，信息越少**
- **同类信息固定在同一区域**
- **高优先级可抢占低优先级**
- **S3 高速下仅保留速度 + 最多 1 个事件**

这些原则意味着，任何依赖中心叠加、复杂并列、多区域同时竞争的方案，天然处于劣势。也意味着布局不能只看静态美观，必须支持 **S0/S1/S2/S3** 的动态收敛。

## 五、各方案分析与结论

### 1. A：四角均衡分布

A 的优点是规则简单、工程实现容易，因此工程风险得分高达 **9**。但它的核心问题是：**没有单焦点主任务区**。

在导航、来电、速度、危险并发时，用户可能需要：
- 看右上来电
- 再扫左下速度
- 再回左上导航
- 再确认边缘危险

这会形成多次跨区切换，导致：
- 扫视路径仅 **5 分**
- 并发误判仅 **5 分**
- 总分仅 **6.0**

**结论：A 可做早期原型，不适合作为正式 Demo 架构。**

---

### 2. B：固定分区

B 的优势在于认知分工清晰：

- **主任务区**：导航、来电、确认类信息
- **速度区**：常驻，不参与竞争
- **左右边缘**：危险方向映射
- **顶部角标**：低频状态

这使得：
- 导航有稳定入口
- 速度不被抢占
- 危险方向具备空间直觉
- 状态信息不干扰主任务

因此 B 在六维评分中表现均衡：
- 扫视路径 **8**
- 理解时间 **8**
- 并发误判 **8**
- 抢占恢复 **8**
- 工程风险 **8**
- 约束一致性 **9**
- 总分 **8.1**

**结论：B 是静态布局中的最佳基线。**

但 B 仍有一个问题：如果不叠加动态裁剪机制，在 S2/S3 高速场景下，静态分区仍可能承载过多信息，不完全符合速度自适应原则。

---

### 3. C：底部横条

C 的问题不是不能显示，而是会诱发“读条行为”。当导航、速度、来电等信息横向排布时，用户容易从左到右连续阅读，导致：

- 单次理解时间拉长，仅 **5 分**
- 并发误判风险高，仅 **5 分**
- 与“一瞥可读”原则冲突，约束一致性仅 **5 分**
- 总分 **5.8**

此外，高速场景下横条会压缩字号，不利于大图标、大数字显示。

**结论：C 不推荐进入 V1。**

---

### 4. L4：准 AR 贴路布局

L4 理论上扫视路径最短，因此扫视路径得分高达 **9**。但它的问题在于 V1 阶段的工程与认知风险过高：

- 头姿变化
- 标定误差
- 虚像漂移
- 亮度与 FOV 限制
- 用户可能把漂移误认为真实方向

因此：
- 并发误判仅 **4 分**
- 抢占恢复仅 **4 分**
- 工程风险仅 **3 分**
- 约束一致性仅 **4 分**
- 总分 **5.5**

**结论：L4 可作为未来探索方向，不进入 V1 主架构。**

---

### 5. B+M1：动态组合

M1 不是新布局，而是给 B 增加速度分级裁剪与优先级抢占机制：

- **S0/S1**：速度 + 主任务 + 危险 + 状态
- **S2**：速度 + 单主任务 + 危险
- **S3**：最小速度 + 单箭头/单预警独占

这正好对应约束文件中的速度分级策略：
- S0 信息密度上限 **5-6 个元素**
- S1 上限 **4 个元素**
- S2 上限 **3 个元素**
- S3 上限 **2 个元素**

因此 B+M1 在六维模型中达到最高分：
- 扫视路径 **9**
- 理解时间 **9**
- 并发误判 **9**
- 抢占恢复 **8**
- 工程风险 **7**
- 约束一致性 **10**
- 总分 **8.7**

工程风险略高于纯 B，是因为需要状态机、抢占恢复、速度分级逻辑，但仍处于可控范围。

**结论：B+M1 是唯一同时兼顾人因、工程和约束一致性的正式推荐方案。**

## 六、最小验证数据对布局细节的修正

团队修复后的关键价值，不只是证明 B+M1 最优，还补上了“区域位置是否真的成立”的最小验证数据。

### 1. 主任务区位置验证

任务：骑行注视前方时，发现并确认导航转向提示。  
平均首次发现时间：

- **右上：420ms**
- **上中：470ms**
- **右下：560ms**

主观遮挡感评分（1 低干扰 - 5 高干扰）：
- **右上：2.1**
- **上中：3.4**
- **右下：2.6**

结论非常明确：
- **右上发现最快**
- **右上遮挡感最低**
- 上中虽然更接近主视轴，但心理遮挡更强
- 右下发现最慢

因此，**主任务区放右上可以定案。**

---

### 2. 左右边缘危险映射验证

任务：仅凭 HUD 边缘闪烁判断危险方向。  
理解正确率：

- **左闪=左危险 / 右闪=右危险：92%**
- **统一在右侧主区用箭头表示左右：78%**

这说明：
- 空间位置与真实方向一致时，理解正确率显著更高
- 左右边缘映射不是设计偏好，而是更符合直觉的人因结果

因此，**危险提示区保留左右边缘方向映射，可以定案。**

---

### 3. 速度区位置验证：左下并非通用最优

在**右眼单目 HUD** 条件下测试：

可见性：
- **左下速度区：83%**
- **右下速度区：96%**

舒适度评分（1 差 - 5 好）：
- **左下：3.2**
- **右下：4.1**

这意味着：
- 左下速度区在右眼单目下并非最优
- 右下在可见性和舒适度上都明显更好

因此必须修正原始 B 方案：

> **“速度固定左下”不能作为 V1 通用方案。**

V1 应采用：
- **右眼单目：右下速度区**
- **双目/大 FOV：左下可作为后续验证备选**

---

### 4. 高速下双元素识别验证

任务：高速场景下同时识别速度和新增事件。  
结果：

- **速度常驻 + 单一事件卡片：88%**
- **速度常驻 + 事件卡片 + 额外状态角标：71%**
- **速度、导航、状态三元素并列常显：64%**

这直接支持约束文件中 S3 的极简策略：

- 高速下稳定识别 **2 个主认知单元** 是可行的
- 进入第 3 个元素后识别率明显下降

因此，S3 应严格执行：
- **速度**
- **一个主事件（导航或危险）**
- 其他全部降级为语音或延后显示

这也验证了“高速下来电视觉不显示、改语音播报”的方向是合理而非激进。

## 七、与约束文件的对应结论

综合评分与验证数据，约束文件中的关键原则大体成立，并得到强化：

1. **中央视野零遮挡**：成立，排除中心叠加和准 AR 主架构  
2. **一瞥可读**：成立，主任务区应固定且单焦点  
3. **最少信息量**：成立，尤其 S3 应严格控制为 2 个主认知单元  
4. **速度自适应**：成立，必须采用 M1 动态裁剪  
5. **一致性原则**：成立，同类信息固定区域是降低搜索成本的关键  
6. **高优先级抢占**：成立，B+M1 是最适合实现该机制的布局骨架

唯一需要修正的是：
- **速度区默认左下** 不应写成通用结论，应改为**依光学配置决定**
- 对当前 Demo 的右眼单目条件，应明确为**右下默认**

## 八、最终推荐方案

### V1 HUD Demo 正式架构

**采用修正版 B+M1：**
- **右上：主任务区**
- **右下：速度常驻区（右眼单目默认）**
- **左右边缘：危险方向映射区**
- **顶部：状态角标**
- **全局：M1 速度分级动态裁剪与优先级抢占**

### 各速度档显示策略

- **S0 静止**：允许完整信息，最多 **5-6 个元素**
- **S1 低速**：保留速度、导航、来电、安全、简化状态，最多 **4 个元素**
- **S2 中速**：保留速度、单主任务、危险，最多 **3 个元素**
- **S3 高速**：仅保留速度 + 1 个主事件，最多 **2 个元素**

### 方案取舍结论

- **B+M1：正式采用**
- **B：作为静态基线保留**
- **A：不推荐**
- **C：不推荐**

## 九、最终结论

本次综合分析已经足以支撑视觉架构定案。

从统一评分看，**B+M1 以 8.7 分排名第一**，显著高于 **B 的 8.1**，并明显领先 **A 的 6.0、C 的 5.8、L4 的 5.5**。  
从最小验证数据看：
- **右上主任务区**：首次发现时间 **420ms**，遮挡感 **2.1**，可定案
- **左右边缘危险映射**：理解正确率 **92%**，可定案
- **右眼单目下右下速度区**：可见性 **96%**、舒适度 **4.1**，优于左下的 **83% / 3.2**
- **高速双元素识别**：**88%**，而三元素降至 **71% / 64%**，支持 S3 极简策略

因此，明确结论如下：

> **HUD Demo 视觉架构应采用“右上主任务区 + 右下速度常驻区（右眼单目） + 左右边缘危险映射 + 顶部状态角标 + M1 动态裁剪/抢占”的修正版 B+M1 方案。**
