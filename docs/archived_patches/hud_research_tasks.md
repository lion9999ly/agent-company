# 研究 B：HUD 显示布局方案深度分析

> 类型：多 Agent 协作任务（CPO + CTO + CDO + Critic）
> 优先级：P0
> 预计耗时：2-3h
> 输出域：knowledge_base/lessons/hud_layout

## B.0 分析目标

对比四种 HUD 信息布局方案（及其动态组合），输出推荐方案及量化依据。结论将直接决定 HUD Demo 的视觉架构。

---

## B.1 四种候选方案

| 方案 | 描述 | 示意 |
|------|------|------|
| A. 全屏 | 信息铺满整个 HUD 可用 FOV | 整块区域承载所有信息 |
| B. 左右分屏 | 左半区 + 右半区，中间留空 | 类似汽车仪表左右分区 |
| C. 四角分布 | 左上 + 右上 + 左下 + 右下，中央完全留空 | 战斗机 HMD 思路 |
| D. 单角显示 | 仅一个角落放信息（如右下） | EyeRide / Skully 的做法 |
| E. 动态组合 | 默认少量区域，事件触发时扩展 | 如平时单角 → 预警时四角 |

注意：最终可能是 E（动态组合），但需要分析 A-D 各自的优劣后才能确定组合策略。

---

## B.2 分析维度

### 维度 1：骑行安全性

- 各方案对中央视野的遮挡程度（量化：遮挡面积占总 FOV 的百分比）
- 骑手视线从路面到 HUD 信息的转移角度和所需时间
- 夜间 / 隧道 / 逆光场景的眩光干扰程度
- 紧急场景（突发障碍物、急刹）下的信息获取速度
- 引用人因工程 / HMI / 注意力分配相关研究论文或标准
- 参考 ISO 15008（道路车辆 — 驾驶员视觉信息呈现）对 HUD 位置的建议

### 维度 2：信息架构适配性

对照 PRD 中的 7 个 HUD 页面态：
1. 骑行主界面（速度 + 状态图标）
2. 导航态（箭头 + 距离 + 路名 + ETA）
3. 来电态（来电者 + 接听/拒接）
4. 组队态（队友位置 + 语音状态）
5. 预警态（预警图标 + 方向 + 距离）
6. 录制态（时长 + 存储）
7. 设置态（亮度/音量调节）

分析要点：
- 各方案能承载的最大信息密度
- 多信息并发时的排布能力（导航 + 来电 + 预警同时出现怎么办？）
- 信息优先级的视觉表达能力（颜色、位置、大小、动画）
- 预警方向感知：威胁来自左前/右前/左后/右后时，哪种布局能让骑手用余光感知威胁方向？

### 维度 3：光学实现可行性

- Free Form 方案下各布局的光学复杂度和成本（单棱镜能覆盖四角吗？需要几个光学模组？）
- 衍射光波导方案下各布局的可行性（单片波导的有效显示区域形状限制）
- FOV 利用率：有效信息面积 / 总 FOV 面积
- 对 eye box 尺寸的要求（四角布局需要更大 eye box？）
- 四角分布如果需要多个光学模组 → 成本、重量、功耗的影响
- 单模组实现偏心显示的技术可行性

### 维度 4：竞品实际布局做法

**逐竞品分析其显示布局：**

| 竞品 | 需分析的布局问题 |
|------|----------------|
| Shoei GT-Air 3 Smart (EyeLights Gen3) | 投影位置在镜片哪个区域？信息如何分区？全屏/局部？ |
| iC-R Helmets (各型号) | 1080p 屏幕是全屏使用？后视画面和导航信息如何分区？占比多少？ |
| TVS + Aegis Rider | 双目方案左右眼各显示什么？空间锚定让信息"悬浮"在路面上？ |
| EyeLights EyeRide V2/V3 | 右眼小棱镜内的信息排布？CarPlay 映射时的 UI 裁剪策略？ |
| MOTOEYE E6+ | 显示区域大小？CarPlay 映射方式？后视和导航共存时的画面分割？ |
| BMW ConnectedRide Smartglasses | 右镜片投影的具体位置（偏上？偏中？）？信息层级切换交互？ |
| Skully AR-1 | 右下角小屏——为什么选右下？后视画面独占还是与导航共存？ |
| CrossHelmet X1 | 360° 后视 + HUD 的布局方式 |
| Forcite MK1 | LED 矩阵的信息承载方式（只能做简单图标？） |

**跨行业参考：**

| 参考领域 | 需分析的布局问题 |
|---------|----------------|
| 汽车 W-HUD | 挡风玻璃投影区域位置、信息分区（速度/导航/ADAS 告警）|
| 汽车 AR-HUD | 信息叠加在真实道路上的 overlay 策略、近场 vs 远场分区 |
| 军用 HMD — F-35 HMDS | 战斗机如何在全视野 HMD 上做多区域信息分配？优先级切换？ |
| 军用 HMD — Apache IHADSS | 单目 HMD 的信息密度上限和布局 |
| 消费 AR 眼镜 | HoloLens / Magic Leap / Xreal 的默认信息窗口位置和大小 |

### 维度 5：用户认知负荷

- 多区域信息分散 vs 单区域集中的认知成本对比（心理学研究）
- 骑行高速场景（80-120km/h）下的注意力分配特征
- 摩托车骑手 vs 汽车驾驶员的视觉扫视行为差异（头部自由度更大、没有 A 柱遮挡、需要看后视镜等）
- 信息呈现的「渐进披露」策略：平时极少量 → 事件发生时增量显示 → 事件结束后回收
- 多个角落同时有信息时，骑手的注意力竞争问题
- 动画/闪烁作为优先级信号 vs 作为干扰源的平衡点

### 维度 6：动态组合方案的可行性

如果推荐动态组合（方案 E），需详细分析：
- 默认态显示几个区域？哪些区域？
- 各页面态触发时的区域扩展/收缩规则
- 状态切换的过渡方式（渐入渐出？瞬切？）是否增加认知负荷
- 预警态的方向映射逻辑：左前方威胁 → 左上区域闪烁？可行性？
- 7 个页面态各自的最优区域分配建议表
- 多页面态叠加时的优先级栈：预警 > 来电 > 导航 > 组队 > 音乐 > 录制 > 主界面（是否合理？）
- 高优先态抢占区域时，被抢占的低优先信息去哪？隐藏？缩小？移位？
