## 一、错误链复盘

### 1.1 错误传递链

```
错误起点：用户说"Meta Ray-Ban Display"
    ↓
我的错误假设：用户说的是"Meta Ray-Ban（无显示）"
    ↓
错误分析：把它当作"无AR显示的智能眼镜"分析
    ↓
错误评审：问Gemini"这个产品是否应纳入竞品"
    ↓
Gemini回答：验证了我的错误假设
    ↓
结论：整个流程是表演，没有发现根本性错误
```

### 1.2 我犯的三个层级错误

| 层级 | 错误 | 说明 |
|------|------|------|
| **L1: 产品识别** | 认错产品 | 把 "Display版本" 认成 "无显示版本" |
| **L2: 信息验证** | 没有验证 | 没有搜索确认产品规格 |
| **L3: 评审设计** | 评审了错误前提 | Gemini只评审了我给的信息，没有反向验证 |

## 二、根因分析

### 2.1 为什么交叉验证失效？

**设计缺陷**：我只让Gemini评审"逻辑推理"，没有评审"事实前提"。

```python
# 我的评审请求（错误）
{
  "question": "Meta Ray-Ban（无AR显示）是否应该纳入竞品？"
}

# 正确的评审请求应该是
{
  "verify_product": {
    "name": "Meta Ray-Ban Display",
    "verify": "这个产品是否存在？规格是什么？是否有AR显示功能？"
  },
  "verify_logic": "如果它是XX规格，是否应该纳入竞品？"
}
```

### 2.2 为什么我没有先验证产品？

**心理捷径**：我假设用户说的是我知道的产品（Meta Ray-Ban智能眼镜），而不是先搜索确认。

**正确做法**：任何产品名称，都应该先搜索验证，而不是假设。

## 三、正确流程设计

### 3.1 竞品筛选五步法（修正版）

```
Step 1: 产品信息核验（必做）
  - 搜索产品名称，确认是否存在
  - 获取官方规格（价格、重量、功能、发布时间）
  - 标注信息来源和置信度

Step 2: 候选产品扫描
  - 搜索同品类产品
  - 搜索跨品类竞品
  - 用户提及的产品

Step 3: 产品信息核验（对每个候选产品）
  - 确认产品名称正确
  - 获取官方规格
  - 标注置信度

Step 4: 竞品分层
  - 按竞争关系分层
  - 记录分层理由

Step 5: 交叉评审（包含反向验证）
  - 让评审模型独立搜索验证产品信息
  - 评审竞品分层逻辑
  - 发现遗漏和错误
```

### 3.2 交叉评审的正确设计

```json
{
  "review_request": {
    "phase1_fact_check": {
      "task": "独立搜索并验证以下产品信息",
      "products": ["Meta Ray-Ban Display", "INMO Air3"],
      "verify": ["产品是否存在", "官方规格", "价格", "是否有AR显示"]
    },
    "phase2_logic_check": {
      "task": "评审竞品分层是否合理",
      "after_phase1": "只有phase1通过后才进行"
    },
    "phase3_missing_check": {
      "task": "检查是否有遗漏的竞品",
      "method": "独立搜索市场"
    }
  }
}
```

## 五、这次教训

| 教训 | 具体内容 |
|------|----------|
| **不要假设** | 产品名称要逐字确认，"Display"可能意味着有显示功能 |
| **验证先行** | 分析前必须验证产品信息，而不是分析后再验证 |
| **评审要反向** | 评审模型应该独立验证前提，而不是只验证逻辑 |
| **承认无知** | 不知道产品时应该先搜索或问用户，而不是假设 |
