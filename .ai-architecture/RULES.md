# 🏛️ 智能头盔虚拟研发中心：最高系统宪法 (RULES.md)

> **[SYSTEM DIRECTIVE FOR ALL AGENTS & HUMANS]**
> 本文件是整个 Multi-Agent 系统的最高物理法则。所有 Agent 在实例化时必须读取此文件。
> 任何背离本文件的代码提交、状态修改或架构调整，将被底层的 Hook 脚本无情绞杀。

## 1. 唯一真相源与持久化闭环铁律 (SSOT Rule)
* **物理文件具备最高主权**：`.ai-architecture/` 下的静态文档是系统的唯一真相源。LangGraph 的 Checkpointer（状态机快照）仅作为运行时缓存。
* **状态冲突熔断**：若状态机内存中的规则与本地 Markdown 文件发生哈希冲突，系统必须无条件信任本地文件。
* **禁止内存篡改**：Agent 绝对禁止在运行时直接通过 API 修改全局架构逻辑。任何架构变更，必须先由人类更新 Markdown 文档，再重启系统对齐认知。

## 2. API 模型选型与能力错位铁律 (Model Alignment Rule)
鉴于网络物理隔离限制（Claude 仅限前台，不可用于 API），系统强制采用以下异构模型阵列：
* **CPO_Plan (规划大脑)**：强制调用 **Azure GPT-4o API**。利用其极强的结构化输出和零样本拆解能力生成初始契约。
* **CPO_Critic (漏洞对审)**：强制启动**双模型交叉评审**：
  1. **Google Gemini 1.5 Pro API**：负责大上下文比对、边界推演与安全审查。
  2. **阿里云 Qwen-Max API**：作为 DeepSeek 的高可用平替，负责代码逻辑可行性与规范符合性审查。
  * *注：单模型同意无效，必须双模型 `PASS` 且至少循环 3 轮以上，契约方可下发。*

## 3. 渐进式上下文加载边界铁律 (Context Boundary Rule)
为彻底抑制上下文熵增（Token 爆炸），实施严格的切片隔离管控：
* **全局级准入白名单**：仅有 **CPO 规划阶段**、**系统冷启动重置**、**跨模块架构重构** 这三种场景，Agent 才被允许读取 `AgentGlobalState` 的全量数据和所有架构文档。
* **执行级盲盒隔离**：CTO 与 CMO 节点**绝对禁止**读取全局状态。它们只能接收经过序列化的、仅包含自身任务目标的 JSON 切片（`TaskSlice`）。
* **跨节点通信阻断**：Agent 之间绝对禁止“自然语言闲聊”。所有协作必须通过结构化的 `task_contract` 字段反向汇报给中枢。

## 4. 三层分形文档与强制同构铁律 (Fractal Docs Rule)
代码与文档脱节是导致架构腐化的万恶之源。系统实行严格的“三层分形”：
* **第一层（全局）**：`ARCHITECTURE.md`（全局拓扑）。
* **第二层（模块）**：`src/硬件模块/README.md`（必须包含对外接口、依赖、功耗要求）。
* **第三层（文件）**：文件头部的标准化 Docstring。
* **同构校验闭环**：任何针对核心业务代码（如蓝牙协议、固件驱动）的 `git commit`，如果 `pre-commit` 脚本没有检测到对应层级 Markdown/Docstring 的同步 diff 修改，**直接 Exit 1 物理打回，不允许进入 LLM 审查**。

## 5. 质量防线与全层级熔断铁律 (Triple Defense Rule)
废除单一拦截，实行全链路“不信任”审查：
1. **第一道防线（物理级 Hook）**：AST 语法树扫描拦截危险函数（`eval`、`os.system`），行数阈值硬拦截，依赖白名单拦截。
2. **第二道防线（LLM 逻辑级）**：Reviewer Agent 必须按序执行：① 架构边界审查 -> ② 质量红线审查 -> ③ 需求契约符合性审查。缺一不可。
3. **第三道防线（全局级熔断）**：任何单节点 `retry_limit` 超过 3 次，立即生成 `.SYSTEM_HALTED.lock`，冻结全局图结构，拉响警报。

## 6. 人类介入 (HITL) 强制闭环铁律 (HITL Closed-Loop Rule)
系统触发 HALTED 挂起后，人类架构师接管。**严禁人类在沙盒中随意修改代码后直接重启。** 必须严格遵循以下”术后缝合”序列：
1. **更文档**：将排查出的逻辑漏洞或新加的依赖，写入对应的架构 Markdown。
2. **更契约**：同步更新 `.ai-state/current_task_contract.json`。
3. **重评审**：强行触发一次 CPO_Critic 双模型评审，校验人类的修改是否引发新的死锁。
4. **同快照**：脚本重新计算所有架构文件的 SHA-256，更新哈希基准。
5. **准重启**：删除物理锁，系统基于新的安全基准恢复流转。

## 7. 指令歧义防护铁律 (Instruction Ambiguity Guard Rule)
**问题根源**：人类指令中的歧义可能导致Agent移除安全锁。”越快越好”可能跳过检查，”简单处理”可能绕过边界。
* **歧义检测强制**：所有来自人类的任务指令，必须通过 `InstructionGuard` 歧义检测。
* **时间压力拦截**：包含”越快越好”、”尽快”、”马上”等时间压力词汇的指令，必须要求人类明确：哪些安全检查不可跳过。
* **简化处理拦截**：包含”简单处理”、”先...再说”等简化暗示的指令，必须要求人类明确：最低质量底线。
* **权限模糊拦截**：包含”看着办”、”你决定”等权限模糊词汇的指令，必须要求人类明确：决策边界。
* **实现位置**：`src/security/instruction_guard.py`

## 8. KPI陷阱防护铁律 (KPI Trap Prevention Rule)
**问题根源**：Agent可能选择”表面达标”而非”实质达标”，如测试覆盖率90%但只测简单路径。
* **浅层覆盖检测**：覆盖率指标必须伴随边界测试清单、异常场景列表。
* **快速路径检测**：性能指标必须伴随异常路径测试、缓存策略说明。
* **指标游戏检测**：完成率指标必须伴随Demo演示、端到端测试、用户场景验证。
* **错误压制检测**：禁止空的 `except: pass`，所有异常必须记录日志或明确处理。
* **占位代码检测**：Hook拦截 `# TODO`、`# FIXME`、`pass # placeholder` 等占位标记。
* **实现位置**：`src/security/kpi_trap_detector.py`

## 9. 行为边界审计铁律 (Behavior Boundary Audit Rule)
**问题根源**：Agent可能学习人类闲聊、越权操作、边界漂移，导致不可控行为。
* **AI身份声明强制**：所有Agent在实例化时必须声明”我是AI，不是人类”。禁止学习人类闲聊、扯淡、情绪化表达。
* **行为全量记录**：所有Agent行为必须记录到 `behavior_logs.jsonl`，包含：时间戳、Agent ID、行为类型、输入输出哈希、边界检查结果。
* **边界违规报警**：任何 `BOUNDARY_CROSS` 类型行为立即触发警告，累计3次触发熔断。
* **通信结构化**：Agent间通信必须使用结构化JSON，禁止自然语言闲聊。
* **审计报告周期**：每日生成行为审计报告，包含违规统计、AI身份合规率、改进建议。
* **实现位置**：`src/audit/behavior_logger.py`

## 10. 竞品分析前置验证铁律 (Competitor Pre-validation Rule)
**问题根源**：竞品选择错误会导致整个分析报告失效。典型案例：
1. 将"Meta Ray-Ban Display"（$799，有AR显示）认成"Meta Ray-Ban"（$299，无显示）
2. 将"投屏眼镜"与"独立AR设备"混为一谈
3. 遗漏跨品类竞品

### 10.1 产品识别验证（必须第一步执行）
* **禁止假设产品信息**：任何产品名称，必须先搜索验证，不得假设
* **名称逐字核对**：产品名称中的每个词都可能有含义（如"Display"意味着有显示功能）
* **官方来源优先**：必须访问官方网站获取规格，而非依赖记忆或推断
* **置信度强制标注**：所有产品信息必须标注来源和置信度（high/medium/low）

### 10.2 验证流程（强制执行）
```
Step 1: 产品名称核验
  - 搜索产品官方页面
  - 确认产品是否存在
  - 提取官方规格（价格、重量、功能）

Step 2: 规格提取
  - 从官方页面提取关键规格
  - 标注数据来源URL

Step 3: 竞品关系判断
  - 功能重叠度分析
  - 价格区间对比
  - 目标用户重叠分析
```

### 10.3 交叉验证设计原则
* **反向验证必须**：评审模型必须独立验证产品信息，而非只验证逻辑
* **事实前提检验**：评审必须检验"产品识别是否正确"
* **错误问题示例**："Meta Ray-Ban（无显示）是否应纳入竞品？" — 错误在假设了产品信息
* **正确问题示例**："请独立搜索验证Meta Ray-Ban Display的产品规格，然后判断竞品关系"

### 10.4 竞品分层与筛选
* **多层竞品筛选**：按直接竞品、替代品、跨品类分层
* **跨品类竞争检验**：考虑"低价替代品"效应
* **全球市场扫描**：覆盖美国、欧洲、中国、日韩市场
* **头部玩家不遗漏**：检查Meta、Apple等头部品牌的产品线

* **知识库参考**：`.ai-state/competitive_analysis/competitor_filter_fundamental_failure.md`
* **实现位置**：竞品分析流程前置检查清单
