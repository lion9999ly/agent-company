> 以下是你必须无条件遵守的物理法则。任何违背以下法则的输出，都将被物理拦截脚本无情销毁，并触发最高级别的系统告警。

## 1. 物理边界与唯一真相源 (SSOT) 铁律
* **只读禁区**：`.ai-architecture/` 目录下的所有文件为系统的“最高宪法”。你**仅有读取权限，绝对禁止任何修改或覆写尝试**。
* **唯一运行时内存**：所有的任务分发、进度流转、协作沟通，必须通过读写 `.ai-state/` 目录下的标准化 JSON/Markdown 文件完成。
* **禁止私下传话**：Agent 之间绝对禁止通过自然语言上下文的“闲聊”传递关键信息。上游的输出必须写入 `.ai-state/`，下游从该目录精确读取。

## 2. 三不原则与质量防线铁律
* **不生产废品**：所有代码、文案的生成，必须严格符合 `.ai-architecture/01-quality-redlines.md` 中定义的量化硬阈值（如：单文件 $\le 800$ 行）。
* **零逃逸拦截**：CTO 团队生成的任何代码，在提交给 LLM 逻辑审查前，**必须且必定**先流经本地 Hook 脚本（Lint/Test）。Exit code 非 0，立即打回，不允许任何狡辩。
* **不接受/传递废品**：CPO 下发的 `task_contract.json` 是唯一验收标准。Reviewer Agent 必须逐字段校验，一项不符，全盘打回。

## 3. 分形文档强制同构铁律
* **文档即架构**：本系统强制实行「全局 -> 模块 -> 文件」三层文档结构（详见 `.ai-architecture/05-document-standards.md`）。
* **修改代码必先改文档**：任何 Agent 在对 `src/` 目录下的业务架构进行调整前，必须先同步更新对应的 `README.md` 或头部注释。
* **文档脱节即视为Bug**：拦截脚本将强制执行“文档-代码一致性校验”。发现文档落后于代码的提交，直接视为严重 Bug 并打回重做。

## 4. 跨模型对审与防幻觉铁律
* **规划阶段双审制**：CPO_Plan (负责拆解) 必须经过 CPO_Critic (负责挑刺) 至少 3 轮的跨模型审查（Cross-Model Verification）。在审查未通过、未生成最终静态 JSON 契约前，禁止向下游分发任何任务。
* **渐进式上下文加载**：你只被允许读取当前子任务所必须的最小上下文切片（详见 `.ai-architecture/04-context-guidelines.md`）。严禁擅自申请读取全局历史对话，从源头抑制熵增。

## 5. 人类介入 (HITL) 闭环铁律
* 任何单节点重试超过 **3 次**，系统强制挂起并记录至 `.ai-state/human_intervention.log`。
* 人类介入并修改状态后，系统重启必须严格执行以下恢复序列：
  1. 重新读取 `.ai-architecture/` 同步最新规则。
  2. 验证当前 `.ai-state/current_task_contract.json` 的哈希一致性。
  3. 执行通过后，方可解除挂起状态继续流转。
